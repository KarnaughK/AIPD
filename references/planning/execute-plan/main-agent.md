# 阶段 4: 执行计划（多 Agent 调度模式）

**目标**: 主 Agent 作为调度者，派发任务给子 Agent 执行，自身保持空闲随时响应用户

**触发条件**: `plan/` 下存在进行中的计划（非 archive/ 目录中的文件）

## 核心理念

- **主 Agent = 项目经理**：调度任务、审查结果、跟用户聊天
- **子 Agent = 独立开发者**：拿到精确的步骤文件，独立完成开发-调试-自检全循环
- **目的不是并发加速，而是保持主 Agent 上下文干净 + 随时响应用户**

---

## 执行前检查（重要！）

**当用户提出需求或功能时，AI 必须先读取 `_adoc/index.md` 大索引，检查 `business/` 和 `tech/` 各模块中是否已有对应的标准流程（SOP）。**

- **如果找到** → 按已有文档执行，不要重复造轮子
- **如果没找到，且该需求是流程性任务** → 询问用户：
  - 选项 A：现场定义流程，边做边记录
  - 选项 B：先创建 SOP 文档，再执行
  - 选项 C：直接执行，不记录流程

---

## 执行步骤

### 步骤 0: 创建开发分支（首次执行时）

在开始执行 Plan 之前，先创建独立的开发分支：

```bash
# 检查当前分支
git branch --show-current

# 创建并切换到新分支（使用 Plan 版本号命名）
git checkout -b v0.x-功能名称

# 例如：git checkout -b v0.2-sitemap
```

**分支命名规则**：
- 使用 Plan 的版本号 + 功能简称
- 例如：`v0.2-sitemap`、`v0.3-validator`

**为什么要用分支**：
- 隔离开发，不污染 main 分支
- 可随时回滚，零成本放弃
- 代码和文档在同一分支，保证同步
- 用户验收通过后再合并

### 步骤 1: 加载上下文

```bash
# 读取大索引（获取全局上下文）
cat _adoc/index.md

# 读取当前 plan
cat _adoc/plan/v0.x-xxx.md

# 注意：不需要读 tech/ 和 business/ 的详细文档
# 这些由子 Agent 根据步骤文件自行读取
```

### 步骤 2: 检查进度

检查 Plan 主文档中的迭代步骤表格：
- 哪些步骤已完成？
- 下一个待执行的步骤是什么？
- 检查 `plan/steps/` 目录下的步骤文件是否存在

如果步骤文件缺失，先创建（参考 `@templates/step.md`）。

### 步骤 3: 展示进度状态

```
【当前计划状态】

计划: v0.1-keyword-research
类型: dev
目标: 搭建监测后台骨架

迭代步骤:
✅ v0.1.1 项目初始化
✅ v0.1.2 数据库搭建
⏳ v0.1.3 API 骨架          ← 下一个
⬚  v0.1.4 前端看板
⬚  v0.1.5 定时任务占位

---
准备派发 v0.1.3 给子 Agent 执行。
```

### 步骤 4: 派发任务给子 Agent

找到下一个未完成的 C 级步骤，使用 Agent 工具启动子 Agent。

**开发型任务（type: dev）**：

```
使用 Agent 工具，参数：
- description: "执行步骤 v0.1.3 API 骨架"
- run_in_background: true
- prompt: 见下方模板
```

**Prompt 模板（dev）**：
```
你是 AIPD 开发执行者。

## 你的任务
1. 阅读步骤文件：{step_file_path}（绝对路径）
2. 阅读步骤文件中「上下文文档」列出的所有文档
3. 按「任务清单」逐项执行（写代码、调试）
4. 按「验收标准」逐项自检
5. 自检通过 → 返回：「步骤 {step_id} 已完成」
6. 自检不通过 → 修复后重试（最多 3 次）
7. 3 次仍失败 → 返回：「步骤 {step_id} 失败：{具体原因}」

## 约束
- 只做步骤文件中列出的任务，严禁做额外的事
- 严格遵循上下文文档中的技术约束
- 不要返回完整代码内容，只返回完成通知（节省主 Agent 上下文）
- 如果步骤文件中有「不做」部分，严格遵守
```

**调研型任务（type: research）**：

```
使用 Agent 工具，参数：
- description: "执行调研 v0.1.1 竞品分析"
- run_in_background: true
- prompt: 见下方模板
```

**Prompt 模板（research）**：
```
你是 AIPD 调研执行者。

## 你的任务
1. 阅读步骤文件：{step_file_path}（绝对路径）
2. 按「调研目标」执行调研（搜索、读文档、分析）
3. 将调研结果写入：{result_file_path}
4. 返回：「调研 {step_id} 已完成，结果见 {result_file_path}」

## 约束
- 调研结果必须写入文件，不要在返回消息中包含大段内容
- 区分事实和推测，推测部分标注「推测」
- 标注信息来源
```

**审查型任务（type: review）**：

```
Prompt 模板（review）：

你是 AIPD 代码审查者。

## 你的任务
1. 阅读步骤文件：{step_file_path}（绝对路径）
2. 按「审查范围」读取相关代码
3. 按「审查标准」逐项检查
4. 将审查报告写入：{result_file_path}
5. 返回：「审查 {step_id} 已完成，报告见 {result_file_path}」

## 约束
- 只读不改（除非步骤文件明确要求修复）
- 问题按严重程度分级：Critical / Warning / Info
```

### 步骤 5: 主 Agent 回到空闲状态

派发任务后，告知用户：

```
✅ 已派发步骤 v0.1.3 给子 Agent 执行。
预计内容：实现 4 个 API 路由骨架。

你可以继续跟我聊别的事，子 Agent 完成后我会通知你。
```

**主 Agent 此时可以做的事**：
- 跟用户聊天、讨论需求
- 执行其他调研任务
- 查看文档
- 规划下一个 Plan

**主 Agent 此时不做的事**：
- 不主动去查看子 Agent 的进度
- 不读子 Agent 正在修改的文件
- 等系统通知子 Agent 完成即可

### 步骤 6: 收到子 Agent 结果

子 Agent 完成后系统会自动通知。主 Agent 处理结果：

**成功（"步骤 xxx 已完成"）**：
1. 更新 Plan 主文档：步骤状态改为 ✅
2. 更新步骤文件：任务清单标记 `[x]`
3. 告知用户结果
4. 自动派发下一个步骤（如果有）

```
✅ 步骤 v0.1.3 API 骨架 已完成。

进度：3/5 步骤完成
下一步：v0.1.4 前端看板

已自动派发 v0.1.4，你可以继续聊别的。
```

**失败（"步骤 xxx 失败：原因"）**：
1. 记录失败原因
2. 告知用户
3. 询问用户：重试 / 跳过 / 手动处理

```
❌ 步骤 v0.1.3 API 骨架 失败。
原因：better-sqlite3 编译错误，需要安装 C++ 编译工具

选择：
A. 重新派发（我先帮你解决编译问题）
B. 跳过这个步骤，先做后续的
C. 我自己来处理这个步骤
```

### 步骤 7: 所有步骤完成 → 主 Agent 验收

当所有 C 级步骤都标记为 ✅，进入主 Agent 验收阶段：

#### 7.1 自动验收测试

主 Agent 根据 Plan 中的验收标准，自动执行测试：

```
🎉 所有开发步骤已完成！现在进行整体验收...

【自动验收测试】
✅ 验收标准 1: xxx - 通过
✅ 验收标准 2: xxx - 通过
✅ 验收标准 3: xxx - 通过
...
```

**验收方式**：
- 启动服务器（如需要）
- 执行 API 测试
- 检查数据库状态
- 验证文件是否生成

#### 7.2 更新相关文档

验收通过后，主 Agent 自动更新相关文档（在当前开发分支上）：

根据 Plan 中的"文档同步"清单，逐项检查并更新：
- `tech/xxx/README.md` - 补充新功能说明
- `business/xxx/action-plan.md` - 更新任务状态
- `_adoc/index.md` - 更新模块索引

```
【文档同步】
✅ 更新 tech/keyword-research/README.md
✅ 更新 business/keyword-research/action-plan.md
✅ 检查 _adoc/index.md（无需更新）
```

#### 7.3 提交代码

文档更新完成后，提交所有更改：

```bash
git add .
git commit -m "feat: 完成 v0.x-功能名称

- 实现 xxx
- 实现 xxx
- 更新相关文档"
```

#### 7.4 通知用户验收

```
✅ Plan v0.x-功能名称 开发完成！

【完成情况】
✅ v0.x.1 xxx
✅ v0.x.2 xxx
...

【自动验收】已通过
【文档更新】已完成
【代码提交】已提交到分支 v0.x-功能名称

---
🌐 服务已启动：http://localhost:3000
请手动验收功能是否符合预期。

验收通过后回复"OK"或"通过"，我将执行归档和合并分支。
```

---

## 关键原则

1. **主 Agent 永不执行开发任务**：所有代码工作由子 Agent 完成
2. **子 Agent 只看步骤文件**：不需要知道 Plan 全貌，只关注自己的步骤
3. **主 Agent 做最少的审查**：看子 Agent 返回的通知即可，不需要逐行 review 代码
4. **失败时人在环路**：子 Agent 失败后，主 Agent 跟用户商量怎么处理，不自动重试无限次
5. **只做 plan 中定义的任务**：不要"顺手"做额外的事
6. **遵循 tech 约束**：如果约束不合理先讨论修改

---

## 降级模式

如果用户选择不使用子 Agent（如简单任务、紧急修复），主 Agent 可以退回到直接执行模式：

1. 跳过步骤 4-6
2. 主 Agent 自己按任务清单逐项执行
3. 完成后更新 Plan 文档

触发降级的信号：
- 用户说"你自己来做吧"
- 只有 1-2 个简单任务
- 子 Agent 不可用

---

## 遇到问题时的排查流程

**当执行过程中遇到 bug 或异常时，按以下顺序排查**：

### 1. 先查调试地图

每个 tech 模块都有 `debug-map.md`，记录了：
- 数据流路径
- 常见问题排查步骤
- 快速验证命令

**查看方式**：
```
读取 _adoc/tech/{项目名}/debug-map.md
```

例如：
- 后端问题 → `_adoc/tech/keyword-research/debug-map.md`
- 前端问题 → `_adoc/tech/seo-dashboard/debug-map.md`

### 2. 按地图验证每一层

调试地图会告诉你：
- 数据库有数据吗？
- API 返回正确吗？
- 前端收到数据吗？

**逐层验证，快速定位问题在哪一层**。

### 3. 修复后更新地图

如果发现新的问题模式，更新 debug-map.md 添加到"常见问题"中。
- 紧急 bug 修复
