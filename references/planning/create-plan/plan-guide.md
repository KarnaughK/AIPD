# Plan 编写指南

Plan 是迭代计划文档，定义"这次要做什么"。

## 核心原则

### 1. 短生命周期

Plan 是"用完即归档"的文档。

- 创建 → 执行 → 完成 → 归档
- 一个 plan 通常对应 1-2 周的工作量
- 完成后移动到 `archive/` 目录

### 2. 可执行性

Plan 中的任务应该是明确可执行的。

**好的任务**：实现搜索 API 端点
**不好的任务**：优化搜索功能（太模糊）

### 3. 范围可控

每个 plan 的范围应该足够小。

- 宁可多创建几个 plan
- 也不要一个 plan 塞太多任务
- MVP 优先

---

## Plan 类型

Plan 不只是"开发任务"，还可以是调研、审查等。头部用 `类型` 字段标记：

| 类型 | 标记 | 用途 | 子 Agent 做什么 |
|------|------|------|----------------|
| **dev** | 默认 | 开发功能 | 写代码、调试、自检 |
| **research** | `research` | 调研分析 | 搜索、读文档、返回结论 |
| **review** | `review` | 代码/方案审查 | 读代码、分析问题、返回报告 |

**示例头部**：
```markdown
# Plan: v0.1-keyword-research
> **类型**: dev
> **本次迭代目标**: 搭建监测后台骨架
```

## 版本号规范（三级 A.B.C）

每个 B 级功能版本是一个**目录**，包含 Plan 主文档和所有 C 级步骤文件：

```
plan/
├── vA.B-{name}/              # B 级功能版本（目录）
│   ├── plan.md               # Plan 主文档
│   ├── vA.B.1-{name}.md      # C 级迭代步骤
│   ├── vA.B.2-{name}.md
│   └── ...
```

| 级别 | 含义 | 谁创建 | 谁执行 | 文件位置 |
|------|------|--------|--------|---------|
| **A** | 大版本（里程碑发布） | 用户决定 | - | - |
| **B** | 功能版本（一次讨论产出） | 主 Agent + 用户 | `plan/vA.B-name/plan.md` |
| **C** | 迭代步骤（最小可交付单元） | 主 Agent 拆分 | 子 Agent | `plan/vA.B-name/vA.B.C-name.md` |

**工作流**：
```
用户跟主 Agent 聊"做一个列表页功能"
→ 主 Agent 规划 v0.1-list-page/（B 级功能版本目录）
→ 目录内创建 plan.md + C 级步骤：
   v0.1-list-page/
   ├── plan.md
   ├── v0.1.1-basic-table.md
   ├── v0.1.2-pagination.md
   ├── v0.1.3-filter.md
   └── v0.1.4-read-status.md
→ 子 Agent 串行执行每个步骤
→ 主 Agent 始终空闲，随时响应用户
```

**示例**：
```
plan/
├── v0.1-关键词监测骨架/        # B 级功能版本
│   ├── plan.md                # Plan 主文档
│   ├── v0.1.1-项目初始化.md    # C 级步骤
│   ├── v0.1.2-数据库.md
│   └── v0.1.3-API路由.md
├── v0.2-search/               # 另一个 B 级功能版本
│   ├── plan.md
│   └── ...
└── archive/                   # 已归档
```

## 文档结构

### 1. 目标（必须）

本次迭代要达成什么。

```markdown
## 1. 目标 (Objectives)

- **Infra**: 完成 Nuxt 4 + SQLite 基础环境搭建
- **Feature**: 实现 Prompt 的 CRUD 与版本管理
```

### 2. 涉及的模块（必须）

```markdown
## 2. 涉及的模块

- `business/prompt/spec-prompt-assets.md` - 实现 1.1 和 1.2 部分
- `business/test/spec-test-framework.md` - 实现基础运行功能
```

### 3. 约束（必须）

```markdown
## 3. 约束 (Constraints)

- **System**: 严格遵循 `tech/xxx/README.md` 的技术规范
- **Scope**: 暂不实现 1ToN 测试和 Workflow 集成
```

### 4. 任务清单（必须）

```markdown
## 4. 任务清单 (Tasks)

### 4.1 基础设施
- [ ] 初始化项目
- [ ] 配置数据库连接

### 4.2 功能开发
- [ ] 实现列表页
- [ ] 实现详情页
```

### 5. 风险（必须）

```markdown
## 5. 风险 (Risks)

- **LLM 稳定性**: 准备 Mock 数据
```

### 6. 验收标准（可选但推荐）

```markdown
## 6. 验收标准

- [ ] 能创建、编辑、删除 Prompt
- [ ] 每次保存自动生成新版本
```

### 7. 迭代步骤（多 Agent 模式必须）

当使用子 Agent 执行时，需要将任务清单拆为 C 级步骤：

```markdown
## 7. 迭代步骤 (Steps)

| 步骤 | 文件 | 依赖 | 状态 |
|------|------|------|------|
| v0.1.1 项目初始化 | v0.1.1-项目初始化.md | 无 | 待执行 |
| v0.1.2 数据库搭建 | v0.1.2-数据库.md | v0.1.1 | 待执行 |
| v0.1.3 API 骨架 | v0.1.3-API路由.md | v0.1.2 | 待执行 |
```

每个步骤文件与 `plan.md` 同目录（同一个功能版本目录内），格式见 `@templates/step.md`。

## C 级步骤拆分规范

### 拆分原则

每个 C 级步骤必须满足：

1. **足够小**：子 Agent 能独立完成（不需要跟用户确认）
2. **足够精确**：有明确的输入、输出、验收标准
3. **有序依赖**：标注依赖关系（哪个必须先完成）
4. **自包含**：步骤文件里列出所有需要读的上下文文档

### 粒度判断

**太大**：
- 包含多个独立功能点
- 子 Agent 需要做太多决策
- 验收标准超过 5 条

**太小**：
- 改一行配置
- 创建一个空文件

**合适**：
- 一个独立的功能点（如"实现列表展示"）
- 2-5 条验收标准
- 子 Agent 能跑完开发-调试-自检全循环

### 步骤间依赖

- 大多数步骤是串行的（后一个依赖前一个的产出）
- 偶尔有可并行的步骤（独立模块），但默认串行执行
- 在 Plan 主文档的"迭代步骤"表格中标注依赖关系

## 归档规范

完成后将整个功能版本目录移动到 archive/：

```bash
# 归档整个功能版本目录（含 plan.md 和所有步骤文件）
mv _adoc/plan/v0.2-search/ _adoc/plan/archive/v0.2-search/
```

归档步骤：
1. 所有任务/步骤都标记为完成（`- [x]`）
2. 文档同步检查完成
3. 询问用户该 Plan 属于哪个模块
4. 更新 `_adoc/plan/index.md`（在对应模块下添加记录）
5. 移动整个目录到 `archive/`

## 任务拆分原则

### 粒度判断（B 级任务清单）

**太大**：
- 任务描述超过一行
- 完成需要超过 1 天
- 包含多个独立步骤

**太小**：
- 5 分钟内能完成
- 过于琐碎（如"创建文件夹"）

**合适**：
- 2-4 小时可完成
- 一个明确的可交付物
- 可独立验证

### 任务分组

按阶段或模块分组：

```markdown
### 基础设施
- [ ] ...

### 功能开发
- [ ] ...

### 测试验证
- [ ] ...
```

## 与业务模块的关系

### Plan 引用业务模块

Plan 中的任务应该对应业务模块中的能力：

```markdown
## 涉及的模块

- `business/prompt/spec-prompt-assets.md`
  - 实现 "1. 提示词定义" 部分
```

### Plan 不重复 Spec

Plan 不需要重复 Spec 中的详细描述，只需要：
- 引用 Spec
- 说明本次实现哪些部分

## 常见问题

### Q: 一个 Plan 应该包含多少任务？

A: 通常 5-15 个任务。太少说明粒度太粗，太多说明范围太大。

### Q: Plan 执行到一半想加任务怎么办？

A:
1. 讨论是否真的需要
2. 如果需要，更新 Plan 文档
3. 不要私自增加范围

### Q: Plan 中的任务做不完怎么办？

A:
1. 标记未完成的任务
2. 讨论是否继续当前 Plan
3. 或者归档当前 Plan，创建新 Plan

### Q: 需要同时进行多个 Plan 吗？

A: 不建议。专注一个 Plan，完成后再开始下一个。

### Q: 什么时候用 C 级步骤？

A: 当使用子 Agent 执行时必须拆分。如果主 Agent 自己执行（简单任务），可以不拆。
