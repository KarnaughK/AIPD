# 阶段 2: 设计功能模块

**目标**: 基于项目方向，设计具体的业务模块和技术模块，产出 `business/` 和 `tech/` 文档

**触发条件**: `intent.md` 完整，但 `business/` 为空或文件很少

## 执行步骤

### 步骤 1: 复述项目方向

```bash
cat _adoc/intent.md
```

```
【当前项目方向】

项目: xxx
愿景: xxx
核心目标: xxx

---
基于这个方向，我们来设计需要哪些业务模块和技术模块。
```

### 步骤 2: 引导模块拆分

- "要实现这个愿景，用户需要完成哪些核心任务？"（→ 业务模块）
- "技术上需要哪些项目/服务来支撑？"（→ 技术模块）
- "有没有优先级？哪些是 MVP 必须的？"

### 步骤 3: 主动提供建议

基于 intent 分析：
- 推测可能需要的业务模块和技术模块
- 画出模块之间的依赖关系
- 区分"核心功能"和"增强功能"

### 步骤 4: 收集模块信息

对每个确定的模块，收集：
- 核心能力（一句话描述价值）
- 用户旅程（关键交互流程）
- 验收标准（怎么算做完了）
- 依赖关系（依赖/被依赖）

### 步骤 5: 创建模块文档

```
_adoc/business/
├── 模块A/
│   ├── README.md          # 模块说明 + Spec
│   └── sop-xxx.md         # 标准流程（按需）
└── 模块B/
    └── README.md

_adoc/tech/
├── index.md               # 技术模块索引 + 环境说明
├── 项目A/
│   └── README.md          # 项目概述 + 技术栈
└── 项目B/
    └── README.md
```

### 步骤 6: 更新索引

更新 `_adoc/index.md` 大索引，添加新模块。

```

- landing  - 落地页
```

## 关键原则

- 每个模块应该是可独立交付的产品能力
- 模块之间尽量低耦合
- 先设计核心功能，增强功能可以后续补充

---

## 模块设计方法论

### 模块化判断标准

- 这个能力可以单独交付吗？
- 这个能力有独立的用户价值吗？

如果两个都是"否"，考虑合并到其他模块。

### 粒度判断

**太大的信号**：
- 模块描述超过 3 页
- 包含多个独立的用户旅程
- "核心能力"需要用"和"连接多个能力
→ 拆分成多个模块

**太小的信号**：
- 不能独立交付
- 必须和另一个模块一起才有意义
- 验收标准只有 1-2 条
→ 合并到相关模块

**合适的粒度**：
- 一个明确的核心能力
- 2-5 个相关的用户旅程
- 4-8 条验收标准
- 1-2 页内容

### 用户旅程写法

描述"用户能做什么"，不是"系统怎么实现"。

**好的写法**：用户可以创建 Prompt 并保存为新版本
**不好的写法**：调用 POST /api/prompt 接口，数据存入 SQLite

示例：
```markdown
### Journey: 创建新提示词
1. 用户点击"新建"，输入名称并选择分组
2. 在编辑器中定义 Input Variables
3. 编写 Prompt Body，引用变量
4. 保存，系统自动生成 v1 版本
```

### 验收标准写法

粒度要适中：
- **太粗**：用户能管理 Prompt（无法验证）
- **太细**：点击按钮后 200ms 内响应（过于技术化）
- **合适**：编辑器支持变量高亮与自动补全（可验证、面向用户）

### 依赖关系处理

- 依赖过多 → 考虑合并成一个模块，或重新划分边界
- 跨模块功能 → 确定主要归属，在相关模块的依赖关系中标注
