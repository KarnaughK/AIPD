# Agent Team 使用指南

## 强制要求

**AIPD 使用 Agent Team，不使用 Sub Agent（内置 Agent 工具）。**

---

## Agent Team 的上下文加载机制

### 核心原理

Agent Team 创建的子 Agent 是**独立的 Claude Code 实例**：
- 不继承主 Agent 的对话历史
- 不知道主 Agent 读过什么文件
- 工作目录继承主 Agent 的 `cwd`
- 所有上下文通过 `prompt` 参数传入

### 工作模式

```
主 Agent：
1. 准备好完整的 prompt（包含任务描述 + 文件路径）
2. 创建 Agent Team 子 Agent
3. 等待子 Agent 返回结果
4. 上下文保持干净

子 Agent：
1. 读取 prompt 中指定的文件
2. 执行任务
3. 返回简洁通知
```

---

## AIPD 的策略

**一件事 = 一个固定的 SOP prompt + 每次不同的变量**

### 组织方式

```
_adoc/plan/v0.x-功能名/
├── plan.md                    # B 级计划（主 Agent 读）
└── steps/
    ├── v0.x.1-步骤名.md       # C 级步骤（子 Agent 读）
    ├── v0.x.2-步骤名.md
    └── v0.x.3-步骤名.md
```

**关键点：**
- 步骤文件（step.md）= 子 Agent 的完整指南
- 主 Agent 只需要知道步骤文件的路径
- 子 Agent 读取步骤文件后，按指引执行

## Prompt 最佳实践模板

### 模板结构

```
你是 AIPD {角色名}。

## 你的任务

1. 读取步骤文件：{step_file_path}（绝对路径）
2. 按步骤文件中的「上下文文档」列表，逐一读取
3. 按步骤文件中的「任务清单」逐项执行
4. 按步骤文件中的「验收标准」自检
5. 完成后返回：「步骤 {step_id} 已完成」

## 变量（本次执行的具体参数）

{variables}

## 约束

- 只做步骤文件中列出的任务
- 严格遵循上下文文档中的技术约束
- 返回简洁通知，不要返回完整代码内容
```

---

## 步骤文件（step.md）结构

步骤文件是子 Agent 的完整执行指南，必须包含以下部分：

### 必需字段

```markdown
# v0.x.y - 步骤名称

## 目标
简短描述这个步骤要达成什么

## 上下文文档
列出子 Agent 需要读取的文档（使用绝对路径）：
- /path/to/tech/xxx/README.md - 技术约束
- /path/to/business/xxx/index.md - 业务规则
- /path/to/dev/xxx/debug-map.md - 调试地图

## 任务清单
- [ ] 任务 1
- [ ] 任务 2
- [ ] 任务 3

## 验收标准
- 标准 1
- 标准 2
- 标准 3

## 不做（可选）
明确列出不要做的事情
```

### 关键要点

1. **上下文文档必须用绝对路径**：子 Agent 继承工作目录，但明确的绝对路径更可靠
2. **任务清单要具体可执行**：每项任务都是明确的动作
3. **验收标准要可量化**：子 Agent 能自己判断是否通过
4. **"不做"很重要**：防止子 Agent 过度发挥

## 主 Agent 的调用方式

### 创建 Agent Team 子 Agent

```javascript
// 伪代码示例
createAgent({
  name: "worker-v0.9.1",
  agentType: "general-purpose",
  model: "claude-opus-4-6",
  prompt: `你是 AIPD 开发执行者...`,  // 完整 prompt
  cwd: "/Users/xxx/project",           // 工作目录
  backendType: "in-process"
})
```

### 等待结果

主 Agent 创建子 Agent 后：
- 不主动查看子 Agent 进度
- 等待子 Agent 返回通知
- 收到通知后更新 Plan 状态
- 继续派发下一个步骤

## 最佳实践总结

### 1. 步骤文件是核心
- 步骤文件包含子 Agent 需要的所有信息
- 主 Agent 只需传递步骤文件路径 + 变量
- 步骤文件可复用（固定的事 + 不同的变量）

### 2. 路径使用规范
- 步骤文件中的上下文文档：使用绝对路径
- 主 Agent 传递给子 Agent 的路径：使用绝对路径
- 避免相对路径导致的工作目录问题

### 3. 返回通知要简洁
- 子 Agent 只返回：「步骤 {step_id} 已完成」或「步骤 {step_id} 失败：{原因}」
- 不返回完整代码内容
- 不返回详细执行过程
- 目的：节省主 Agent 上下文

### 4. 变量传递方式
- 简单变量：直接写在 prompt 中
- 复杂变量：写入文件，传递文件路径
- 批量变量：使用 JSONL 格式，传递变量 ID

### 5. 失败处理
- 子 Agent 失败后，主 Agent 询问用户：重试 / 跳过 / 手动处理
- 不自动无限重试
- 记录失败原因，便于排查

## 完整示例

### 步骤文件：v0.9.1-批量导入脚本.md

```markdown
# v0.9.1 - 批量导入脚本

## 目标
创建脚本，从 txt 文件批量导入网站到数据库

## 上下文文档
- /Users/xxx/project/_adoc/dev/keyword-research/README.md - 技术架构
- /Users/xxx/project/keyword-research/db/schema.sql - 数据库结构

## 任务清单
- [ ] 创建 scripts/import-online-game-sites.js
- [ ] 读取 online-game-sites.txt
- [ ] 解析每行为域名
- [ ] 插入到 sources 表
- [ ] 输出导入统计

## 验收标准
- 脚本能成功运行
- 145 个网站全部导入
- 无重复记录
- 输出导入成功数量

## 不做
- 不要修改数据库结构
- 不要添加额外字段
```

### 主 Agent 的 Prompt

```
你是 AIPD 开发执行者。

## 你的任务

1. 读取步骤文件：/Users/xxx/project/_adoc/plan/v0.9-批量导入与爬取优化/steps/v0.9.1-批量导入脚本.md
2. 按步骤文件中的「上下文文档」列表，逐一读取
3. 按步骤文件中的「任务清单」逐项执行
4. 按步骤文件中的「验收标准」自检
5. 完成后返回：「步骤 v0.9.1 已完成」

## 变量（本次执行的具体参数）

- 导入文件：online-game-sites.txt
- 目标数量：145 个网站

## 约束

- 只做步骤文件中列出的任务
- 严格遵循上下文文档中的技术约束
- 返回简洁通知，不要返回完整代码内容
```

### 子 Agent 执行流程

1. 读取步骤文件 v0.9.1-批量导入脚本.md
2. 读取上下文文档（技术架构、数据库结构）
3. 创建脚本文件
4. 实现导入逻辑
5. 测试运行
6. 自检验收标准
7. 返回：「步骤 v0.9.1 已完成」

### 主 Agent 收到结果后

1. 更新 plan.md：v0.9.1 标记为 ✅
2. 更新步骤文件：任务清单标记 [x]
3. 告知用户
4. 派发下一个步骤 v0.9.2

- [ ] 返回通知简洁（不包含完整代码）
