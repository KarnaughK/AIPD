# AIPD Inbox

Inbox 是未整理信息的收件箱，只负责接住临时材料，不承诺执行、不承诺稳定、不承诺归属。

## 使用原则

- 这里只记录“先存一下，回头再判断”的信息。
- 不在这里拆 work package，不在这里承诺要做。
- 不把这里的内容直接当作 L1-L5 的稳定认知。
- 整理时再判断是否迁移到 L2 / L3 / L4 / L5 / OKR / Case，或直接删除。
- 如果已经明确要推进，创建 Case；如果已经是稳定认知，使用 Weave 回写。

### 2026-06-18 - 任务从模糊到清晰的前置讨论层

来源：
- 当前对话，用户在复盘 token 消耗效率、OpenClaw 维护方式、桌面端开发卡点和自有项目定任务方式时提出的第二个灵感。

原始记录：
- 用户观察到自己已经购买高额度会员，但实际 token 消耗速度只能达到 5x 到 10x 范畴，没法像 OpenClaw 那样高速消耗。
- 一个原因是当前主要在给别人开发项目，任务推进受客户反馈、需求解释、业务理解和沟通节奏影响；这会引入理解成本，降低 token 产出效率。
- 如果是自己开发，只要自己的逻辑完备，就能更快往下推进；但新的瓶颈会变成“任务是否足够清晰”和“自己是否熟悉目标领域”。
- 例如想给当前 API 造桌面端时，卡点不只是实现，而是用户对桌面端不熟悉、不理解，不希望 AI 直接上手做出一坨后再返工。
- 因此很多开发并不是从清晰任务开始，而是要先经历一个“从模糊到清晰”的过程：像早期 AI 图片生成一样，从低像素到高像素，从大模块、大流程开始，再逐层细化。
- AIPD 的 L1 到 L5 本身也有逐层细化含义，但它更像知识库层级，跨度较大；过去一两个月主要细化了执行层，例如 case、phase、work package、上下文隔离和代码开发。
- 接下来可能需要细化“讨论层”或“定任务层”：在 Case Goal / Think 之前，帮助用户把模糊想法、陌生领域、新产品方向逐步澄清为可执行任务。
- 当前给别人开发时，任务通常来自外部；如果给自己开发，任务需要自己定义。因此 AIPD 需要加强“清晰任务 / 讨论任务 / 定任务”的能力，而不只是接收已经讨论好的任务后设计 case 执行。

待判断：
- 是否需要在 `aipd-case` 的 Goal / Think 前补一个更轻的前置能力，例如“idea clarify / task shaping / 从模糊到清晰”？
- 是否应进入 L3 Core，作为 AIPD 从认知层级到任务生成的核心流程补充？
- 是否应进入 L4 Product，成为用户可见的“讨论任务 / 定任务 / 澄清需求”功能线？
- 是否需要设计一套从 L1/L2 模糊意图逐步下降到 L4/L5/case 的对话 SOP？
- 是否需要和上一条“横向扩展模型”合并为 AI 原生开发的完整前置方法：先澄清任务，再横向铺开，再并行执行？

### 2026-06-18 - 上下文解耦的横向扩展模型

来源：
- 当前对话，用户在讨论 OpenClaw token 效率、Agent 并行维护和 AIPD 上下文解耦时提出的第一个灵感。

原始记录：
- 用户提出可以用“空间换时间 / 空间换复杂度降低”来进一步阐述上下文解耦。
- 传统软件工程像“平地起高楼”：整体架构垂直叠高，上下层关系越来越多，必须依赖完整设计、施工校验和统一架构一致性，才能保证几十层高楼不塌。
- AI Agent 时代更像“3D 打印二层小楼”：Agent 很快能打印出低层、局部、独立的小楼；但如果把一堆二层小楼硬拼成高楼大厦，整体会坍塌。
- 因此新的架构方向不是继续把所有东西垂直垒高，而是找一块足够大的平地，把能力设计成可以横向铺开、横向扩展、横向迭代的一组小楼。
- 对代码和功能设计来说，这意味着在上下文解耦之后，应进一步追求“横向无限拓展”：功能拆成多个彼此上下文牵连少的平面单元，让多个 Agent 可以并行完成。
- 例子：一个功能需要 5 个接口，如果这 5 个接口是横向摊开的、没有强上下文依赖，就可以由 5 个 Agent 瞬间并行完成。
- 这个灵感可能补充 AIPD 之前的纵向逻辑上下文解耦：纵向解耦负责降低理解复杂度，横向拓展负责把解耦结果转化为 Agent 并行生产能力。

待判断：
- 是否应进入 L3 Core，作为“上下文解耦 / 横向能力 / Agent 并行生产”的稳定核心认知？
- 是否需要和 `_adoc/L3-core/horizontal-capabilities.md` 中的横向功能能力合并？
- 是否需要形成一篇面向用户的 docs/modules 文档，用“高楼 vs 3D 打印小楼”解释 AI 原生代码架构？
- 是否需要进一步抽象为功能设计原则：优先设计低耦合、可并行、可独立验证的横向单元？
