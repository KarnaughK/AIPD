# AIPD Intent

## 当前方向

AIPD 是面向 AI 时代的软件开发框架，用来组织人和 AI 从项目方向到代码实现再到经验回写的完整协作过程。

这里的“开发框架”不是传统代码框架。传统代码框架主要约束代码实现里的前端、后端、移动端或桌面端；AIPD 约束的是 AI 和人如何共同推进一个软件项目。

AIPD 关注的不只是写代码阶段，而是覆盖项目方向、用户痛点、外部调研、核心抽象、功能设计、代码实现和后续迭代。它希望让项目在 AI 深度参与开发后，仍然能稳定保留方向、判断、上下文、执行状态和经验沉淀。

## 三个大方向

### 项目知识库

项目知识库负责沉淀项目认知，也就是 AIPD 中的 L1 到 L5：

- L1：项目方向、目标和长期取舍。
- L2：用户、场景、痛点、竞品、玩法范式和外部调研。
- L3：项目成立所依赖的核心抽象、核心对象、核心流程或核心模型。
- L4：核心模型落成产品功能后的功能边界、用户可见行为和业务规则。
- L5：产品功能落到代码时的工程规则、跨模块约定和实现边界。

知识库的目标不是堆文档，而是让 Agent 在进入任务前能读到正确上下文，在做判断时能继承项目历史取舍。

### 任务执行系统

任务执行系统负责把较大的软件开发事项从聊天里固定下来，形成可恢复、可派发、可验收、可回写的执行过程。

它包含 OKR、Case、Step、分身 Agent、执行记录、归档和 Weave Candidate 等能力。OKR 负责阶段目标，Case 负责一次事项的目标、边界和上下文索引，Step 负责可执行的小单元，Weave 负责把稳定经验回写到项目知识库或局部 README。

任务执行系统不是为了把 Agent 限死成按钮流程，而是为了让人和 AI 在长任务、多分支和上下文压缩后，仍然能知道当前在做什么、为什么做、做到哪里、下一步应该接哪里。

### AI 原生代码架构

AI 原生代码架构负责探索更适合 AI 理解、修改和扩展的代码组织方式。

传统代码组织常常优先横向分层和复用；AI 深度参与后，更需要关注上下文解耦、纵向黑箱、降低牵连面、局部 README 和代码入口可检索性。

这不是要求所有项目立刻采用某种固定代码形态，而是承认 AI 时代的代码维护成本结构正在变化：局部重复和文件变长不一定是第一优先级问题，隐性依赖、跨模块牵连和上下文跳转失控反而更容易破坏 Agent 的执行稳定性。

## 核心取舍

- AIPD 不替代 Coding Agent。Codex、Claude Code、Cursor Agent 或其他 Agent 负责具体执行；AIPD 负责让项目认知、任务状态和经验沉淀可被继承。
- AIPD 不做传统意义上的前端、后端、移动端或桌面端代码框架。
- AIPD 不把所有内容塞进一个超长提示词，而是优先分层存储、扁平检索、局部任务自足和逐步回写。
- AIPD 不把项目管理、流程管理和代码框架混成一个大而全工具；它优先解决 AI 和人如何共同推进软件项目的问题。
- AIPD 可以渐进采用：先建立知识库最小闭环，再进入 Case / Step 开发逻辑，最后再探索 AI 原生代码架构。

## 适用边界

AIPD 更适合长期迭代、AI 深度参与、存在业务语言、历史取舍、跨模块规则、多 Agent 执行、任务恢复和经验沉淀需求的软件项目。

它尤其适合“一人公司”、单人开发者或一个人配合 AI 开发大型项目的场景：人需要同时承担方向判断、产品设计、研发执行和长期维护，AIPD 可以帮助把这些判断和执行状态沉淀下来，让 AI 更稳定地接住上下文。

如果项目只有几个文件，或者只是一次性原型，AIPD 可能太重；此时只保留最小 README、少量上下文索引或临时 case 即可。

## 当前阶段重点

- 稳定 AIPD Skill 本体，让 `/aipd*` 系列能力能在真实项目中可用。
- 继续完善项目知识库、Case / Step、分身 Agent、Weave 和 SOP 的协作链路。
- 当前研发和验证优先适配 Codex，同时保留 Claude Code 等平台的可迁移空间。
- 在 AIPD Desktop 方向上探索更好的 AI 聊天、case 展示、上下文快捷引用和项目文件预览体验，但 Desktop 不应成为 AIPD Skill 的运行前提。
