# SOP 索引

SOP 是以 Agent 为运行时的可复用 AI 原生程序。

这里的“程序”不是狭义代码，而是回到 program / procedure 的本义：一组可重复、可执行、可检查的步骤。代码、脚本、API、报表和外部工具都可以被 SOP 调用，但它们不是 SOP 本身。SOP 的主体是交给 Agent 执行的完整过程，包括输入、步骤、工具调用、分析判断、输出和收尾。

## 定位

- SOP 不属于 L1-L5 知识库正文，也不等同于 Learn / Weave。
- L1-L5 负责提供项目认知和上下文。
- SOP 负责把上下文、工具和 Agent 判断编排成可复用执行流程。
- Case 可以记录某次 SOP 的具体执行实例；Step 可以承载一次执行中的局部任务。
- Weave 可以在 SOP 执行后判断稳定新信息是否回写到 L1-L5、map、局部 README 或 case。

一句话：

```text
SOP = 目标 + 输入 + 步骤 + 工具调用 + Agent 判断 + 输出 + 记录/回写
```

## 目录结构

当前只建立最小壳子，不预设复杂分类。

遇到一个可复用 SOP 时：

1. 在 `_adoc/sop/` 下创建一个目录。
2. 把相关说明、脚本入口、执行记录样例、输入输出约定或草稿放进去。
3. 在 `_adoc/sop/map.md` 登记索引。
4. 如果该 SOP 高频使用，再回写 `_adoc/map.md` 增加一跳入口。

示例：

```text
_adoc/sop/
├── index.md
├── map.md
├── seo-keyword-research/
│   └── README.md
└── daily-site-report/
    └── README.md
```

## SOP 候选判断

适合先塞进 `_adoc/sop/` 的内容：

- 某个项目动作会重复执行。
- 这个动作需要 Agent 按步骤调用工具、读取数据、分析判断并输出结果。
- 代码只是流程中的一个环节，还需要 Agent 做二次分析、汇报、跟进或分流。
- 这个动作和当前项目强相关，不适合作为 AIPD 通用 skill。

暂不需要放入 SOP 的内容：

- 单次聊天过程。
- 单纯知识判断或项目事实。
- 只需要写入 L1-L5 的稳定认知。
- 只是一段独立脚本，没有 Agent 分析、判断或收尾流程。

## 索引与存储

SOP 的核心要求是能被 Agent 找到和复用。存储位置可以按项目实际情况演进，索引要尽早建立。

- `_adoc/map.md`：项目总图，暴露高频 SOP 入口。
- `_adoc/sop/map.md`：SOP 局部地图，登记 SOP 目录、用户说法、适用范围、输入和输出。
- `_adoc/sop/{sop-name}/`：某个 SOP 的材料和执行说明。

Weave 判断 SOP 归属时，不强行按业务 / 技术拆进 L4 或 L5。若新信息是“可重复执行的项目动作”，优先进入 `_adoc/sop/`；若只是该动作背后的稳定知识，再回写 L1-L5、局部 README 或 map。
