# AIPD 核心认知

L3 Core 在 AIPD 中不只表示“数据模型”。它存放项目成立所依赖的核心认知、核心对象、核心关系、核心流程和项目成立模型。

对业务项目来说，L3 可能主要是合同、产品、任务、交付记录等核心业务模型。对 AIPD 这种框架项目来说，L3 主要是支撑后续 L4 产品能力和 L5 工程实现的底层认知。

## AIPD 的八个核心成立模型

AIPD 当前的 L3 先收敛为八个核心成立模型。它们回答的不是“有哪些 skill / 功能 / 文件”，而是 AIPD 靠哪些内部模型解决 L2 中的痛点，并长期约束 L4 产品能力和 L5 工程实现。

| 核心模型 | 核心问题 | 典型落地 |
|---|---|---|
| 项目知识库维护模型 | 项目知识如何分层、存储、更新、回写并保持可信 | L1-L5、局部 README、map、`aipd-weave` |
| Map-first 上下文检索模型 | Agent 如何先通过 map 命中该读的上下文，而不是默认依赖 RAG、全文搜索或多层目录跳转 | `_adoc/map.md`、L3/L4/L5 map、局部 README、兜底搜索与 map 回写 |
| 文件优先上下文承接模型 | 长任务如何不依赖聊天记忆，而是通过文件 checkpoint 承接状态、边界和恢复入口 | case.md、phase artifact、work package、局部 README、map |
| Think / 任务澄清决策模型 | 模糊想法或 case 推进中的未知如何通过讨论、调研、方案比较和决策出口，变成清晰方向、设计输入或被 kill / defer / research / weave | Case Think phase、discussion / research / options / decision |
| 任务执行模型 | 短周期目标如何通过 Case Contract / Think / Design / Execute / Verify / Close 变成可恢复、可验收、可关闭的执行闭环 | OKR、Case、Work Package、执行记录、归档 |
| Agent 协作思考模型 | Main Agent、分身 Agent、角色 Agent 如何分工，如何保护主线判断和上下文预算 | Agent Entry、分身 Agent、角色 Agent、平台 agent 指引 |
| SOP / AI 程序模型 | 可重复的 Agent 行为如何沉淀成以 LLM / Agent 为运行时的 AI 原生程序 | `_adoc/sop/`、SOP map、工具调用、执行实例 |
| AI 原生代码架构模型 | 代码如何更适合 AI 读取、修改、扩展和验收 | 上下文解耦、纵向黑箱、黑箱上移、Decouple first, DRY later、技术栈经验库 |

这些模型之间有明确分工：

- 项目知识库维护模型回答“知识怎么存、怎么更新”。
- Map-first 上下文检索模型回答“Agent 怎么读、怎么命中、怎么不迷路”。
- 文件优先上下文承接模型回答“长任务怎么在压缩、中断、换 Agent 后继续，不依赖聊天记忆”。
- Think / 任务澄清决策模型回答“模糊想法或 case 推进中的未知如何变清晰，以及是否值得继续推进”。
- 任务执行模型回答“短周期目标怎么设计、执行、恢复、验收和关闭”。
- Agent 协作思考模型回答“多个 Agent 如何保护主线并分工探索”。
- SOP / AI 程序模型回答“重复动作如何沉淀成可执行的 AI 原生程序”。
- AI 原生代码架构模型回答“真实代码如何适配 AI 协作”。

Inbox、Update、Learn、Desktop、Mermaid 等不是当前 L3 的独立成立模型。它们是这些核心模型在 L4 / L5 / SOP / 流程系统中的功能化结果或工具入口。

## 模型底层倾向不能靠上下文临时改写

AIPD 的一个底层判断是：大模型有自己的默认行为倾向，不能指望一次上下文或一段提示词彻底改变它的核心思维方式。

上下文可以校准模型的角色、任务边界、输出格式和当前约束，但很难把模型的底层反应模式临时改成另一种模式。就像一个偏内向的人和偏外向的人面对聚会邀请时，第一反应已经不同；后续可以“装成”另一类人，但默认判断路径已经发生了偏移。模型也是类似：如果一个模型天然倾向于直接执行、过度抽象、横向分层或顺着用户继续扩展，只靠一句“不要这样”通常不能稳定扭转它。

这条认知直接影响 AIPD 的设计方式：

- **不要只讲基础逻辑**：不要期待模型听完一段抽象原则后，就能长期按人的方式发散和自我约束。
- **多用案例校准**：用具体正反例让模型看到什么是对的、什么是错的，例如“垒高楼式 step”与“二层小洋楼式横向铺模块”的对比。
- **多用边界约束**：把允许做什么、不做什么、验收什么、破坏什么写清楚，让模型在边界内运行。
- **选择合适模型和结构**：如果某个模型天然倾向不适合当前任务，优先换模型、换架构或增加外部校验，而不是只靠提示词强行扭转。
- **让架构适配模型**：AI 原生代码架构不是让模型模仿人类工程习惯，而是利用模型擅长处理大量局部上下文的能力，把任务设计成边界清楚、案例明确、可验收的上下文黑箱。

因此，AIPD 的上下文解耦不是普通提示词工程。它承认模型底层倾向难以临时改写，所以把稳定判断写进 L3，把行为边界写进 case / work package / README，把例子写进 Design 和 docs，让 Agent 通过可检索的案例和边界被约束，而不是每次从抽象大道理重新理解。

## Think：人和 AI 的高带宽思考缓冲层

Think 是 AIPD 的高带宽思考缓冲层。当前 AIPD 已把 Think 收敛为 Case 生命周期内的 Think phase，用来处理一个短周期目标推进中的未知、调研、比较和抉择；Case 创建前的模糊想法也应先形成 Case Contract，或通过 Think 判断是否值得继续推进。

Think 存在的原因是：人和 AI 的沟通瓶颈通常不在执行速度，而在任务是否足够清晰。很多想法一开始只是方向、疑问或冲动，需要经过讨论、调研、方案比较、风险判断和范围收敛，才能决定是否进入执行。如果这些过程只停留在聊天上下文里，长讨论、深度调研和多轮取舍会快速吞掉上下文，而且后续不可恢复、不可审计、不可派发。

因此，Think 应作为可恢复的思考阶段，承载讨论主线、调研资料、方案比较、当前判断和决策出口。它可以包含 deep research / 深度检索类工作，但这些资料先属于 Think 的调研上下文；只有稳定结论才通过 Weave 回写 L2 / L3 / L4 / L5。

Think 的默认出口不是“无脑创建 case”，而是明确流向：

- **Create**：想法已经清晰且值得执行，写入 Case Contract，再进入 Design / Execute 后续 phase。
- **Kill**：不值得做，记录终止原因，避免后续重复消耗。
- **Defer**：值得但不是现在做，保留延后原因和触发条件。
- **Research**：信息不足，继续调研或派发调研分身。
- **Weave**：讨论产生稳定认知，但不一定形成执行事项。
- **Continue**：仍处于讨论中，继续澄清。
- **Design**：已在 case 内形成推进结论，进入 Design phase。

这就是“把是否值得做、怎么继续做变成状态”的含义：不是把讨论当作一次性聊天，而是让它有实体、有恢复点、有决策出口，并能和 Case、Weave、Inbox、SOP 发生清楚关系。

## Case Design：找到复杂度爆点

Case Design 是任务执行模型中的关键 phase。它的目标不是完整抽象所有概念，而是找到复杂度爆点，并对爆点做最小必要解耦，让后续执行可以横向铺模块，而不是纵向堆版本。

复杂度爆点是本 case 最容易膨胀、耦合、返工或变成巨型函数的节点。Design 应先识别爆点，再决定哪些东西必须解耦、哪些东西不需要提前抽象。

例如搜索列表里，爆点可能是 API 调用前的参数组装，而不是列表渲染本身。一个合理设计可以让每个 Filter 自治产出 postValue，Controller 只管触发搜索，Pagination 作为带搜索语义的特殊 Filter。这样新增筛选项时只横向增加模块，不把逻辑堆回搜索主函数。

这条规则直接约束 Work Package：工作包不是“先做 list、再叠分页、再叠搜索、再叠筛选”，而是围绕 Design 确认的架构边界横向铺模块。

## 纵向模块与横向能力入口

AIPD 仍然需要同时区分“纵向概念模块”和“横向功能能力”：

- `_adoc/L3-core/vertical-concept-modules.md`：纵向概念模块，说明 L1-L6、OKR、Case、Work Package、Agent Entry、Agent 使用方案分别是什么。
- `_adoc/L3-core/horizontal-capabilities.md`：横向功能能力，说明 map 检索、Case 系统、Weave 反向编织、OKR 对齐、Agent Entry、未来上下文服务如何串起纵向模块。

核心判断：纵向模块负责“东西放哪一层”，横向功能负责“Agent 做事时怎么把这些东西串起来”。L1-L5 本身是项目讨论、判断和经验沉淀的文档层，同时也为 Agent 理解、管理和修改 L6 代码提供上下文。

## L1-L5 与流程状态隔离

L1-L5 是长期知识库，负责给 Agent 提供方向、外部世界、核心模型、产品功能和跨模块工程规则。它们不承载具体 case、当前 case 状态、OKR 执行状态或一次性执行过程。

Case / Work Package / OKR 是流程状态模块。只有创建、执行、恢复、归档 case，或明确查看 / 更新 OKR 时，才读取 `_adoc/case/` 和 `_adoc/okr/`。普通开发、找代码、查业务规则、查页面或组件实现时，应从 L3 / L4 / L5 / 局部 README 进入，再下钻到 L6 代码，不把历史 case 当作上下文来源。

## 对外教学三主线

AIPD 面向用户解释时，可以按教学主线讲清楚，而不是一上来展开所有纵向模块和横向能力：

1. **知识库最小闭环**：讲知识库本体、知识怎么存、知识怎么取。L1-L5 是知识库本体，L6 是被索引的真实代码；Weave / 反向编织负责维护，map 负责读取。
2. **Case / Work Package 开发逻辑**：讲短周期目标如何通过 Case Contract / Think / Design / Execute / Verify / Close 完成闭环。OKR、Case、Work Package、分身 Agent、执行记录、归档和 Close 归档候选都服务这条执行线。
3. **Think / Design 决策逻辑**：讲模糊想法和推进中的未知如何通过讨论、调研、方案比较、复杂度爆点识别和最小解耦设计，变成可执行工作包，或被 kill / defer / research / weave。
4. **SOP / Agent 原生程序**：讲项目里可重复执行的动作如何沉淀为以 Agent 为运行时的 AI 原生程序。SOP 可以调用代码和工具，但主体是目标、输入、步骤、分析判断、输出和收尾。
5. **AI 原生代码架构实验**：讲代码组织如何进一步适配 AI，包括纵向黑箱、降低牵连面和 Decouple first, DRY later。这是探索性主线，不应写成已经定型的硬规范。

这些主线是面向用户理解 AIPD 的教学结构，不替代七个 L3 核心成立模型，不替代 L1-L6 纵向概念模块，也不替代 map、Case、SOP、Weave、Agent Entry 等横向功能能力。

## L3 不等于狭义数据模型

L3 Core 的核心问题是“项目内部靠哪些稳定模型成立”，而不是“有哪些数据库表”或“有哪些核心对象”。

对很多业务系统来说，核心对象、数据模型和状态流转确实是 L3 的主体。但这只是常见形态，不是 L3 的边界。不同项目可以有不同类型的成立模型：

- 工具型产品通常依赖领域对象、核心流程和数据关系。
- 游戏、内容、媒体、社区或 SEO 型网站可能同时依赖产品模型、增长模型、内容模型、流量模型、留存模型和商业模型。
- AIPD 这类框架项目则依赖上下文解耦、黑箱上移、扁平化检索、Case / Work Package 和 Weave 等执行认知模型。

L2 负责记录外部世界，例如用户、场景、痛点、竞品、玩法范式、市场观察、流量来源和变现形态。L3 负责把这些外部观察抽象成项目自己的稳定成立方式。L4 再把 L3 的核心模型功能化，形成用户可见的产品能力和业务行为。

因此，判断某条信息能否进入 L3 时，优先看它是否已经从“外部观察”变成了“本项目长期遵守的内部模型”。如果它只是竞品事实、用户反馈或市场资料，先放 L2；如果它会长期约束多个产品功能，才沉淀到 L3。

## 上下文解耦

AIPD 的方向仍然是为项目架构一套知识库，增强 AI 干活时获取上下文、理解历史取舍和遵守执行经验的能力。

“上下文解耦”不是把 AIPD 从知识库方向改成另一套方向，而是 AIPD 在 AI 执行侧的一条核心认知：人与 AI 在编写任务、拆分任务、对齐目标之前，应尽量把任务设计成小而自足的上下文黑箱。

## 和记忆方案的区别

很多 Agent 记忆方案或流程方案，关注的是上下文关系串联：过去某一刻说过的一句话，未来某个阶段可以被提取出来，再和新的上下文建立关联。

AIPD 不否认这种能力有价值，但任务执行时不能主要依赖“未来能任意提取过去记忆”。更稳定的方式是：先通过项目知识库沉淀大规则、全局约定、领域语言、输入输出和历史取舍，再让每个具体任务拿到相对解耦的局部上下文。

## 任务设计目标

上下文解耦主要约束的是任务设计，而不是项目方向本身。

在创建 case、拆 work package、写上下文索引、派发分身 Agent 前，人与 AI 对齐工作的目标是：

- 把任务边界缩小到一个相对独立的黑箱。
- 写清楚这个黑箱需要遵守的大规则、输入、输出、验收口径和不做范围。
- 避免让执行 Agent 必须靠中短期记忆，把 A、B、C 多段聊天强行串起来才能开工。
- 让 Agent 在黑箱内部可以自由实现，只要运行结果符合输入输出逻辑。

这种取舍来自 AI 时代的工程现实：当模块足够小、边界足够清楚时，代码内部细节不一定要先追求最优。就像上层语言替代汇编、自动内存管理逐步变得可接受、前端页面体积约束随时代放宽一样，AI 写出的小块代码只要可运行、可验收、可替换，就有工程价值。有需要再调，没需要不调。

## 横向分层和纵向黑箱

上下文解耦不是在传统开发体系里增加一种新写法，而是改变代码组织的优先级。

传统开发体系通常优先做横向分层：controller、service、repository、model、util、component、store、api 等职责各自拆开，再通过调用关系织成网。它的主要收益是人类读写时能复用、归类、控制文件长度和统一实现。但对 AI 来说，这种结构经常把完成一个局部任务所需的上下文打碎：一个页面、接口或组件的输入、处理、业务判断和输出分散在多个文件中，Agent 必须来回跳转才能拼出完整链路。

AI 更适合纵向黑箱：一个页面、一个模块、一个组件、一个接口或一个 case step 尽量形成从输入到输出的自足上下文。一个接口可以是一个文件，也可以是一个文件夹里的三五个文件；关键不是物理文件数量，而是这个黑箱内部的文件只服务当前黑箱，外部不能随意引用它们作为共享能力。

因此，AI 友好的目标不是“代码无依赖”，而是“理解无跳转”。当 Agent 处理当前黑箱时，应尽量一次读到源头输入、中间处理、业务规则、用户可见输出和验收条件。文件变长、局部重复或目录内文件变多，不再自动构成坏味道；真正危险的是当前任务必须跨多个横向依赖才能理解，而且任一改动都可能牵出外部调用面。

## AI 时代的成本结构

AI 的 token 生成和批量修改成本远低于人类手写代码。代码量膨胀、局部重复和多消耗一些 token，通常是可后处理成本；牵连不确定性则会直接吞掉 AI 的优势。

传统工程里，“一个方法被多个地方引用”常被视为复用收益。但在 AI 执行场景中，如果这个方法携带隐含业务语义，改动它就必须审计所有调用方，局部任务会变成全局影响分析。AI 擅长在清楚边界内高速生成和修改，不擅长在隐性依赖网里保证每个外部使用者的语义都不被破坏。

因此，新的优先级应是：

- 先降低牵连面。
- 再控制代码量。
- 最后再抽取复用。

也可以概括为：宁可先膨胀，不要先耦合。局部黑箱稳定以后，再通过后置抽取、二次打包、索引、测试或基础能力沉淀降低 token 和代码量。

## Decouple first, DRY later

AI 友好开发不应把 DRY 放在第一阶段。更稳的顺序是先 Decouple，再 DRY。

前期允许不同页面、接口、组件或 step 在自己的黑箱内各写各的，哪怕出现一定重复。这样做的收益是每个黑箱都能独立读取、独立修改、独立验收，改一个不会牵出一片。等多个黑箱的重复逻辑经过实践证明稳定，再考虑抽成共享能力。

后置抽取不能只看代码相似度。业务逻辑经常是“现在看起来一样，未来会分叉”。只有当某段逻辑形成稳定底层能力，抽出去以后能降低理解跳转、降低牵连面，或提升独立验收和替换能力时，才值得横向沉淀。

因此，重复三次不自动证明应该抽象；调用次数多也不自动证明 helper 合理。抽象的判断标准不是减少字符，而是减少理解成本和外部牵连。

## 和面向指令 / 面向状态的关系

“横向分层 vs 纵向黑箱”不是“面向指令 vs 面向状态”的同义词。

面向指令和面向状态主要描述程序内部如何表达行为：是通过动作序列推进，还是通过状态及其派生结果组织。它们在旧开发体系和 AI 友好开发体系中都可以存在。

上下文解耦讨论的是另一条轴：代码和任务上下文应该如何组织，才能让 AI 稳定理解、修改和验收。一个纵向黑箱内部可以用面向状态的方式建模，也可以用明确指令流程实现。关键是当前黑箱的输入、状态、动作、输出和验收口径能否在局部上下文内读清楚。

## 决策杠杆与黑箱上移

AI 可以快速生成大量上下文，但系统生产力的瓶颈会逐渐转移到人的决策能力上。人不可能逐行、逐函数、逐文件审查 AI 产生的一切细节；如果决策仍停留在低层细节，算力越多，反而越容易把人拖入审批和理解成本。

AIPD 的上下文解耦，本质上是在把黑箱边界上移。

传统高级语言把机器码、内存细节和底层指令黑箱化，让人可以在逻辑层做决策。AI 协作进一步把局部实现细节黑箱化，让人可以在模块关系、输入输出、验收口径和系统约束层做决策。

抽象层级提高通常会牺牲一部分底层控制力，但换来更大的决策杠杆：同样一次人的判断，可以约束和产出更大范围的工作结果。

因此，AIPD 不应追求让人管理更多 AI 生成细节，而应追求：

- 黑箱边界清楚。
- 黑箱输入输出明确。
- 黑箱内部可替换、可验证。
- 人的决策位置持续上移。
- 单次决策影响的生产结果持续扩大。

这也是 AIPD 区别于普通工作流编排的地方：工作流是在固定路径里运行任务；AIPD 追求的是提高人类决策所在的抽象层级，让 AI 算力服务于更高层的创造性判断。

## 黑箱上移和封装的区别

封装是构造黑箱的一种工程手段，主要回答“如何隐藏内部细节，只暴露稳定接口”。黑箱上移是 AI 协作下的认知和决策策略，主要回答“人的决策应该停在哪个抽象层级，哪些细节不再值得人成为决策对象”。

因此，黑箱上移不等于传统封装。一个黑箱内部可以有封装，也可以没有封装；只要边界、输入、输出和验收条件清楚，它仍然可以作为上层决策对象。AI 协作下，黑箱内部甚至可以允许一定的复制粘贴，用局部上下文完整性换取 AI 阅读和修改效率。

AIPD 对封装的态度不是反封装，而是反对让执行 Agent 为了理解当前黑箱，不断跳转到外部封装里补上下文。封装只有在形成稳定底层能力、明显降低当前黑箱复杂度，或能提升独立验收和替换能力时才值得引入。

## AI 友好的局部代码组织

从黑箱上移和上下文解耦可以推导出一条工程倾向：AI 友好的代码不一定是最少重复、最多抽象的代码，而是当前黑箱内上下文贴得足够近、决策线索足够显性的代码。

在前端实现中，这意味着：

- 简单展示逻辑能在 `template` 内表达时，不必先挪到 `script` 再间接引用。
- 表格列如果每列都有业务微调，优先直接展开写在 `template`，不要为了“优雅”先拼列配置再 `for` 循环渲染。
- 拼 API 参数时，优先让变量声明、转换逻辑和使用位置贴近；不要把十几个 `const` 全堆在方法顶部，再在后面分散使用。
- 组件之间传递事件时，优先传原始值和触发时机，让接收方在自己的上下文里判断；避免直接传“你应该做什么动作”。
- 只有当某块逻辑已经形成独立决策单元，继续留在当前文件会破坏黑箱可读性时，才把它提升成新的黑箱。

这条规则的核心不是否定复用，而是不要挑战大模型的注意力。代码相关线索离得越远，AI 越容易漏读、错连或过度抽象。局部重复在传统工程里常被视为冗余，但在 AI 执行效率面前，只要范围可控、边界清楚、验收容易，它可以是可接受的成本。

优化局部代码组织时，要守住“源头-中间-结尾”的边界。源头通常是 API 返回、接口协议、用户输入或业务原始字段；结尾通常是用户可见的字段、列、文案、按钮、组件和交互结果。这两端默认是需求边界，不由 Agent 凭“看起来没用 / 可以更简洁”自行删除或改语义。AI 友好优化主要发生在中间链条：删掉多余的转译层、常量链、一次性 computed、字符复用型 helper 和远离现场的展示配置，但优化前后应保持同一批源头输入和同一批用户可见输出。

因此，“减少二级黑箱”不是“减少功能点”或“删除展示项”。例如把配置数组 `v-for` 渲染的表格列改成 `template` 里直接展开，目标是让每列字段、插槽、格式化和显隐条件更贴近现场；改完以后原本应该展示的列仍然应该存在。只有当需求、接口、调用关系或用户明确确认某个展示项已经废弃时，才删除源头字段、用户可见结尾或整块组件。

黑箱上移以后，还要避免在最小黑箱内部继续制造二级黑箱。以 Vue SFC 为例，当一个组件已经是当前实现的最小黑箱时，组件内部应优先保持逻辑透明。不要把简单元数据判断、一次性展示条件、短表达式包装成结论型 `computed`、`const`、helper 或常量链。

判断一个命名是否值得存在，不看它是否“语义化”，而看它是否减少读者跳转。如果读者看到变量名后仍必须跳到 `script`，再跳到常量，才能理解真实条件，这个命名就是二级黑箱。只有复杂兼容、重复多处、存在副作用隔离，或表达稳定领域概念时，才值得抽出命名。

因此，在最小黑箱内部不要用“避免魔法数字”“统一格式化”“微小性能优化”等传统理由，反向制造远离现场的常量链和泛用 helper。简单枚举值、一次性条件、空值兜底和短映射如果只服务当前组件，优先保持现场透明；只有当抽象让读者少跳一步时，它才是正收益。

审查 AI 友好代码时，应把“后续阅读和修改成本”当成一等优化理由。一个问题即使不会造成运行 bug，只要它让读者在模板、脚本、常量和 helper 之间多跳转，也应明确列为可优化项。不要因为运行风险低，就把明确的 AI 可维护性问题说成“不必改”。

调用次数多不自动证明 helper 合理。若一个 helper 的主要收益只是减少重复字符，而不是减少理解跳转，它属于字符复用型抽象；在 AI 友好代码里优先级很低。字段展示应优先让字段值、空值兜底、单位、枚举文案和格式化规则在当前行或相邻代码里可见。

对信息展示型组件，模板里的字段行就是主要业务内容，不要把“模板变长 / 字符变多”自动视为坏事。字段展示规则现场可见，通常比通过泛用 helper 减少字符更重要。审查这类组件时，可以承认内联会更长，但仍要判断它是否更适合 AI 后续修改。

## 提示词边界和模型思维方式

AIPD 早期尝试过一种思路：给 Agent 一套底层认知或 SOP，让它沿着这套认知改变后续发散路径。这个思路对人类协作经常有效，因为人可以在理解一个底层逻辑后，主动改写自己的判断起点，并代入对方的方法论继续推导。

但对大模型，这条路有明显边界。提示词不能重训模型参数，也很难真正改写模型的“第一反应”。模型的潜意识式优先级来自训练和当前推理机制；上下文提示只能影响当次注意力分配、候选答案排序和局部约束。若强行塞入一套和模型默认工程直觉冲突的长 SOP，模型未必会按这条链路思考，反而可能在多组规则之间打结：一边机械执行“透明化”，一边忽略需求边界，最终把“优化中间链条”误解成“删除看似无用的展示结果”。

因此，AIPD 对提示词的定位应更克制：

- 底层认知可以写，但主要用于说明方向，不假设它能稳定改造模型潜意识。
- SOP 可以用，但更适合短链路检查和输出验收，不适合承担“让模型换一种思维方式”的目标。
- 真正稳定的提示词资产通常是边界、反例、禁区和验收口径，而不是长篇推理教程。
- 与其要求 Agent “像人一样理解这套哲学”，不如顺着模型已有工程直觉，在容易误判的位置加护栏。

这并不否定“修改原点会改变发散树”这个方法论本身。它对人类、组织共识和可训练系统依然成立；只是当前不可训练的大模型 Agent 不是这种对象。对它们来说，更现实的策略是：短原点提供方向，硬边界限制坏结果，反例锚点抵消默认直觉，输出验收暴露偏差。

这也解释了为什么成熟 Agent 提示词往往包含大量边界判断。那不一定是因为边界式提示词更优雅，而是因为在无法重训模型的条件下，边界、反例和验收比“试图改造思维方式”更稳定。

## 组件关系类比

如果一个组件开关会影响另一个组件：

- 强耦合做法是，第一个组件直接调用第二个组件的“受影响方法”。第二个组件不知道谁控制了自己，关系被写死在调用链里。
- 上下文解耦做法是，第一个组件发布自身状态或事件，第二个组件在自己的上下文里判断是否隐藏、展示或更新。

事件广播本身可能带来可预测性问题，但它让局部上下文更独立。AIPD 的任务拆分也是类似思路：全局规则负责提供共同约束，局部任务负责在自己的黑箱内完成输入输出。

## 对 AIPD 的要求

AIPD 后续设计 case、step、map、局部 README 和分身 Agent 派发时，应优先追求：

- 全局规则清楚。
- 局部上下文自足。
- 任务输入输出明确。
- 跨模块关系显式。
- 对中短期隐式聊天记忆的依赖最小。

## 扁平化检索

扁平化检索是上下文解耦在项目记忆读取侧的延伸。

AIPD 的项目记忆不能只考虑“人类怎么维护”，还要考虑“AI 怎么读取”。传统目录树适合人类维护，但多跳检索对 AI 不稳定。如果一次任务需要按 `总 map -> L4 map -> feature map -> 代码入口` 连续跳转，任何一跳命名抽象、索引不完整或没有被读取，Agent 都可能中断路径并开始猜。

AI 更适合一次读取一个相对厚、结构化、信息密度高的检索总图，再按需进入细节 map。100K 以内的结构化索引对很多 AI 任务仍然可处理，而多个 10K 文件的连续跳转，表面上更省上下文，实际链路更脆弱。

因此 AIPD 的 Memory Retrieval 应采用“大地图 + 细节 Map”的解耦结构：

- 大地图负责高频入口、关键词、功能线、核心概念、工程规则和资源入口的扁平索引。
- 细节 Map 负责某个功能线、概念或规则域的完整资源索引。
- 大地图可以适度重复细节 Map 的稳定入口，用重复换取第一跳命中率。
- 不要求 Agent 通过多层目录链自行发现关键文档。
- 人类维护路径可以分层，AI 读取路径要尽量一跳命中。

这条认知和上下文解耦的底层原则一致：把 AI 必须依赖的上下文关系显式化、扁平化、可一次性读取，而不是藏在连续跳转链条里。

## 代码阅读器观察

AI 当前阅读代码时，经常仍然沿用人类 IDE 的文件跳转模型：先读一个文件，再根据 import 逐个打开依赖文件。但 AI 更需要的是任务相关的上下文切片，而不是完整文件树。

理想的 AI 代码阅读器可以根据当前文件的实际引用关系，只展开被使用的函数、类型、常量和业务逻辑；未使用内容先折叠或摘要。这样 Agent 读一个文件时，可以同时看到必要依赖，而不是在多个文件之间来回跳转。

这个方向暂不作为 AIPD 的近期实现目标。它当前实现成本高，对近期产出提升不直接；先记为观察项：实现不了，却可能有点用，哪天闲了再试，或搜搜有没有现成方案。

该工具如果未来实现，需要避免把小文件无限展开成巨大上下文包。它应有预算和边界：默认只展开实际引用、一层优先、超过预算则摘要，并允许 Agent 按需继续展开某个 symbol。

## 对 AIPD 检索设计的要求

AIPD 后续设计 map、aipd-case 和 update 时，应区分“存储结构”和“读取结构”：

- L1-L5 是项目记忆的存储结构，负责让认知分层、有边界、可维护；L6 是具体代码实现，不在 `_adoc/` 内单独建层。
- map 是项目记忆的读取结构，负责让 AI 从用户自然语言快速命中应读上下文。
- `_adoc/map.md` 应作为检索总图，承担高频入口的扁平索引。
- L3 / L4 / L5 / feature / 局部 README 的 map 可以存在，但不应成为 AI 找到关键入口的唯一必经链路。
- 如果某条功能线、核心概念或工程规则经常被用户提起，它应该在总 map 中有直接入口。
- case 的上下文索引是一次任务的临时压缩路径，应从总 map 和相关细节 map 中提炼，不重新发明一套孤立索引。
