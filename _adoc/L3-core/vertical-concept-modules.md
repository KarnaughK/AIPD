# AIPD2 纵向概念模块

本文记录 AIPD2 的纵向概念模块。纵向模块回答的是：项目认知、执行状态、代码实现和 Agent 协作规则应该按什么类别沉淀。

这些模块不是同一个目录树里的同质文档，而是 AIPD2 解释项目的分类轴。每个模块相对独立，也会被横向功能串起来使用。

## 核心判断

纵向模块负责“东西放哪一类”。它们本身可以用于讨论沉淀、判断记录和经验回写，也可以给 Agent 理解、管理和修改 L6 代码提供上下文。

L1-L5 不是 L6 的纯索引层。L1-L5 本身就是项目认知文档；L6 是真实代码实现，不在 `_adoc/` 内单独建层。

## 模块清单

| 纵向模块 | 核心问题 | 典型载体 |
|---|---|---|
| L1 Intent | 项目为什么存在，长期方向和取舍是什么 | `_adoc/L1-intent/` |
| L2 Research | 方向所处的外部世界是什么，包括用户、场景、需求、痛点、竞品、玩法/行业范式、市场观察和调研结论 | `_adoc/L2-research/` |
| L3 Core | 项目内部靠哪些核心模型成立，包括核心对象、领域语言、核心关系、核心流程、产品成立模型、增长模型或商业模型 | `_adoc/L3-core/` |
| L4 Product | 核心模型落成哪些产品功能、功能边界、业务规则和用户可见行为 | `_adoc/L4-product/` |
| L5 Dev | 产品功能落到代码时，跨模块、跨平台、跨页面都要遵守的业务实现逻辑、工程规则和协作约定是什么 | `_adoc/L5-dev/` |
| L6 Code | 真实代码实现在哪里、怎么工作 | 项目源码目录，不建 `_adoc/L6/` |
| OKR | 当前阶段目标、关键结果和判断标准是什么 | `_adoc/okr/` |
| Case | 一次具体事项的目标、边界、上下文索引和验收状态是什么 | `_adoc/case/{case}/case.md` |
| Step | Case 内的执行、派发、恢复和验收单元是什么 | `_adoc/case/{case}/steps/` |
| Agent Entry | Agent 进入项目时先读什么、按什么规则行动 | `AGENTS.md` |
| Agent 使用方案 | 主 Agent、分身 Agent、角色 Agent 如何协作 | `AGENTS.md`、平台 agent 配置、L5 文档 |

## L2 不是单纯痛点层

L2 Research 负责记录方向所处的外部世界。痛点是 L2 的一种产物，但不是 L2 的全部。

适合放入 L2 的信息包括：

- 用户、场景、需求和痛点。
- 竞品、同类产品、同类玩法、行业范式和市场观察。
- 用户现有替代方案、流量来源、变现方式、内容供给方式等外部事实。
- 从外部资料中形成的调研结论和机会判断。

例如游戏网站中，同类猜词网站、流行玩法、SEO 入口、广告形态和玩家回访方式，首先属于 L2 的外部观察。它们可以影响 L3 和 L4，但不应直接混同为项目内部模型。

## L3 是项目成立模型层

L3 Core 不应被窄化成“数据对象层”。核心对象、数据模型和状态流转是 L3 的常见载体，但 L3 更本质的问题是：这个项目内部靠哪些稳定模型成立。

不同类型项目的 L3 重点可以不同：

- 工具型 / 业务型产品：常常偏核心对象、领域语言、核心流程和数据关系。
- 游戏、内容、社区、媒体或 SEO 型产品：除了产品核心模型，还可能包含增长模型、内容模型、流量模型、留存模型和商业模型。
- 框架型项目：可能偏上下文模型、执行模型、检索模型、记忆回写模型等底层认知。

判断某个信息是否属于 L3，可以问：

- 它是否已经从外部观察抽象成“我们自己的成立方式”。
- 去掉它，项目是否仍然成立。
- 它是否会长期约束多个 L4 功能，而不是只服务某个功能点。

例如猜词游戏的谜底词、猜测、反馈规则和玩法模式是产品核心模型；如果项目要作为商业网站成立，SEO 流量模型、回访模型和广告/订阅变现模型也可能成为 L3 的核心成立模型。

## L4 的双重职责

L4 Product 既解释产品功能本身，也可以按产品功能串联 L6 代码入口。

因此 L4 不应被写成单纯的代码地图。一个 L4 功能线至少要说清：

- 这个产品功能是什么。
- 它解决什么用户场景。
- 它的功能边界和不做范围是什么。
- 它依赖哪些 L3 核心模型、核心对象、领域语言或状态流转。
- 为了实现这个功能，相关页面、接口、数据对象、权限和 L6 代码入口在哪里。

L4 是 L3 的功能化结果。例如 L3 中的 SEO 流量模型，到了 L4 可以落成多语言落地页、词包页面、每日挑战分享、sitemap 生成和 meta 信息策略等产品功能或业务行为。

## L5 的双重职责

L5 Dev 既沉淀业务实现逻辑和跨模块工程规则，也为 Agent 修改 L6 提供稳定上下文。

L5 不只是“研发规范”，也不是远离代码的万能实现地图。它更适合承载：

- 权限、路由、菜单、按钮显隐等跨模块规则。
- 第三方插件、前后端约定、平台适配等稳定实现逻辑。
- 某类产品功能落到代码时必须遵守的业务实现规则。
- 调试经验、工程约束和通用 SOP。

页面、弹窗、组件内部的最后一层实现地图，优先放代码目录就近 `README.md`。

因此 L5 map 只记录跨模块、长期稳定、需要多个代码入口共同遵守的规则。某个页面、弹窗、组件、脚本或功能局部的组件关系、数据流和入口文件，应该贴近 L6 代码沉淀到就近 README 或局部 map，而不是大老远塞回 L5。

## 与横向功能的关系

纵向模块只负责分类和沉淀。Agent 真正做事时，需要横向功能把多个纵向模块串起来。

例如：

- map 检索会从用户自然语言串到 L3 / L4 / L5 / 局部 README / L6。
- Case 创建会从目标、边界、OKR、L1-L6 上下文中提炼 case 和 steps。
- Case 执行会按 case / step 恢复任务状态，再读取必要的 L3-L6 上下文。
- 经验回写会把执行中产生的新判断写回 L3 / L4 / L5 / 局部 README / map。

详细横向功能见 `_adoc/L3-core/horizontal-capabilities.md`。
