# 分支：2026-07-10 近期 Codex 聊天审计

## Branch goal

从今天最近且可访问的 Codex 会话中抽取有限样本，识别 Agent 实际动作、必要工作、潜在冗余和可优化点。

## Trigger

来自 Case Contract；用户明确要求“看一下最近的几个聊天都做了什么内容”。

## Scope / stop condition

- 优先最近 3-6 个有实质内容的会话；至少覆盖开放讨论与工程 / 流程任务（若样本存在）。
- 只读取与任务、工具动作、等待、重复和最终产出有关的消息，不复述完整对话。
- 能形成稳定工作分类和至少 3 个有证据的效率观察后停止；缺失精确耗时 / token 时明确标记。

## Evidence

### 数据口径

- 数据源：`~/.codex/state_5.sqlite` 与 2026-07-10 对应 rollout JSONL。
- 样本：今天 4 个已经完成实质工作的 root 会话；全部为 `gpt-5.6-sol / ultra`。
- `tokens` 是 Codex 累计 accounting，包含每次模型 / 工具往返时重复携带的输入和 cached input；不能直接当成新鲜生成 token 或 API 账单。
- 完整 turn 的首个可见 token 多在 2.7-9.3 秒出现；慢主要发生在后续持续推理、工具调用、子 Agent 协调和汇总阶段。

### Agent 实际做了什么

1. **确认文档化记忆共识**：认知核对、授权政策跨文件审计、构建安装验证、Source Graph 设计和 ADOC 路径复盘。
   - 必要工作：授权政策审计、构建安装验证、硬依赖 / 语义依赖区分。
   - 潜在冗余：同一文件优先问题派出 3 个重叠 Agent；Main 又重复扫描；同一长 Skill 在多轮 follow-up 中反复读取；部分 Agent 被追问 3-7 轮。
2. **梳理项目唯一入口**：项目认知扫描、助手工作空间设计、Codex 请求链、CLIProxyAPI、Hermes / GA 适配核查。
   - 必要工作：CPA request path、Hermes auth、GA provider 三条证据线相对独立。
   - 潜在冗余：Main 与 child 同时大规模检索；一个讨论型 API 问题被扩成多个项目的实现级代码考古。
3. **检查项目状态**：git 状态、安装产物比对、Case 归档、提交与推送。
   - 大部分动作必要。
   - 但四轮显式工具 wall time 约 1.2 / 0.6 / 1.1 / 10.7 秒，对应 turn 为 116 / 74 / 231 / 150 秒；由差额推断，瓶颈主要不在 shell，而在模型往返和长上下文重放。
4. **整理项目并对齐 OKR**：项目认知、代码现实、远端 OKR 三线汇总和产品边界讨论。
   - 首轮三线并行合理。
   - 后续纯产品讨论再次派两个重叠 Agent，边际收益有限。

### 跨样本模式

- Main 和 child 经常同时调查同一证据面，没有形成 single owner。
- child 常被多轮 follow-up，而不是一次返回压缩结论。
- 最重会话的 Main 约有 68 次 `spawn / list / wait / send / followup` 协调调用；协调本身形成大量模型回合。
- 长入口 Skill 在每轮重新触发时被完整读取，说明真正要瘦身的是路由入口，而不是让 Agent“读快一点”。
- 开放讨论经常被升级成实现级审计，即使本轮最终不改文件。

### 与 GPT-5.5 High 的有限对照

最近若干 `gpt-5.5 / high` 任务的完成 turn 中位数大约为 31-91 秒；今天 Ultra 为 158 秒，方向上明显更慢。但对照任务、AIPD 规则和是否启用子 Agent 都不同，不能把差异单独归因于 Sol 模型或 Ultra，必须做同 prompt A/B。

## Conclusion

- 用户对“数分钟才完成”的体感有日志支持；这不是首 token 慢，而是 Agent 持续展开工作、反复携带上下文和等待并行结果。
- Ultra 的质量收益确实用于了跨项目、跨证据线的深度工作，其中不少结果有价值；问题是它被设成全局默认后，也会把状态查询、已知路径核对和开放聊天推向深度执行。
- 最优先优化的不是禁止子 Agent，而是：证据面单一 owner、0 / 1 / 2-3 级 fan-out、child 单轮返回、减少轮询 / follow-up、讨论任务先给当前结论再决定是否深挖。
- 本轮自身也复现了问题：3 个普通 child 使用 `fork_turns="all"`，并继承 Sol Ultra；这应成为后续调度规则的直接反例。
