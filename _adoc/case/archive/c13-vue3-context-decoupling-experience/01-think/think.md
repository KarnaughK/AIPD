# Think：Vue3 上下文解耦实践经验提炼

## 当前问题

用户已经明确要沉淀的五个方向，但正式经验应拆成一条还是多条、每条的读取时机、四个组件的真实协议、ModalBox 命名迁移影响面，以及目标项目 L5 中还有哪些稳定经验，都需要先由代码和项目认知证据校准。

## 调研边界

### 查什么

- `QlmModalBox` 的实现、README / 注释、导出方式、注册方式和多个真实调用案例。
- `provider-hook.md` 以及实际 provide / inject 使用，校准它与 props / emits、列表项通信的边界。
- `QlmForm`、`QlmSearch` 的实现、配套 hooks / controller、示例页面和文档。
- 目标项目的页面架构图 / Mermaid 文件、页面 README 与其实际覆盖情况。
- `_adoc/L5-dev/` 中与 Vue3 前端实践有关的稳定规则和额外经验候选。
- package scripts、类型检查、测试和构建入口，为后续 Verify 做准备。

### 不查什么

- 不扫描与这五条主线无关的全部业务代码。
- 不把业务页面的私有字段、接口或权限规则提炼成 AIPD 通用经验。
- 不在 Think 阶段改代码或正式经验库。

## 预期输出

- 四个核心方向的事实证据、调用路径和关键文件。
- ModalBox 改名的完整影响面及兼容策略判断。
- 页面架构图应复制规则还是建立索引的判断。
- L5 额外候选清单及收录优先级。
- 一条或多条经验的拆分建议，作为 Design 输入。

## 停止条件

- 每个必查方向至少有实现证据、使用证据和边界判断。
- `QlmModalBox` 的所有仓库内引用已可枚举。
- L5 文件已完成按标题 / 关键词的全量盘点，并对相关文件做必要精读。
- 已能形成无需继续盲扫的经验结构方案。
