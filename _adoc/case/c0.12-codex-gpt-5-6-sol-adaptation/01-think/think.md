# Think：Codex GPT-5.6 / sol / 极高适配

## 调研前 checkpoint

- **当前问题**：用户观察到新模型“极高”模式在开放问题上常需数分钟；需要判断质量收益是否值得，并找出 AIPD 可减少无效深思考的机制。
- **触发来源**：新建 Case，不是后续 phase 回跳。
- **范围**：有限抽样 2026-07-10 最近几次可访问 Codex 聊天；核验官方 / 本地可观察模型事实；审计 AIPD Agent Entry、Case、Learn 和 Codex Agent 调度规则。
- **不查**：不全量分析历史会话；不猜测未公开内部链路；不做长时间重复 benchmark；不修改源码。
- **预期输出**：会话工作分类与耗时源、事实边界、模型使用决策树、3-6 条 AIPD 优化方向和最小验证计划。
- **停止条件**：30 分钟到时，或已经取得足以回答“默认用什么、何时升级、AIPD 改哪里”的证据，以先发生者为准。
- **结论回流位置**：先回 `case.md` 的 Think 摘要与 Close 候选；用户确认后再进入 Design。

## 关键问题

1. 今天最近几次对话里，Codex 实际把时间花在了哪些工作上？
2. 哪些动作是任务本身必要，哪些可能由过深推理、重复检索、方案展开或主 Agent 串行执行造成？
3. “GPT-5.6 / sol / 极高”中哪些是公开 / 可观察事实，哪些只是推断？
4. AIPD 应如何按任务价值、歧义、错误代价和可验证性选择 reasoning 深度？
5. 应把优化放在项目 `AGENTS.md`、通用 Agent Entry、Codex 平台规则、skill 还是配置层？

## 选项比较

| 选项 | 速度 / token | 质量与覆盖 | 适用性 | 判断 |
|---|---|---|---|---|
| A. Sol Ultra 全局默认 | 最慢、child 和上下文重放最多 | 对高价值、可并行复杂任务最强 | 与高频互动和 AIPD 积极调度叠加后过重 | 不推荐 |
| B. Sol High 默认，按任务升 xhigh / max / Ultra | 平衡；与用户 GPT-5.5 High 习惯最可比 | 保留 Sol 主模型质量，必要时升级 | 能先分离“模型慢”与“Ultra 调度慢” | **推荐** |
| C. Terra Medium / High 默认，Sol 按需 | 日常最快 / 更省 | 复杂开放问题可能需要升级 | 若用户把响应速度放在第一位，适合作第二阶段 | 候选 |

### 推荐任务路由

| 任务形状 | 推荐 model / effort | child 策略 |
|---|---|---|
| 单一事实、已知文件、状态 / Git、短讨论 | Terra Medium 或 Sol Medium / High | 0 child |
| 日常开发、普通分析、单领域多入口 | Sol High | 0-1 个独立 retriever |
| 高风险但不可自然拆分的架构 / 算法判断 | Sol xhigh；测得收益后才 Max | 不为了档位强行拆 child |
| 两个以上独立研究 / 审查 / 验证包，允许后台数分钟 | Sol Ultra，或 Sol High Main + Terra workers | 首轮 2-3 个正交 child，禁止重复证据面 |

注意：`xhigh`、`max`、`ultra` 必须按内部值书写；简中客户端把 xhigh 与 Ultra 都显示成“极高”，只写中文会继续混淆。

## 决策结论

1. **不要把 Ultra 设为日常全局默认。** 它不是普通“更认真一点”，而是最大推理叠加自动委派；今天的慢来自模型深思考、工具往返、19 个 child、重复上下文和尾部等待共同作用。
2. **立即可执行但本轮不代用户修改的选择**：保持 `gpt-5.6-sol`，先把默认 effort 改回 `high`；用相同任务与 `medium` 对照。只有可明确拆成独立工作线的深度任务才手动选 Ultra。
3. **AIPD P0 方向**：加入 fast / balanced / deep 预算、single-owner evidence、0 / 1 / 2-3 fan-out、最小 fork、child 角色 model / effort、一次性压缩返回和协调停止条件。
4. **AIPD P1 方向**：把常驻 `AGENTS.md` 压成路由与硬边界，Case / Learn / Codex 细节渐进加载；对方案比较加 Gate 和 timebox。
5. **先做 A/B 再改大规则**：今天样本能证明“当前组合慢”，不能单独证明 Sol 模型本身慢，也不能证明 Ultra 的质量增益是否值得。

### 30 分钟后的推荐执行顺序

1. 用户回来后先确认是否接受“Sol High 默认、Ultra 按任务开启”。
2. 进入 Design，只设计 P0：模型 / effort 路由、child fan-out、fork 与 evidence owner；不一次性重写 checkpoint / Verify。
3. 建一个小型 transcript 评测 SOP，重放 8-12 个代表任务。
4. A/B 证明 P0 有效后，再做 Agent Entry / skill 瘦身；最后评估 Context Receipt 和证据复用。

## 未解决风险

- 本地 transcript 能给出客户端时间戳和 token accounting，但没有服务端每个内部阶段的精确耗时，也不能把 accounting 直接当成计费 token。
- 简中“极高”重名是当前客户端可观察行为，后续版本可能修正；规则应绑定内部值而不是 UI 文案。
- 今日样本以开放问题为主，不能直接代表常规代码修改任务。
- AIPD prompt 只能约束外部行为，不能保证减少模型内部不可见推理；因此 config / agent TOML / runtime fan-out 必须与文字规则一起验证。

## 2026-07-14 短暂重开与修正结论

### 触发与边界

- **触发**：用户复盘 21 分钟真实执行轨迹后指出，Case / Goal 已经能够承担长期状态恢复；子 Agent 不应继续被当成所有检索、实现和 Work Package 的默认执行者。
- **本轮不再调研**：已有 transcript 审计和现行规则 diff 足以支持方向修正；继续增加样本不会改变当前最小改动。
- **停止条件**：明确状态分解与运行时调度的关系、子 Agent 的两个核心价值，以及浏览器场景的暂定边界后，直接返回 Design。
- **返回位置**：`02-design/design.md`。

### 修正后的概念模型

1. **Case / Work Package 负责状态与验收分解**：它们决定目标怎样持久化、恢复和验收，不直接决定由谁执行。
2. **Main / Child 属于运行时选择**：一个内聚模块或高耦合工作包可以由 Main 连续完成；不能因为已经拆出 Work Package 就默认创建子 Agent。
3. **子 Agent 的上下文隔离价值**：适用于长文档、长日志、大量页面结构、批量扫描等高噪声过程；必须最小化继承上下文、由唯一 evidence owner 承担，Main 不重复调查，只接收压缩结果。
4. **子 Agent 的并发加速价值**：只在存在两条以上真正独立、可同时推进且协调成本低于时间收益的工作线时成立。
5. **主线耦合是反向信号**：需要持续继承刚完成的设计、代码和调试判断时，由 Main 连续执行通常更快、更稳定。

### 浏览器场景暂定边界

- 浏览器操作属于高噪声、外部状态持续变化的特殊场景，但当前不固化秒级超时、重试次数或完整 SOP。
- 已有稳定 SOP 时可以优先交给子 Agent 执行；新流程、异常状态或路径不确定时，先与用户沟通再决定由 Main 探索还是派发，不允许无边界深入和盲目绕路。
- 具体异常阈值在真实执行反馈中补充，不阻塞本轮调度规则落地。

### 覆盖原结论的最终决定

- 保留“Sol High 作为日常基线、Ultra 按任务开启”的模型建议。
- 将原来的“复杂检索 / 文件修改 / 每个 Work Package 默认派发”改为收益判定：只有**上下文隔离收益**或**真实并发收益**明确，并且大于启动、继承、协调、等待、合并和重复工作的成本时，才优先派子 Agent。
- Work Package 与 Agent 派发正式解绑定；Case / Goal 提供恢复能力后，Main 可以直接完成一个内聚模块并在里程碑写回状态。
- A/B 评测从前置门槛降为后续 Verify 手段；先应用已确定的最小规则，再根据真实执行反馈修正不确定细节。
