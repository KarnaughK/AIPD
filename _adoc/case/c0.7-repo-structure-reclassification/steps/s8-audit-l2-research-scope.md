## 4. L2 应围绕的核心痛点与需求线

### 4.1 解决 AI Agent 的上下文问题

当前 AI Agent 已经能搜索、读文件、跑命令和修改代码，但它很少能在查询完一个东西后主动丢弃查询过程，只保留稳定结果上下文。

这带来两个问题：

- Agent 为了找到正确代码和背景，会吞下大量过程上下文。
- 下一次任务仍然需要重新搜索、重新判断、重新踩坑。

所以 AIPD 需要建立项目知识库，把“查询后的稳定结果”沉淀下来，解决 Agent 寻找代码、理解边界和继承经验困难的问题。

### 4.2 解决项目文档完整性问题

项目知识库不只是 AI Agent 时代才出现的问题。

在人类开发时代，一个项目也很难长期拥有完整、可读、可持续维护的知识库。常见情况是：

- 文档只记录某个阶段的版本，不断叠加，而不是形成项目完整描述。
- 很多历史原因、业务边界和产品取舍藏在代码、聊天、Issue、PR 或个人记忆里。
- 后来的人或后来的 Agent 很难知道“为什么现在是这样”。

因此，AIPD 的第二条痛点线是：如何通过知识库持续迭代维护项目描述，并让知识库成为上游事实源，反过来约束和解释后续实现结果。

### 4.3 解决 AI 长时间工作的任务连续性问题

AI 可以一次性处理很多局部任务，但长时间工作会遇到上下文压缩、聊天分叉、执行记录散落、目标漂移和中途恢复困难。

这不是单纯的文档问题，而是任务组织问题：

- 一个大事项不能只靠单条聊天推进。
- 执行过程需要目标、边界、上下文索引、step、验收标准和回写位置。
- 人和 AI 都需要知道当前做到哪里、为什么做、下一步接哪里。

因此，AIPD 需要用 OKR、Case、Step、执行记录、分身 Agent 和 Weave Candidate，解决 AI 长时间工作和长任务恢复的问题。

### 4.4 解决 AI 开发代码时的架构无限拓展问题

现代软件开发不再适合先把全部需求想清楚，再一次性设计开发。真实项目往往是先上线一部分功能，看反馈，再持续加工。

在人类主导的代码架构里，很多结构是边写边改形成的；这种结构在 AI 深度参与后会暴露新的问题：

- 横向分层和过早复用会让上下文被打散。
- 共享 helper / service / 组件容易形成隐性依赖，改一个地方牵出一片。
- AI 读取代码时需要跨很多文件拼上下文，漏读和误判概率升高。
- 项目越迭代，越需要一种能持续扩展、又适配 AI 上下文读取能力的代码组织方式。

因此，AIPD 的第四条痛点线是：如何让代码架构在敏捷迭代和 AI 协作下继续扩展，重点从传统“先抽象复用”转向上下文解耦、纵向黑箱、降低牵连面和局部可检索。

这条线后续需要有技术栈维度的沉淀方式。可以按技术名称建立目录，例如 `Vue`，在目录下沉淀该技术体系里的 AI 原生开发经验：

- 如何画该技术体系的 MMD / Mermaid 架构图。
- 哪些架构代码更适合 AI 理解、修改和扩展。
- 哪些组件、目录组织、provider、状态组织或局部 README 写法更适合 AI 协作。
- 相关生态技术可以复用同一技术体系经验，例如 Nuxt 可以读取 Vue 目录中的通用经验。

这部分不是 L2 的具体实现规则，但 L2 需要记录这个需求来源：AI 原生代码架构不是抽象口号，需要按技术栈积累可复用经验库。

### 4.5 借助 Agent 行动能力沉淀 AI 程序

除了痛点，AIPD 还需要承接一个新的能力机会：更充分地借助 AI 的智能能力，以及 Agent 叠加 AI 智能能力后形成的强行动能力。

这种能力不完全等同于传统软件开发里的代码自动化。Agent 可以用较低的研发成本、较高的运行成本，产出一种可执行的 AI 程序。它离开 LLM / Agent 运行时就无法运行：执行主体不是代码运行时，而是 LLM 或 Agent 根据步骤、上下文、判断条件和工具调用来完成任务。

因此，AIPD 需要用 SOP 目录记录这些 AI 层面的 Agent 运行时工具功能。它们可以承接过去繁琐、重复、但很难完全写成传统脚本的任务，例如上线流程、日报、检查、整理、审计、发布前确认等。

这条线不一定是“痛点”，更像是 AI 时代出现的新需求：如何把 Agent 能做的复杂操作固化成可复用、可恢复、可迭代的 AI 原生程序。

### 4.6 未整理信息的暂存与分流

AI 协作过程中会产生大量未定型信息：灵感、待确认问题、临时材料、半成型判断、可能的 case、可能的知识库条目。

这些信息既不适合立即写进长期知识库，也不一定已经够资格创建 case。如果没有暂存区，它们会继续散落在聊天里，后续很难找回，也会干扰稳定认知。

因此，AIPD 需要 Inbox 这类中间层，用来临时接住未整理信息，再判断它后续应该 weave 进知识库、转成 case、进入 SOP，还是直接丢弃。

### 4.7 AI 工作的控制、验收和信任

AI 能力增强后，另一个需求是人如何控制、验收和信任 AI 的工作结果。

这包括：

- 人如何知道 AI 做了什么。
- 哪些结论只是一次性过程，哪些应该沉淀。
- 哪些改动已经验收，哪些还需要回看。
- 哪些经验可以进入 Weave Candidate。
- 什么时候可以归档、提交或进入下一步。

这条线对应 Case 验收、Step 执行记录、Weave Candidate、case-archive、git push 等能力。它解决的不是让 AI 更能干，而是让 AI 做完之后，人能收得住、验得清、信得过。

### 4.8 AIPD 框架自身的经验回流

AIPD 不只是给业务项目沉淀知识，它本身也是一个会在真实使用中不断暴露问题的框架。

真实协作里会出现：

- skill 行为不符合用户真实工作流。
- case-create / case-run / weave 的边界需要调整。
- Agent 没有按预期读取 map、case 或 SOP。
- 用户在对话里反复纠正同类问题。
- transcript 中暴露出框架规则、模板或 Agent 指引缺口。

因此，AIPD 需要 `aipd-learn` 这类框架自迭代能力，把真实使用经验回流到 AIPD 的 skill、模板、Agent 行为规则和项目认知中。

### 4.9 已接入项目的 AIPD 架构升级迁移

AIPD 这个项目本身会持续迭代：Agent Entry、map 结构、case 模板、L3-L5 边界、SOP 约定、skill 行为规则都可能升级。

这会带来一个独立需求：已经接入 AIPD 的项目，如何在 AIPD 架构演进后快速、安全地更新到新规则。

这个更新不能简单覆盖项目文件，因为项目自己的 L1-L5、case、map、局部 README 和历史经验都可能已经形成稳定认知。它需要审计差异、识别旧结构、给出更新方案，并在用户确认后合并新模板和新规则。

因此，AIPD 需要 `aipd-update` 这类能力，让已有项目可以跟随 AIPD 框架持续升级，同时不破坏项目已有认知。

### 4.10 桌面端文件查询与观看需求

随着 AIPD 使用变大，配套文件会越来越多：L1-L5、case、step、OKR、Inbox、SOP、MMD、docs、局部 README、执行记录、参考资料。

只靠普通文件树或聊天窗口会越来越难快速查询、定位和观看这些内容。用户需要一个更贴近 AIPD 组织方式的桌面端：

- 快速浏览 AIPD 文件树。
- 按 L1-L5、Case、Step、OKR、Inbox、SOP 组织入口。
- 快速查看 MMD、Markdown、case、step 和参考资料。
- 在聊天时快速引用当前正在看的对象。

这条线是 AIPD Desktop 的需求来源之一。它不是为了做完整 IDE，而是为了让 AIPD 文件体系随着规模扩大后仍然可查、可看、可用。

### 4.11 CLI Agent 桌面化使用需求

当前最强的编程 Agent 很多仍以 CLI 或接近 CLI 的形态存在。Codex 虽然有桌面端，但主要组织形式仍是工作空间下的对话列表；它没有按 AIPD 的 L1-L5、Case、Step、SOP、MMD 等工作对象组织界面。

这和“文件查询与观看需求”有关，但不是同一件事：

- 文件查询与观看解决的是 AIPD 文件体系变大后如何快速看材料。
- CLI Agent 桌面化解决的是强 Agent 的主要交互形态仍偏终端 / 对话列表，不适合承载 AIPD 的项目组织方式。

因此，AIPD Desktop 还需要承担一层“桌面化 Agent 使用容器”的需求：保留 Codex / Claude Code / OpenCode 等强 Agent 的执行能力，同时提供更适合 AIPD 的输入、对话、上下文引用和项目对象组织方式。

### 4.12 人与 AI 的高效交流与 MMD 表达

当 AIPD 的工作效率提高后，新的瓶颈会变成人和 AI 的交流速度。

很多复杂结构只靠文字描述会很慢，包括：

- 架构关系。
- 模块边界。
- Case / Step 拆解。
- 数据流和状态流。
- Agent 协作路径。
- 桌面端或产品结构。

MMD / Mermaid 可以把复杂关系压缩成图，让人和 AI 更快对齐结构、发现缺口、讨论边界。这条线不是单纯“画图好看”，而是把图作为人机协作中的高带宽表达方式，降低沟通成本和误解成本。

## 5. 已有材料中属于 L2 的稳定线索

### 5.1 用户与使用者画像

当前 L1 已确认 AIPD 尤其适合：

- 一人公司。
- 单人开发者。
- 一个人配合 AI 开发大型项目的场景。
- 已经让 AI 深度参与开发，但发现上下文管理比生成代码更难的项目。

这些属于 L2 的用户画像 / 使用场景，应进入 L2，而不是只留在 L1 的适用边界里。

### 5.2 场景变化：从 Vibe Coding 到 Agent Coding

docs 已经描述了外部开发方式变化：

- 早期 AI Coding 更像局部协作，人手动 `@` 文件、解释背景、控制上下文。
- Codex、Claude Code、Cursor Agent 等工具出现后，Agent 可以自己搜索、读文件、跑命令、定位入口、修改多个文件。
- Agent 自主性提高后，问题从“AI 能不能写代码”转向“Agent 能不能稳定读到正确上下文”。

这是 AIPD 的核心外部背景，应作为 L2 的“行业 / 玩法变化”记录。

### 5.3 主要痛点

README 和 docs 已经稳定列出几个痛点：

- Agent 进入项目后不知道项目为什么长成现在这样。
- 代码表达当前实现，但很难表达历史原因、产品边界和长期取舍。
- README 通常只写安装和入口，Issue / PR / 聊天记录太碎。
- 聊天上下文会断，需求、猜测、否定、临时方案和稳定判断混在一起。
- Agent 改共享方法时，可能不知道外部隐含依赖。
- 做完任务后，如果经验不回写，下一次 Agent 仍会重新踩坑。

这些是 L2 的痛点材料，不应只散在 README 和 docs 中。

### 5.4 竞品 / 范式 / 工具生态

当前材料中已经出现的外部工具和范式包括：

- Coding Agent：Codex、Claude Code、Cursor Agent。
- Agent 客户端 / 套壳项目：FanBox、OpenCovibe、codex-app-server-web、Claude Code Web 等。
- 程序化接口路线：Codex App Server、Codex SDK、`codex exec --json`、Codex MCP server、Claude Agent SDK、OpenCode 原生接口。
- 桌面技术路线：Electron、Tauri。

这些不应在 L2 写成工程选型细节，而应写成外部生态观察：AI 开发工具正在从 CLI / IDE 插件扩展到 Agent 工作台、聊天客户端、programmatic interface 和多 Agent adapter。

### 5.5 AIPD Desktop 的需求来源

`c0.8-aipd-desktop-zero` 记录了一个重要外部 / 用户场景：

- Codex 桌面端按“聊天”组织工作。
- AIPD 按 Case / Step 组织工作。
- 两者组织单位不一致，导致创建 case、执行 case、修改 case、讨论 step 分散在多个聊天里。
- 用户在并行多条线时很快找不到刚才在哪个聊天里讨论过什么。
- CLI 输入对语音输入和鼠标编辑不友好，正常 UI 输入框有实际价值。

这属于 L2 的“用户工作流痛点”，不是 L4 Desktop 功能设计本身。L4 再根据这些痛点定义 AIPD Desktop 的产品边界。

## 6. L2 应补的结构建议

建议后续创建 `_adoc/L2-research/index.md`，至少包含：

1. **用户画像**：一人公司、单人开发者、AI 深度参与的大型个人项目、需要长期维护上下文的项目。
2. **核心痛点线一：Agent 上下文问题**：Agent 查询过程上下文过重，难以只保留稳定结果；需要知识库沉淀查询后的可复用结果。
3. **核心痛点线二：项目文档完整性问题**：项目缺少完整、可持续维护的知识库，很多细节长期藏在代码和碎片记录中。
4. **核心痛点线三：AI 长时间工作问题**：长任务需要从聊天迁移到 OKR / Case / Step / 执行记录 / Weave Candidate 等可恢复结构。
5. **核心痛点线四：AI 原生代码架构扩展问题**：敏捷迭代和 AI 协作要求代码架构更重视上下文解耦、纵向黑箱、降低牵连面和局部可检索；后续需要按技术栈沉淀 AI 原生架构经验，例如 Vue / Nuxt 体系。
6. **需求机会线：SOP / AI 程序**：借助 Agent 行动能力，把复杂、重复、需要判断的任务沉淀为由 LLM / Agent 运行时驱动的 AI 原生程序。
7. **需求线：未整理信息暂存与分流**：用 Inbox 接住未定型材料，再判断 weave、转 case、转 SOP 或丢弃。
8. **需求线：AI 工作控制、验收和信任**：让人能知道 AI 做了什么、如何验收、哪些经验应回写、什么时候可以归档或提交。
9. **需求线：AIPD 框架自迭代**：通过 learn / transcript / case 经验回流，持续修正 AIPD 自身 skill、模板和 Agent 行为规则。
10. **需求线：已接入项目的 AIPD 架构升级迁移**：AIPD 自身持续迭代后，已有项目需要通过 update 安全同步 Agent Entry、map、case 模板、L3-L5 结构和新规则。
11. **需求线：Desktop 文件查询与观看**：随着 AIPD 文件体系扩大，需要更贴合 L1-L5 / Case / Step / SOP / MMD 的查询、观看和引用界面。
12. **需求线：CLI Agent 桌面化使用**：强编程 Agent 多以 CLI / 对话列表形态存在，需要桌面端承载更适合 AIPD 的输入、上下文引用和项目对象组织方式。
13. **需求线：MMD 高带宽交流**：用 Mermaid / MMD 提升人与 AI 对齐复杂结构、边界、流程和架构的速度。
14. **外部趋势**：从 Vibe Coding 到 Agent Coding，Agent 自主搜索和执行能力增强。
15. **工具生态观察**：Codex / Claude Code / Cursor Agent，CLI / IDE / Desktop / App Server / SDK / MCP 等不同形态。
16. **Desktop 场景来源**：聊天组织单位和 AIPD Case / Step 组织单位不一致；普通终端输入体验不足。
17. **非目标用户 / 不适用场景**：一次性原型、极小项目、完全不需要长期上下文沉淀的任务。

## 7. 不建议写入 L2 的内容

- AIPD 自身三条主线的内部模型细节，这属于 L3。
- Case / Step 的字段、模板和执行机制，这属于 L3 / L4 / case 模板。
- Vue / Nuxt 等具体技术目录结构、组件模板、MMD 画法和代码组织规范，这属于 L3 / L5 / 技术栈局部 README。
- SOP 的具体执行模板、参数、状态记录和工具调用细节，这属于 SOP / L3 / L4 / L5。
- Inbox 的具体收件格式和整理流程，这属于 Inbox 模块 / L4 / L5。
- Learn 的具体 transcript 审计规则和回写包格式，这属于 Learn 功能线 / L4 / L5。
- Update 的具体审计清单、合并策略和模板差异处理流程，这属于 Update 功能线 / L4 / L5。
- AIPD Desktop 的具体功能列表、技术栈、AgentAdapter 接口，这属于 L4 / L5 / Desktop case。
- Codex / Claude Code / OpenCode 的具体接入协议、会话模型和 adapter 设计，这属于 L5 / Desktop case。
- Mermaid 图的具体语法、渲染脚本和图形规范，这属于 MMD / Mermaid skill、L4 / L5 或局部 README。
- Codex App Server、Tauri、Vue、Rust 的具体工程实现细节，这属于 L5。
- 具体脚本路径、构建命令和 skill 文件布局，这属于 L5 / 局部 README。

## 8. 风险判断

如果 L2 继续缺失，后续 Agent 容易把 AIPD 误解成内部自洽的文档 / skill 体系，却不知道它针对的是哪类用户、哪种 AI 开发方式变化、哪些具体工作流痛点。

尤其是 Desktop 方向，如果没有 L2 支撑，很容易滑向“做一个功能很多的 Agent 客户端”，而不是围绕一人公司 / 单人开发者 / AIPD Case 组织单位痛点做增强。
