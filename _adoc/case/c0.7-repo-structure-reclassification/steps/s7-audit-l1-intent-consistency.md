## 3. 当前 L1 已覆盖内容

当前 L1 已经表达了几个基本判断：

- AIPD 是面向 AI 协作的软件项目认知框架。
- AIPD 不替代具体 Coding Agent。

## 4. 与当前稳定表达的主要差异

### 4.1 L1 缺少新版一句话定位与框架定位

当前讨论确认，L1 不能直接复用 docs 的讲故事和引导顺序。docs 面向人建立理解路径，L1 面向项目意图层，应更直接回答 AIPD 是什么、为什么成立、长期覆盖哪些方向。

目前更合适的一句话定位是：

```text
AIPD 是面向 AI 时代的软件开发框架，用来组织人和 AI 从项目方向到代码实现再到经验回写的完整协作过程。
```

这里的“开发框架”不是传统代码框架。传统代码框架主要约束代码实现里的前端、后端、移动端或桌面端；AIPD 约束的是 AI 和人如何共同推进一个软件项目。

### 4.2 L1 缺少 AI 原生开发框架的概念转变

当前 L1 仍偏“项目认知框架 + Codex Skill”的表达。它没有讲清 AIPD 作为 AI 原生开发框架的新产物，覆盖的不只是写代码阶段，而是从项目方向、用户痛点、外部调研、核心抽象、功能设计、代码实现到后续迭代的完整协作过程。

这部分不应写成 docs 那种“从 Vibe Coding 到 Agent Coding”的叙事，而应抽象为 L1 的项目判断：AI 时代的软件开发对象已经从“代码实现过程”扩展为“人和 AI 如何共同维护项目认知、任务执行、代码架构和经验回写”。

### 4.3 L1 缺少三个长期大方向

当前讨论确认，L1 可以用三个大方向展开一句话定位：

- **项目知识库**：负责 L1 到 L5 的项目认知，沉淀方向、痛点、调研、核心抽象、功能边界和工程规则。
- **任务执行系统**：负责 OKR、Case、Step、分身 Agent、执行记录和经验回写，让大任务可恢复、可派发、可验收。
- **AI 原生代码架构**：负责面向真实代码实现的组织实验，让代码更适配 AI 理解、修改和扩展。

这三个方向和 docs 的三条主线底层一致，但 L1 应写成项目方向，不写成人类阅读教程。

### 4.4 L1 缺少适用 / 不适用边界

README 已经说明：

- AIPD 适合 AI 深度参与、项目持续迭代、存在业务语言、历史取舍、跨模块规则、多 Agent 执行和经验沉淀需求的项目。
- 如果项目只有几个文件，或者只是一次性原型，AIPD 可能太重。

当前 L1 没有这层取舍，Agent 读 L1 后不容易判断 AIPD 的边界和推荐使用场景。

### 4.5 L1 过度突出当前 Codex 适配

“Codex 优先”仍然是当前阶段策略，但它更像短期研发重点，不应压过 L1 长期方向。

当前 L1 的“当前重点”集中在 Codex 分身 Agent、Codex custom agents、case-run 派发链路上，容易让 Agent 误判 AIPD 的长期目标就是 Codex 适配。后续应保留 Codex 优先作为阶段策略，但把长期方向放在前面。

### 4.6 L1 缺少渐进采用原则

docs 已经明确第一次完整路径：

- 先建立知识库最小闭环。
- 再用 Case / Step 做大任务。
- 再把稳定经验回写。
- 最后再考虑 AI 原生代码架构。

这个渐进性应进入 L1 的核心取舍：AIPD 不要求项目一次性上完整体系，而是可以先拿到最小闭环价值。

### 4.7 L1 缺少“不是什么”的边界

README 和 docs 已经表达：

- AIPD 不是 Vue、Next 或 Spring 这类代码框架。
- AIPD 不替代 Coding Agent。
- AIPD 不只是流程工具、规格工具或单次任务规范。
- AIPD 不是上来就要求重构代码架构的硬规范。

当前 L1 只写了“不替代具体 Coding Agent”和“不继续扩张成完整通用开发纪律框架”，边界还不够完整。

## 5. 建议回填到 L1 的方向

后续刷新 `_adoc/L1-intent/intent.md` 时，建议按以下结构组织：

1. **一句话定位**：AIPD 是面向 AI 时代的软件开发框架，用来组织人和 AI 从项目方向到代码实现再到经验回写的完整协作过程。
2. **框架边界**：传统代码框架主要约束代码实现里的前端、后端、移动端或桌面端；AIPD 约束 AI 和人如何共同推进一个软件项目。
3. **覆盖过程**：AIPD 覆盖项目方向、用户痛点、外部调研、核心抽象、功能设计、代码实现和后续迭代，而不是只管理开发阶段。
4. **三个大方向**：项目知识库、任务执行系统、AI 原生代码架构。
5. **核心取舍**：不替代 Coding Agent，不把所有内容塞进一个提示词，不把项目管理、流程管理和代码框架混成一个大而全工具；优先让项目认知、执行状态和代码组织都变得更适合 AI 协作。
6. **适用边界**：适合长期迭代、AI 深度参与、一人公司、单人开发者或一个人配合 AI 开发大型项目的场景；一次性原型或小项目可能过重。
7. **当前阶段重点**：Codex 优先、Skill 本体稳定、case-run / 分身 Agent / Weave 链路完善、Desktop 作为增强壳探索。

## 6. 不建议写入 L1 的内容

- 具体 skill 命令清单。
- 具体构建 / 安装命令。
- `aipd-skill/`、`aipd-desktop/` 的详细目录结构。
- Codex agent `.toml`、平台适配、脚本路径等工程细节。
- Case 模板字段、Step 执行格式等 L4 / L5 / case 细节。

这些内容应继续留在 README、docs、L4、L5、局部 README 或具体源码目录中。

## 7. 风险判断

如果不刷新 L1，后续 Agent 进入项目时会更容易把 AIPD 理解成“Codex Skill + case-run 工具集”，而不是“AI Agent 深度参与开发后的项目长期上下文框架”。这会影响后续 L2-L5 的判断方向，也会让 Desktop case 的定位变窄。
