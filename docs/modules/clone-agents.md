# 分身 Agent

AIPD 的分身 Agent 不是每个任务的默认执行者，也不必总是继承主 Agent 的完整聊天。

更准确的模型是：Case / Goal / Work Package 承接长期状态，Main / Child 是运行时选择。分身读取完成局部分支所需的最小上下文，用于隔离高噪声过程、并发推进独立工作线或进行独立复核。

## 主 Agent 做什么

主 Agent 保留主线：

- 用户沟通。
- 意图判断。
- 目标边界。
- 分身调度。
- 方案审核。
- 最终判断。
- 状态写回。

主 Agent 可以直接完成内聚、高耦合、上下文规模可控的实现与验证，不必为了任务类型机械派发。只有过程明显会冲击主线或能够真正并发时，才把对应分支交给分身。

## 分身 Agent 做什么

分身 Agent 负责进入局部分支，完成当前目标。

它可以读取文档、扫描代码、核对接口、运行验证、整理 diff、修改文件，但这些任务类型本身不构成派发理由。完成后只回流压缩结果：

- 结论
- 依据
- 风险
- 建议
- 改动文件
- 验证结果

分身 Agent 不主动继续分身，不擅自扩大范围，不推进下一步。

## Case Execute 和分身

Case Execute 把目标、上下文和验收标准固化为 Work Package，但 Work Package 不等于分身任务。

每个 Work Package 都有明确文件入口、上下文索引、横向模块、验收标准和不做范围。运行时再根据任务形状选择由 Main 连续完成，还是交给分身执行。

普通 AIPD 对话、轻量修改、调研、验证和正式 case 都可以使用同一套分身逻辑。

## 运行时调度

分身不是每个任务的默认步骤。主 Agent 会同时判断：中间材料是否会显著污染主线、是否有两条以上真正独立的并发工作线、任务与主线判断的耦合度，以及启动和合并成本。

- 长文档、长日志、大量页面结构或批量扫描只需要回流少量结论时，分身适合做上下文隔离。
- 两条以上独立、可同时推进且不会写入冲突的工作线，分身适合做并发加速。
- 内聚模块、高耦合实现 / 调试或上下文可控的跨文件修改，主 Agent 可以直接连续完成。

主 Agent 根据运行时收益判定自然选择是否创建分身；平台能力不可用时由 Main 回退执行。用户明确要求不派分身时，遵循用户当前指令。

分身调度只是当前任务内部的执行方式，不扩大用户原始任务范围，也不授予额外的外部副作用权限。install、远端写入、删除等动作仍遵守各自的确认边界。

在 GPT-5.6 Sol Ultra 下，模型本身可能增加委派工作线。AIPD 不再无条件叠加另一层分身 fan-out；日常交互以 High 为基线，只有额外分身仍能带来明确隔离或并发收益时才主动创建。

## fork_context

在 Codex 语境中，AIPD 默认传递最小必要上下文。Case / Work Package 和显式文件足以恢复任务时，优先不 fork 完整聊天；只有分身强依赖尚未落盘的主线判断时，才使用 `fork_context: true`。

无论是否继承，分身都应以 case / phase / work package / `_adoc/` 文件为事实源。每条证据面只设一个 owner，主 Agent 不重复调查，只接收压缩结果。
