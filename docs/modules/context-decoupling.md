# 上下文解耦

上下文解耦是 AIPD 的核心判断之一。

它不是单纯的文档整理技巧，而是 AI 执行速度变快以后，项目如何把知识、任务边界、输入输出、验收口径和回写位置拆成相对独立上下文黑箱。

## 为什么 Agent 自己找上下文后仍然需要知识库

Agent 能搜索、读文件、跑命令，不代表它能稳定读到正确上下文。

真实项目里，源码和零散文档通常不能完整表达：

- 历史原因。
- 产品边界。
- 业务语言。
- 跨模块约定。
- 上一次执行的踩坑经验。

如果每次任务都要靠 Agent 从代码树里重新推断这些关系，它就会在局部上下文里做出看似合理、但不符合长期判断的决策。

AIPD 的做法是把项目知识、任务上下文和经验回写位置显式化。

## 不要只靠上下文改写模型底层倾向

上下文解耦还有一个更底层的原因：大模型的默认行为倾向不能靠一段上下文临时改写。

提示词可以约束当前任务、输出格式和注意事项，但很难让一个模型彻底改变自己的核心反应模式。一个天然倾向“马上执行”的模型，即使被要求先规划，也常常会先滑向执行；一个天然倾向“完整抽象”的模型，即使被提醒保持简单，也容易先把概念层层拆开。上下文能让它“扮演”另一种模式，却不一定能让它真的换一套底层判断。

所以 AIPD 不把希望押在“给模型讲清基础逻辑”上。更稳定的做法是：

- 用实际案例校准模型，而不是只讲抽象原则。
- 用明确边界约束行为，而不是期待模型自觉发散到正确位置。
- 用 case / work package / README 写清目标、输入、输出、验收和禁止事项。
- 用 map 和 L3-L5 把长期原则变成可检索上下文，而不是每次在聊天里临时重讲。

这也是 AI 原生代码架构和传统人类架构的差别。人更擅长从抽象原则和关系中迁移；大模型更擅长在大量局部上下文中匹配、生成和修改。AIPD 要做的是让项目结构适配这种能力：把正确案例、边界和验收口径放到模型能直接读到的位置。

## 黑箱上移

AI 可以快速生成大量局部实现，人的瓶颈会逐渐从“写代码”转移到“做判断”。

人不可能逐行审查 AI 产生的一切细节。更现实的方式是把决策位置上移：让人关注模块关系、输入输出、验收口径和系统约束，把局部实现细节作为可替换、可验证的黑箱。

一个合格的任务黑箱至少要有：

- 清楚的目标。
- 明确的输入和输出。
- 必读上下文。
- 不做范围。
- 验收标准。
- 执行后的回写位置。

## 从横向分层到纵向黑箱

上下文解耦不只是任务拆分技巧，也会影响代码组织方式。

传统开发更习惯横向分层：controller、service、repository、model、util、component、store、api 各自拆开，再通过调用关系组合成完整功能。这个结构适合人类按职责维护，也方便复用和控制文件长度。

但 AI 做局部任务时，横向分层经常会把上下文打碎。一个页面、接口或组件的输入、业务判断、中间处理和输出分散在多个文件里，Agent 必须来回跳转才能拼出完整链路。跳转越多，漏读、错连和误抽象的概率越高。

AI 更适合纵向黑箱：一个页面、一个模块、一个组件、一个接口或一个 step 尽量从输入到输出自足。它可以是一个文件，也可以是一个文件夹里的三五个文件；重点不是文件数量，而是这些文件只服务当前黑箱，外部不要随意引用它们作为共享能力。

所以 AI 友好代码的目标不是“没有依赖”，而是“理解时不需要到处跳”。当 Agent 处理当前黑箱时，源头输入、中间处理、业务规则、用户可见输出和验收口径应尽量贴在一起。

## 新的成本结构

人类写代码时代，代码量膨胀、重复和长文件很快会变成维护成本，因为人写得慢、读得慢、批量修改也慢。

AI 时代的成本结构不同。AI 生成 token 和批量修改的成本更低，局部重复和文件变长不一定是第一优先级问题。更危险的是牵连不确定性：改一个 helper、service 或共享方法时，不知道外部有多少地方依赖了它的隐含行为。

AI 擅长在明确边界内高速生成和修改；它不适合在隐性依赖网里保证每个外部调用方都不受影响。因此新的优先级更接近：

1. 先降低牵连面。
2. 再控制代码量。
3. 最后再抽取复用。

也就是说，宁可先让局部黑箱膨胀，也不要一开始就把它们耦合在一起。代码量和 token 消耗可以后处理；“改一个牵出一片”会直接破坏 AI 的局部执行优势。

## 先解耦，后抽取

AI 友好开发不应该把 DRY 放在第一阶段。更稳的顺序是先 Decouple，再 DRY。

前期可以允许不同页面、接口、组件或 step 在自己的黑箱内各写各的。等重复逻辑经过实践证明稳定，再由人和 Agent 判断是否抽成共享能力、基础设施或二次打包产物。

这个过程有点像“后置压缩”，但业务逻辑不能只按代码相似度抽取。两段代码现在长得一样，不代表未来会长期共享同一语义。只有当抽取能降低理解跳转、降低牵连面，或提升独立验收和替换能力时，它才是正收益。

因此，重复三次不自动等于应该抽象；调用次数多也不自动证明 helper 合理。抽象的目标不是减少字符，而是减少理解成本和外部牵连。

## 和面向指令 / 面向状态的关系

横向分层和纵向黑箱，不等同于面向指令和面向状态。

面向指令和面向状态主要描述程序内部如何表达行为：是通过动作序列推进，还是通过状态及其派生结果组织。它们在传统开发和 AI 友好开发里都可以存在。

上下文解耦讨论的是另一条轴：代码和任务上下文应该如何组织，才能让 AI 稳定理解、修改和验收。

一个纵向黑箱内部可以是面向状态的，例如订单状态、页面状态、编辑状态；也可以是面向指令的，例如按步骤处理一次提交。关键不在于选哪个范式，而在于当前黑箱的输入、状态、动作、输出和验收口径能否在局部上下文内读清楚。

## map、case、work package、weave 的关系

上下文解耦不是某一个文件完成的，而是多个能力共同服务：

| 能力 | 在上下文解耦中的作用 |
|---|---|
| L1-L5 | 把长期项目知识分层存储 |
| map | 把用户自然语言路由到该读的上下文 |
| case | 把一次事项的目标、边界、上下文索引和验收固定下来 |
| work package | 把 case Execute phase 内的执行目标包拆成可派发、可恢复、可验收的黑箱 |
| weave | 把稳定新信息回写到项目记忆 |

AIPD 不追求把所有信息塞进一个超长提示词。它追求的是分层存储、扁平检索、局部任务自足和逐步回写。

## 和普通封装的区别

封装主要回答“如何隐藏内部细节，只暴露稳定接口”。

上下文解耦回答的是“人的决策应该停在哪个抽象层级，哪些信息要让 Agent 显式读到，哪些局部实现可以作为可验收黑箱”。

一个任务黑箱内部可以有封装，也可以没有封装。只要边界、输入、输出和验收条件清楚，它就能成为上层决策对象。

## 故事原型：垒高楼和二层小洋楼

Case Design 里的“垒高楼 vs 二层小洋楼”是上下文解耦的一个典型故事原型。

旧的 step 思路像垒高楼：第一版先做 list，第二版在 list 上加分页，第三版再加搜索，第四版继续叠筛选。每次新增能力都在原有地基上继续改，地基撑不住时又要回头修；修地基时还要保证前面功能不坏，代码容易越堆越高。

新的 Design 思路更像向外平摊的二层小洋楼：先找到复杂度爆点，设计一个能横向扩展的结构，再让后续模块在这个结构上各自接入。比如搜索列表的爆点不是列表渲染，而是 API 调用前的参数组装；每个 Filter 自治产出参数，Pagination 作为特殊 Filter，页面主干只收集参数并触发搜索。后续新增筛选项时，是横向增加模块，不是在旧搜索函数上继续叠逻辑。

这个故事不只是 case design 的比喻，也说明了为什么 AIPD 要用案例和边界约束模型：只告诉 Agent“不要堆版本”不够；要把什么叫“堆版本”、什么叫“横向铺模块”、验收时看什么，写成可读取的设计材料。

## 和工具对比的关系

AIPD 不替代 Agent、规格工具或流程工具。

它更像项目长期上下文层：

| 工具 / 层次 | 主要解决的问题 |
|---|---|
| Coding Agent | 具体执行、工具调用、代码修改 |
| 流程工具 | 执行步骤和开发纪律 |
| 规格工具 | 单次变更的规格和审查边界 |
| AIPD | 项目长期上下文、历史原因、经验沉淀和上下文解耦 |

这些层次可以叠加使用。AIPD 的重点是让 Agent 在进入任务前读到正确项目记忆，在完成任务后把稳定经验留下来。
