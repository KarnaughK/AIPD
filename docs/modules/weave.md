# Weave

Weave 是 AIPD 的反向编织能力。

它负责把已完成事项、已验证实现、case 归档、代码 diff、错误日志和外部资料中产生的新稳定信息，判断并回写到合适位置。

当一个目标已经完成，你需要决定“下一次 Agent 应该继承什么、由哪个文档负责”时使用 Weave。

## Weave 不是什么

Weave 不是聊天存档。

它不要求把每次对话、每段日志、每个临时猜测都写进项目知识库。这样会污染 `_adoc/`，也会让 Agent 更难检索。

Weave 关注的是稳定信息。未完成 case 中的讨论、设计和未来计划，默认不进入长期知识库。

## 常见回写方向

| 信息类型 | 回写位置 |
|---|---|
| 新核心概念、别名、误解 | L3 |
| 新产品功能边界、业务规则 | L4 |
| 新实现逻辑、跨模块规则、调试经验 | L5 |
| 页面、弹窗、组件内部入口 | 局部 README |
| 高频检索路径 | `_adoc/map.md` 或细节 map |
| 一次性执行过程 | Case / Work Package |
| 未完成 case 中的候选 | Close 归档候选 |

## Close 归档候选

执行型任务不一定直接修改长期 ADOC。

更稳妥的做法是先把候选放到 case 的 Close phase：说明这次任务产生了哪些可能值得回写的信息、当前状态是什么、候选归属是什么。

Case 未完成时，它只是 Close 阶段待复核的归档候选，不是长期知识库内容。Case 完成、实现落地并验收后，再由 `aipd-weave` 统一判断知识归属、索引更新和旧知识冲突。

## 与 case 归档的关系

Case 归档时通常会整理本次事项的结果、风险、经验和 Close 归档候选。

归档不是把 Case 删除，而是把一次任务的执行过程收束。只有已经成为现有项目事实的稳定知识才进入长期项目记忆；未实现设计、未来计划和一次性过程继续留在 Case 记录里。

## Weave 和 Learn

- **Weave** 更新当前项目：L3 / L4 / L5、map 和局部 README。
- **Learn** 更新 AIPD 框架自身：Skill、模板、Agent 规则和经验库。

同一段协作可以同时产生两类候选，但必须分别判断。不要因为 AIPD 框架规则需要改，就把它写进业务项目 L5；也不要用 Learn 代替项目知识回写。

## 一个最小 Weave 流程

1. 从已完成事项、Verify 证据和 Close 候选提取新信息。
2. 区分稳定事实、一次性过程、未实现设计和未来计划。
3. 为稳定事实选择唯一知识 owner，并检查旧文档冲突。
4. 更新正文后，再修需要暴露的新 map / index 入口。
5. 保留 Case 中的执行证据，不把长过程复制进长期知识库。
