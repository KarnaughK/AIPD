# Map 与检索系统

Map 是 AIPD 的检索系统，不只是 `_adoc/map.md` 这个文件。

它的目标是让 Agent 从用户自然语言稳定命中相关上下文，减少乱翻、漏读和误读。

当 Agent 经常全仓搜索、读错层级、需要多层跳转才能发现入口，或同一个业务词在文档和代码里名称不一致时查这篇。

## 两类 Map

AIPD 至少要区分两类 Map。

| 类型 | 服务对象 | 作用 | 例子 |
|---|---|---|---|
| 项目级认知 Map | 当前项目的稳定认知与代码结构 | 帮 Agent 判断任务应读取 L3 / L4 / L5 / 局部 README / L6 中哪些上下文；只有流程任务才进入 Case / OKR | `_adoc/map.md`、L3 map、L4 map、L5 map |
| 具体项目 / 技术栈 Map | 具体项目类型和实现结构 | 帮 Agent 在某类项目里命中页面、组件、provider、数据流、接口和实现入口 | Vue MMD 架构图、前端组件图、React / Node 技术栈 map |

第一类更偏 AIPD 通用认知检索，第二类更偏具体项目实现检索。

两者的共同目标都是：让 Agent 少猜、少乱翻，从用户一句话更稳定地读到正确上下文。

## 总 map 和细节 map

`_adoc/map.md` 是项目总 map，负责第一跳。

它应该暴露高频入口、稳定路径、关键词、功能线、核心概念、工程规则和代码入口。它不写完整正文，但要让 Agent 一次读完后能判断下一步读哪里。

细节 map 负责某个概念、功能线或工程规则的完整资源索引。

关键路径应尽量扁平。不要要求 Agent 必须通过多层目录链才能发现关键文档。

总 map 适合记录：用户说法 / 黑话、标准概念、高频任务入口、必读上下文、代码或局部 README 入口、兜底搜索词。它不记录一次性执行细节，也不复制完整规则正文。

## 局部 README

局部 README 是贴近 L6 的最后一层解释。

它适合记录某个页面、弹窗、组件、脚本或模块内部的入口、数据流、组件关系和修改注意事项。

总 map 可以指向局部 README，但不应该替代局部 README 解释所有实现细节。

## 搜索是兜底，也是地图维护信号

Map-first 不禁止 `rg`、语义搜索或代码探索。它规定顺序：

1. 先用 map 命中已知权威入口。
2. 命中不清楚时再搜索核心词、页面名、接口名和权限码。
3. 搜索发现一次性线索，只用于当前任务。
4. 搜索发现稳定、高频的新入口，候选通过 Weave 回写 map。

map 的质量不是“覆盖所有文件”，而是让下一次同类任务少一次盲搜。

## 当前入口

旧 README 中曾出现 `_adoc/context-map.md` 的说法。

当前 AIPD 仓库的正式项目地图入口是：

```text
_adoc/map.md
```

README 和 docs 统一使用 `_adoc/map.md`。
