# ADOC 分层

ADOC 是 AIPD 的项目认知结构。它把项目长期上下文按层级沉淀，让 Agent 能在改代码前先理解方向、场景、核心模型、产品边界和工程规则。

## L1-L6

| 层级 | 作用 | 典型载体 |
|---|---|---|
| L1 Intent | 项目方向、目标和长期取舍 | `_adoc/L1-intent/` |
| L2 Research | 外部世界，包括用户、场景、痛点、竞品、行业范式、市场观察 | `_adoc/L2-research/` |
| L3 Core | 项目内部靠哪些核心模型成立 | `_adoc/L3-core/` |
| L4 Product | 核心模型落成哪些产品功能、边界和用户可见行为 | `_adoc/L4-product/` |
| L5 Dev | 产品功能落到代码时的跨模块工程规则和实现约定 | `_adoc/L5-dev/` |
| L6 Code | 真实代码实现 | 项目源码目录，不建 `_adoc/L6/` |

L1-L5 不是 L6 的纯索引层。它们本身就是项目讨论、判断和经验沉淀的文档层，也给 Agent 理解和修改 L6 代码提供上下文。

## L2 不只是痛点

L2 记录方向所处的外部世界。痛点只是其中一类。

适合放入 L2 的内容包括用户、场景、需求、竞品、同类玩法、市场观察、流量来源、变现方式和调研结论。

这些外部观察会影响 L3 和 L4，但不要直接混同为项目内部模型。

## L3 是项目成立模型

L3 不等于狭义数据模型。

不同项目的 L3 可以不同：

- 业务系统：核心对象、领域语言、状态流转、数据关系。
- 内容、社区或 SEO 项目：内容模型、增长模型、流量模型、留存模型、商业模型。
- 框架项目：上下文解耦、检索模型、执行模型、回写模型。

判断一个信息是否属于 L3，重点看它是否已经成为本项目长期遵守的内部模型，是否会约束多个 L4 功能。

## L4 的双重职责

L4 既解释产品功能，也可以串联代码入口。

但 L4 不能只写成代码地图。一个 L4 功能线至少要说清：

- 这个功能是什么。
- 它解决什么用户场景。
- 功能边界和不做范围是什么。
- 它依赖哪些 L3 核心模型。
- 相关页面、接口、权限、数据对象和 L6 入口在哪里。

## L5 的双重职责

L5 既沉淀业务实现逻辑和工程规则，也为 Agent 修改代码提供稳定上下文。

它适合承载跨模块、跨平台、长期稳定的规则，例如权限、路由、插件、前后端约定、平台适配和调试经验。

页面、弹窗、组件内部的最后一层实现地图，优先放在代码目录就近 README。

## 纵向模块和横向能力

L1-L6 是纵向模块，回答“东西放哪一层”。

map、case、phase、work package、weave、Agent Entry 是横向能力，回答“Agent 做事时怎么把这些层串起来”。
