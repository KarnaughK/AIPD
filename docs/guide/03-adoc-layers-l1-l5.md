# 03 ADOC L1-L5 分层

项目知识库首先要回答一个问题：信息放在哪里。

AIPD2 用 L1-L5 存储项目认知，用 L6 表示真实代码。

```text
L1 Intent
  -> L2 Research
  -> L3 Core
  -> L4 Product
  -> L5 Dev
  -> L6 Code
```

## L1 Intent

L1 记录项目方向、目标和长期取舍。

它回答：这个项目为什么存在，长期要坚持什么，不要偏到哪里去。

## L2 Research

L2 记录方向所处的外部世界。

它不只是“痛点”层，也包括用户、场景、需求、竞品、行业范式、市场观察、流量来源、变现方式和调研结论。

## L3 Core

L3 记录项目内部靠哪些稳定模型成立。

对业务系统来说，L3 可能是核心对象、数据模型和状态流转。对 AIPD2 这种框架项目来说，L3 更偏上下文解耦、扁平化检索、Case / Step 和 Weave 等执行认知模型。

判断一个信息能不能进入 L3，可以问：它是否已经从外部观察变成本项目长期遵守的内部模型。

## L4 Product

L4 把 L3 核心模型落成产品功能、功能边界、业务规则和用户可见行为。

它既解释功能本身，也可以串联相关代码入口。但 L4 不应该只写成代码地图，它要先说明功能是什么、解决什么场景、边界在哪里。

## L5 Dev

L5 记录产品功能落到代码时的跨模块工程规则和实现约定。

例如权限、路由、第三方插件、前后端约定、平台适配、Agent 调度、构建安装规则等。

页面、组件、弹窗内部的最后一层实现说明，优先放在代码目录就近 README，而不是全部塞进 L5。

## L6 Code

L6 是真实代码实现，不在 `_adoc/` 里建立 `_adoc/L6/`。

它分布在项目自己的源码目录里，例如前端、后端、爬虫、脚本、skills、platforms 等。L1-L5 会引用 L6 入口，但不替代 L6。

## 分层的作用

L1-L5 的重点不是让文档变整齐，而是让 Agent 知道任务涉及哪类上下文。

一次改动可能只需要 L5 和代码，也可能需要先读 L3 核心模型，再读 L4 功能边界，最后进入 L6 实现。

下一章会讲 Agent 怎么从用户一句自然语言找到这些入口。
