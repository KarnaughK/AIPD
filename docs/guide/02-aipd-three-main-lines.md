# 02 AIPD 的三条主线

第一章解释了为什么 Agent Coding 之后，项目不能只靠人临时 `@` 文件和口头补上下文。

AIPD 的新版学习路径可以先按三条主线理解，而不是一上来背所有术语。

```text
第一层：知识库最小闭环
第二层：Case / Work Package 开发逻辑
第三层：AI 原生代码架构实验
```

这三条主线不是互相隔离的硬层级。它们是三个问题域：AI 怎么理解项目，AI 怎么执行大任务，代码本身怎么更适合 AI 维护。

## 第一层：知识库最小闭环

第一层解决的是：Agent 怎么稳定理解项目。

它包含三件事：

- 知识库本体：L1-L5，必要时延伸到被索引的 L6 真实代码。
- 存：通过初始化编织和 Weave / 反向编织，把已完成、已验收的稳定项目认知写回 `_adoc`、局部 README 或 map；未完成 case 候选先留在 Close 归档候选。
- 取：通过 `_adoc/map.md` 把用户自然语言路由到相关 L3 / L4 / L5 / 局部 README / L6 代码入口。

这一层结束后，AIPD 已经可以独立发挥作用。用户不必每次都手动 `@` 文件，Agent 也不必在项目里重新盲搜一遍。

## 第二层：Case / Work Package 开发逻辑

第二层解决的是：Agent 怎么完成一个较大的事项。

聊天上下文会压缩、会中断，也会混进很多讨论、猜测和被否定的方案。Case / Work Package 的作用是把任务从聊天里固定到文件里。

- 飞书 OKR 说明当前阶段为什么做这些事。
- Case 在 `case.md` 记录一次短周期事项的目标契约，并维护 Think / Design / Execute / Verify / Close 状态。
- Think 处理目标不清、信息不足、调研和抉择。
- Design 找复杂度爆点，并做最小必要解耦。
- Work Package 把设计边界下的横向模块变成可执行、可恢复、可验收的目标包。
- 分身 Agent 消化搜索、验证、日志和 diff 等过程成本。
- Close 归档候选把执行中可能需要沉淀的内容留到 case 收口时统一判断。

所以 Case / Work Package 不是知识库本体，而是基于知识库之上的任务执行系统。

## 第三层：AI 原生代码架构实验

第三层解决的是：代码本身怎么更适合 AI 理解、修改和维护。

传统开发常常优先横向分层和复用。AI 做局部任务时，更怕的是上下文被横向依赖打散：改一个 helper、service 或共享组件，就必须重新判断很多调用方的隐含语义。

AI 更适合纵向黑箱：

```text
一个页面 / 一个模块 / 一个组件 / 一个接口 / 一个 step
从输入到输出尽量自足
```

这一层的判断是探索性的。它包括上下文解耦、降低牵连面、Decouple first, DRY later、局部 README 和代码就近地图等思路。它不要求所有项目立刻改造成某种固定架构，但它指出了 AI 时代代码组织可能变化的方向。

## 辅助能力放哪里

有些能力需要讲，但不必单独升成一整层。

| 能力 | 推荐归属 | 说明 |
|---|---|---|
| OKR | 第二层 | 飞书目标管理，帮助判断 case 是否推进阶段目标 |
| Think | 第二层 | Case 前或 Case 内的讨论和决策阶段，处理 Create / Kill / Defer / Research / Weave / Continue / Design |
| Inbox | 辅助能力 | 临时接住未整理想法，后续可 weave 进知识库或转成 case |
| MMD / Mermaid | 按用途归属 | 画稳定认知归第一层，画执行拆解归第二层，画代码组织实验归第三层 |
| Agent Entry | 贯穿三层 | Agent 进入项目后先读什么、按什么规则行动 |
| Weave | 第一层 + 第二层连接处 | 维护知识库，也承接 case / work package 产出的经验候选 |

后面的章节仍沿着这三条主线展开。
