# 02 AIPD 如何让项目记住自己

上一章的问题是：Agent 会执行，但项目不会自动把正确上下文交给它，也不会自动继承上一次工作。

AIPD 用一个持续循环解决它：任务前读认知，任务中把状态留在文件里，任务后把稳定经验写回。

## 一张完整地图

```text
任务前：找到正确认知
  AGENTS.md -> _adoc/index.md -> _adoc/map.md
                              -> L3 / L4 / L5 / 局部 README / L6

任务中：让目标和状态留在文件里
  Case Contract -> Think -> Design -> Execute -> Verify -> Close
                                  -> Work Package -> Main / Child

任务后：把稳定经验留给下一次
  Close 候选 -> Weave -> _adoc / map / 局部 README
```

它不是三个互不相干的子系统。一次真实任务会依次经过这三段。

## 任务前：不是多读，而是读对

Agent 进入项目后先看到 `AGENTS.md`。它不承载所有项目知识，只规定第一跳：

```text
AGENTS.md -> _adoc/index.md -> _adoc/map.md
```

`index.md` 说明项目认知有哪些区域；`map.md` 把用户自然语言、业务词和工程词路由到当前任务真正需要的 L3 / L4 / L5、局部 README 和 L6 代码。

例如“会员导出权限”可能命中：

- L3 的组织、成员和套餐关系。
- L4 的导出功能边界与用户可见失败行为。
- L5 的权限校验和前后端协作规则。
- 导出页面就近 README 和真实接口代码。

Agent 不需要先读完整个 `_adoc`。Map-first 的目标就是用最少上下文找到下一跳；路径缺失时再搜索，并把稳定新入口修回地图。

## 任务中：聊天继续流动，文件承接状态

一次短修改可以直接完成。目标较大、需要调研或必须验收时，AIPD 使用 Case：

```text
Case Contract -> Think -> Design -> Execute -> Verify -> Close
```

Case Contract 固定目标、要做、不做、完成标准和上下文索引。后续 phase 只按当前需要渐进加载：

- 不确定的事实进入 Think。
- 会影响执行边界的判断进入 Design。
- 正式改动按 Work Package 进入 Execute。
- 结果用证据进入 Verify。
- 已完成事项最终在 Close 收束。

每次会影响恢复路径的变化都写一个短 checkpoint：当前问题、已经确认什么、还有什么 open、下一步到哪里。它不是聊天备份；它只保存继续完成目标所需的最小状态。

## 任务后：只回写已经稳定的东西

任务过程中会出现很多材料：尝试过的方案、临时日志、外部调研、设计草案、新规则和最终实现。

它们不能全部进入长期知识库。Weave 只处理已经完成、实现或验证的稳定信息：

| 新信息 | 稳定位置 |
|---|---|
| 核心概念、关系、常见误解 | L3 |
| 产品功能边界和用户可见规则 | L4 |
| 跨模块工程约定和实现逻辑 | L5 |
| 页面、弹窗、组件内部入口 | 就近 README |
| 高频检索路径 | 总 map 或细节 map |
| 一次性过程、未实现设计 | Case / Work Package |

这一步让项目记忆越来越准，而不是越来越长。

## 原来的“三条主线”放在哪里

AIPD 仍然可以从架构上看成三个问题域：

1. **知识库最小闭环**：项目认知怎样存、怎样取、怎样更新。
2. **Case / Work Package 执行系统**：长目标怎样设计、执行、恢复和验收。
3. **AI 原生代码架构实验**：代码怎样降低牵连面，让 Agent 更容易在局部上下文中完成工作。

它们适合解释框架结构，却不适合作为第一次使用的顺序。第一次使用只需要沿着一次真实任务走完整个记忆循环；代码架构实验可以以后再采用。

## 专用能力如何接入循环

- Inbox 在信息还没定型时先接住它。
- 飞书 OKR 给 Case 提供阶段目标对齐，不替代任务执行。
- Goal Mode 让已绑定 Case 在内部 Gate 通过后自主推进，不改变 Case 本体。
- Child Agent 在上下文隔离、真实并发或独立复核有净收益时接入 Execute。
- Learn 把真实协作经验回流到 AIPD 框架自身，和当前项目的 Weave 分工不同。
- Update、Mermaid、Git Push 各自处理结构升级、图示和低风险推送。
