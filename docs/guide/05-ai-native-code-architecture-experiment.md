# 05 AI 原生代码架构实验

第三条主线是实验性的：代码本身应该怎么组织，才更适合 AI Agent 理解、修改和维护。

这不是说传统工程规范失效了，而是 AI 参与开发后，成本结构发生了变化。

## 人的代码架构

人类开发常常依赖横向分层：

```text
controller -> service -> repository -> model -> util
```

前端里也类似：

```text
page -> component -> store -> api -> helper -> constants
```

这种结构对人很合理。人写得慢、读得慢、工作记忆有限，所以需要短函数、小文件、稳定命名和复用来压缩认知。

## AI 的代码架构倾向

Agent 的困难不完全一样。

它可以快速读一大段上下文，也可以批量生成和修改代码。它更怕的是一次局部任务需要在很多文件之间来回跳转，才能拼出完整业务链路。

所以 AI 更适合纵向黑箱：

```text
一个页面 / 一个模块 / 一个组件 / 一个接口 / 一个 work package
从输入到输出尽量自足
```

重点不是“所有代码都写在一个文件里”，而是当前黑箱内部的上下文要贴得足够近。Agent 处理当前黑箱时，应该尽量一次读到源头输入、中间处理、业务规则、用户可见输出和验收口径。

## 新的优先级

AI 时代，代码量、局部重复和文件变长不一定是第一优先级问题。

更危险的是牵连不确定性：改一个 helper、service 或共享方法时，不知道外部有多少地方依赖了它的隐含语义。

因此新的优先级更接近：

```text
先降低牵连面
再控制代码量
最后再抽取复用
```

也就是：

```text
Decouple first, DRY later
```

前期允许页面、接口、组件或 work package 在自己的黑箱内各写各的。等重复逻辑经过实践证明稳定，再判断是否抽成共享能力。

## 代码思路放在哪里

“具体怎么写某种代码”的思路，主要服务第三层。

但它也会和前两层发生关系：

- 如果某个写法已经成为跨模块稳定工程规则，应该回写到第一层的 L5。
- 如果某个写法只是一次 case 里的临时取舍，先留在第二层的 case / work package。
- 如果某个写法体现 AI 原生代码组织范式，可以进入第三层讨论。

所以归属原则不是看文件类型，而是看它解决什么问题。

```text
稳定项目认知 -> 第一层知识库
任务推进状态 -> 第二层 Case / Work Package
代码组织范式 -> 第三层 AI 原生代码架构实验
```

## 为什么说它是实验

知识库最小闭环和 Case / Work Package 已经可以直接用于项目协作。

AI 原生代码架构则更像进一步探索：当项目已经有知识库和任务执行系统后，是否可以让代码本身也长成更适合 Agent 的形态。

这条主线值得讲，但不应该包装成已经定型的行业共识。更合适的表达是：AIPD 在探索 AI 深度参与开发后，代码组织范式可能出现的新方向。
