# 05 让项目从完成的工作里学习

一个 Case 关闭了，不代表它产生的所有内容都该进入长期知识库。

这章先完成记忆循环的最后一步：用 Weave 把已验证的新事实写回。然后再解释，为什么上下文解耦是建立在这个循环之上的进阶方向。

## 先区分结果、过程和候选

一次 Case 可能包含：

- 调研中看过的外部资料。
- 被否决的方案。
- 最终采用的产品规则。
- 为了调试临时加的日志。
- 新增页面的数据流和入口。
- 未来可能想做但本次没做的能力。

把它们全部塞进 `_adoc` 会让知识库迅速变脏。AIPD 的原则是：

- 一次性过程留在 Case / Work Package。
- 未实现设计和未来计划留在 Case，不能写成当前事实。
- 已经完成、实现或验证的稳定信息，才进入 Weave 判断。

## Weave 怎样决定写到哪里

```text
新信息
├── 项目靠什么成立？            -> L3
├── 用户会看到什么规则？        -> L4
├── 多个入口要遵守什么实现约定？ -> L5
├── 只影响一个页面或组件？       -> 就近 README
├── 下一次怎样更快命中？         -> map
└── 只属于这次执行？             -> Case / Work Package
```

例如退款 Case 完成后：

- “发货状态和退款资格的关系”可能进入 L3。
- “已发货订单不可自助退款，用户看到什么提示”可能进入 L4。
- “前后端都必须以服务端资格判定为准”可能进入 L5。
- 退款弹窗的组件和数据流进入页面 README。
- “退款”“已发货”到上述入口的关键词映射进入 map。

Weave 不是自动分类器。它要检查新事实是否真的稳定、是否与旧认知冲突、哪个位置是知识 owner，并避免复制多份完整定义。

## Weave、Learn、Inbox 和 Update 不同

| 能力 | 处理什么 | 不处理什么 |
|---|---|---|
| Inbox | 未整理、低承诺度的信息 | 不自动当成长期事实或待办 |
| Weave | 当前项目已完成、已验证的新知识 | 不做聊天存档，不更新 AIPD 框架本身 |
| Learn | 从 transcript、Case 或用户反馈更新 AIPD 框架经验 | 不替代当前项目 Weave |
| Update | 把新版 AIPD 结构安全合并进已初始化项目 | 不覆盖项目已有认知 |

这四个入口共同控制“什么现在先放着、什么已经可以留下、框架本身怎样进化”。

## 为什么循环跑起来后，代码架构也会变化

当 Agent 经常执行局部任务时，一个新成本会变得明显：理解一次改动需要跳多少上下文，以及改一个共享点会牵出多少隐含调用方。

传统工程常把消除重复放在很高优先级。对 Agent 来说，过早抽成共享 helper、service 或全局状态，可能让一个局部需求必须跨很多文件和语义边界才能读懂。

AIPD 因此探索“上下文解耦”：

- 页面、接口、组件或目标包尽量形成局部自足的纵向黑箱。
- 人把决策上移到输入、输出、规则、边界和验收。
- 局部实现可以替换，只要黑箱契约成立。
- 重复逻辑先在真实使用中稳定，再判断是否值得上移共享。

它的优先级是：

> 先降低牵连面，再控制代码量，最后判断抽象。
> Decouple first, DRY later.

## 这不是必须采用的代码规范

上下文解耦是 AIPD 的进阶实验，不是安装后必须重构整个项目的教条。

它适合在真实问题出现时渐进采用：

- 一个页面每次修改都要跨十几个目录。
- 一个共享 helper 的隐含语义越来越多。
- 多个 Agent 经常因为边界不清修改同一主干。
- 局部模块难以独立验收和替换。

如果当前代码结构已经清楚、牵连可控，就没有必要为了“AIPD 风格”重写。更完整的原理、反例与故事见[上下文解耦](../modules/context-decoupling.md)，可运行的 Vue3 经验源码位于仓库根级 `experience-assets/`。
