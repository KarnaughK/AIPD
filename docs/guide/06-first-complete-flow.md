# 06 第一次完整路径

理解三条主线后，可以把第一次使用 AIPD 想成一个渐进路径。

不是所有项目一开始都要用完整体系。你可以先得到第一层的价值，再逐步进入第二层和第三层。

## 第一步：建立知识库最小闭环

先让项目有一套 Agent 能读的认知入口。

最小目标是：

- 有 `_adoc/index.md`。
- 有 `_adoc/map.md`。
- L1-L5 至少有基本入口。
- 高频功能、核心概念和代码入口能被 map 路由到。
- 新经验知道应该 weave 到哪里。

到这里，Agent 已经可以少靠用户 `@` 文件，更多靠项目自己的知识库定位上下文。

## 第二步：先判断是否需要 AIPD Think（施工中）

如果任务已经清晰，可以直接进入 Case / Step。

如果还只是一个想法、方向、陌生领域或“要不要做”的问题，后续应先进入正在施工中的 AIPD Think：

```text
AIPD Think -> Case Create -> Case Run
```

Think 先沉淀讨论、调研、方案比较和决策出口。只有出口是 Create 时，才继续创建 case。

## 第三步：用 Case / Step 做一个大任务

当任务变大，开始创建 case。

一个最小 case 应该写清：

- 目标。
- 场景分流。
- 上下文索引。
- 本次做什么、不做什么。
- 已确认的 step。
- 验收标准。
- Weave Candidate。

执行时不要靠聊天记忆直接开工，而是让执行 Agent 读取 step 文件、case 文件和 step 列出的上下文。

## 第四步：把稳定经验回写

任务完成后，不要让经验只留在聊天和 diff 里。

判断哪些信息值得进入长期知识库：

- 新核心概念或误解，回写 L3。
- 新产品功能边界，回写 L4。
- 新工程规则或调试经验，回写 L5。
- 新局部入口，回写就近 README。
- 新高频路径，回写 map。
- 一次性过程，留在 case / step。

这一步让下一次 Agent 继承本次经验。

## 第五步：再考虑 AI 原生代码架构

当项目已经能稳定“读上下文、执行任务、回写经验”以后，再考虑代码组织是否需要更 AI 友好。

这时可以观察：

- 某个功能是否经常需要跨很多文件才能理解。
- 某个共享 helper 是否让局部修改牵出一片。
- 某个页面或模块是否可以变成更自足的纵向黑箱。
- 局部 README 或 map 是否能减少 Agent 跳转。

第三层不是上来就必须做的重构，而是在知识库和 Case / Step 跑起来后，进一步提高 AI 协作稳定性的方向。

## 推荐读法

第一次读 AIPD，可以按这个顺序：

```text
01 从 Vibe Coding 到 Agent Coding
02 AIPD 的三条主线
03 知识库最小闭环
04 Case / Step 开发逻辑
05 AI 原生代码架构实验
06 第一次完整路径
```

读完 guide 后，再进入 `docs/modules/` 查具体能力。
