# 04 Case / Step 开发逻辑

知识库解决“Agent 怎么理解项目”。Case / Step 解决“Agent 怎么完成大任务”。

如果一个想法还处在“要不要做、怎么做才值、还缺什么调研”的阶段，它不应该直接进入 Case / Step。AIPD 正在施工的 **AIPD Think** 会作为前置讨论层，先把模糊想法状态化，再决定 Create、Kill、Defer、Research、Weave 或 Continue。

真实任务不会一直停留在一句需求里。它会包含讨论、猜测、否定、临时方案、边界确认、执行结果和验收反馈。如果这些都只留在聊天中，Agent 很难稳定恢复。

Case / Step 的目标是把一次事项变成可恢复、可派发、可验收的文件事实源。

## OKR：长期目标

OKR 不直接替代 case。

它回答的是：当前阶段为什么要做这些事项，什么结果算推进了项目方向。

当项目有多个 case 时，OKR 可以帮助判断：

- 哪些 case 更重要。
- 当前 step 是否还在服务阶段目标。
- 归档时哪些经验值得进入长期知识库。

## Case：一次事项的上下文容器

Case 是一次具体事项的容器。

更准确地说，Case 承接的是已经决定进入执行规划的事项。开放式讨论、深度调研和“是否值得做”的判断，后续应优先放在 AIPD Think 中。

它通常记录：

- 目标。
- 场景分流。
- 上下文索引。
- 要做和不做。
- step 列表。
- 验收标准。
- Weave 候选。
- 归档状态。

Case 的价值不是“列一个 todo”，而是让任务从聊天里落到文件里。

上下文压缩或中断后，恢复链路应该是：

```text
AGENTS.md
-> _adoc/index.md
-> _adoc/map.md
-> _adoc/case/index.md
-> 当前 case.md
-> 当前 step
```

## Step：可执行的小黑箱

Step 是 case 内可派发、可恢复、可验收的执行单元。

一个 step 应该能只凭自己的 step 文件、case 文件和列出的上下文文档独立执行。

它至少要写清：

- 目标。
- 已确认设计输入。
- 必读上下文。
- 任务清单。
- 验收标准。
- 不做范围。
- 执行记录。

如果两个 step 之间必须共享上一轮聊天里的临时判断，说明它们还没有真正解耦。要么合并成一个 step，要么先把共享判断沉淀到 case 文档、README 或图里。

## 分身 Agent：消化过程成本

case-run 中，执行 step 的工作可以交给分身 Agent 或角色 Agent。

分身 Agent 不是低上下文工人。更准确地说，它是从主 Agent 当前认知 fork 出来的克隆体，进入局部探索或执行分支，完成后只回流：

- 结论。
- 依据。
- 风险。
- 建议。
- 改动文件。
- 验证结果。

主 Agent 保留用户沟通、目标边界、最终判断和状态写回。

## Inbox：临时接住未整理想法

Inbox 不是 case，也不是稳定知识库。

它适合接住暂时说不清归属的内容：

- 一段灵感。
- 一个后续可能要做的点。
- 一条外部资料线索。
- 一个还没整理成 case 的问题。

后续再判断这些信息是 weave 进 L1-L5，还是转成 case，还是丢弃。

## Weave Candidate：执行和知识库的接口

Step 执行结束后，不一定直接改长期 ADOC。

更稳妥的做法是先产出 Weave Candidate：说明这次任务产生了哪些可能值得回写的稳定信息，以及建议写到哪里。

之后由 Weave 判断知识归属、索引更新和旧知识冲突。

这就是第二层和第一层的连接：Case / Step 负责执行，Weave 负责让执行经验进入项目知识库。
