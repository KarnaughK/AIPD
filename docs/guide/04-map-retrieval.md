# 04 map 检索

L1-L5 解决“知识怎么存”，map 解决“知识怎么读”。

Agent 进入项目后，如果每次都靠目录名和全文搜索猜入口，链路会很脆弱。AIPD2 使用 `_adoc/map.md` 作为第一跳检索地图，把用户自然语言路由到相关 L3 / L4 / L5 / 局部 README / 代码入口。

## map 不是正文

`_adoc/map.md` 不承载完整解释。

它更像路线图：

```text
用户意图 -> map 命中入口 -> 读取必要认知 -> 进入代码
```

例如用户说“case 执行、分身 Agent、fork_context”，map 应该直接告诉 Agent 去读哪些 L5 规则、case-run skill 和平台 agent-guide，而不是要求 Agent 一层层猜目录。

## 为什么需要第一跳稳定

人类可以靠经验在项目里跳转：先看 README，再看目录，再搜索关键词，再打开几个文件。

Agent 也能这样做，但它的一次漏读可能导致后续推理偏掉。AIPD2 更希望关键路径扁平暴露：高频入口、标准概念、功能线、工程规则和代码入口尽量在总 map 里可见。

这不追求把所有正文塞进一个超长文件，而是把“去哪读”这件事显式化。

## map 和局部 README

项目总 map 负责第一跳。

局部 README 或局部 map 负责最后一层实现说明，例如某个页面、组件、脚本或模块内部的数据流、修改注意事项和代码入口。

两者分工是：

- `_adoc/map.md`：让 Agent 从自然语言找到正确上下文。
- L3 / L4 / L5 map：解释某类概念、功能线或工程规则的详细入口。
- 局部 README：贴近 L6 代码，说明局部实现怎么理解和修改。

## 本仓库的入口

在 AIPD2 仓库中，Agent 执行任务时应优先读取：

```text
AGENTS.md
-> _adoc/index.md
-> _adoc/map.md
```

旧文档中出现过 `_adoc/context-map.md` 的说法，当前正式入口已经统一为 `_adoc/map.md`。
