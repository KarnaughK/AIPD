# 03 知识库最小闭环

AIPD2 的第一层是项目知识库。

这不是“写一套文档规范”，而是让 Agent 进入项目后能稳定知道：项目为什么存在，核心对象是什么，功能边界在哪里，工程规则有哪些，代码入口应该从哪里读。

知识库最小闭环由三部分组成：

```text
知识库本体 -> 存 -> 取
```

## 知识库本体

AIPD2 用 L1-L5 存储项目认知，用 L6 表示真实代码。

| 层级 | 作用 |
|---|---|
| L1 Intent | 项目方向、目标和长期取舍 |
| L2 Research | 外部世界，包括用户、场景、痛点、竞品、行业范式、市场观察 |
| L3 Core | 项目内部靠哪些核心模型成立 |
| L4 Product | 核心模型落成哪些产品功能、边界和用户可见行为 |
| L5 Dev | 产品功能落到代码时的跨模块工程规则和实现约定 |
| L6 Code | 真实代码实现，不单独建 `_adoc/L6/` |

L1-L5 不是 L6 的纯索引。它们本身就是项目讨论、判断和经验沉淀的文档层，也为 Agent 修改 L6 提供上下文。

## 存：编织和反向编织

知识库需要持续维护。

初始化时，可以从项目现状整理出 L1-L5，这可以理解为正向编织。

开发、讨论、调研、debug、case 归档以后，如果产生了稳定信息，就需要 Weave / 反向编织，把它放回合适位置。

常见回写方向：

- 新核心概念、别名、常见误解：回写 L3。
- 新产品功能边界、业务规则：回写 L4。
- 新实现逻辑、跨模块规则、调试经验：回写 L5。
- 页面、弹窗、组件内部入口：回写局部 README。
- 高频检索路径：回写 `_adoc/map.md` 或细节 map。
- 一次性执行过程：留在 case / step。

Weave 不是聊天存档。它只保存未来任务可能复用的稳定判断。

## 取：通过 map 读取

Agent 不能每次都靠目录树和全文搜索猜入口。

`_adoc/map.md` 是项目记忆地图，负责第一跳检索：

```text
用户自然语言
-> map 命中入口
-> 读取必要 L3 / L4 / L5 / README
-> 进入 L6 代码
```

map 不承载完整正文。它更像路线图：告诉 Agent 这个业务词、页面、弹框、权限、接口或工程规则应该去哪里读。

总 map 负责第一跳，细节 map 和局部 README 负责最后一跳。这样既不把所有内容塞进一个超长文档，也不要求 Agent 在多层目录里自己猜。

## 第一层已经能独立使用

只完成知识库最小闭环，AIPD2 已经有实际价值。

例如：

- L3 出来以后，Agent 可以快速理解项目核心对象和领域语言。
- L4 出来以后，用户可以直接用自然语言描述某个功能、页面或弹框，Agent 通过 L4 和 map 找到入口。
- L5 出来以后，Agent 可以继承跨页面、跨模块、前后端协作和权限等规则。

这就是第一层的边界：让项目从“人临时喂上下文”变成“Agent 可以自己读项目认知”。
