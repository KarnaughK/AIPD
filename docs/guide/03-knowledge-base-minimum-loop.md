# 03 先建立最小认知闭环

这一章的目标不是填完一套文档模板，而是让 Agent 第一次进入项目时，能够沿稳定路径读到正确上下文。

## 初始化先建立骨架

安装 AIPD 后，在目标项目调用：

```text
/aipd
```

如果项目还没有初始化，AIPD 会建立类似结构：

```text
AGENTS.md
_adoc/
├── index.md
├── map.md
├── inbox.md
├── L1-intent/
├── L2-research/
├── L3-core/
├── L4-product/
├── L5-dev/
├── case/
├── okr/
└── sop/
```

初始化的正确结果不是“所有目录都写满”，而是：

- Agent 知道先读哪里。
- 项目最重要的稳定事实有位置可放。
- 当前高频任务可以从 map 命中入口。
- 暂时不知道归属的信息有 Inbox 接住。

## L1-L6 不是六份项目介绍

L1-L5 保存不同类型的长期项目认知，L6 是项目真实代码：

| 层级 | 回答的问题 | 例子 |
|---|---|---|
| L1 Intent | 项目为什么存在，长期取舍是什么 | 目标用户、方向、明确不做 |
| L2 Research | 项目面对的外部世界是什么 | 用户、场景、痛点、竞品、行业范式 |
| L3 Core | 项目靠哪些内部模型成立 | 核心对象、领域语言、业务流程、增长模型 |
| L4 Product | 核心模型落成什么用户能力 | 功能边界、业务规则、用户可见行为、实现入口 |
| L5 Dev | 功能怎样跨模块落到工程 | 权限、路由、第三方约定、平台规则、调试经验 |
| L6 Code | 最终实现在哪里、怎么工作 | 前端、后端、脚本和配置等真实源码 |

Case、Work Package、OKR 不属于 L1-L5 正文。它们记录当前目标和运行状态；长期知识库只保留已经稳定的项目认知。

## 先写最有用的，不要追求完整

第一次初始化后，优先补三类内容：

1. **方向边界**：Agent 最容易因为不知道“为什么”而偏离的长期取舍。
2. **核心语言**：用户说法、团队黑话和代码名之间的对应关系。
3. **高频入口**：常见任务应该读哪些规则、README 和代码文件。

不要试图从仓库反推一套百科全书。AIPD 更适合在真实任务中逐步增长：遇到一次稳定事实，确认归属，再通过 Weave 写回。

## Map-first 的第一次读取

假设你让 Agent 修改“订单退款弹窗”。理想读取链路是：

```text
用户说法：订单退款弹窗
-> _adoc/map.md 命中退款功能线
-> L3：订单 / 退款状态关系
-> L4：退款规则和用户可见行为
-> L5：权限、接口错误约定
-> 页面 README：弹窗组件与数据流
-> L6：真实实现
```

总 map 不承载完整正文，只暴露高频关键词、标准概念、权威入口和兜底搜索。页面内部最后一层地图放在代码就近 README，不要把所有细节塞回项目总 map。

## 一次最小检查

初始化后，可以让 Agent 回答一个真实问题：

```text
请按 AGENTS.md、_adoc/index.md 和 _adoc/map.md，
告诉我“退款弹窗”相关的产品规则、工程规则和代码入口分别在哪里。
不要修改文件。
```

检查三件事：

- 它是否先走 map，而不是全仓盲搜。
- 它是否区分产品规则、工程规则和局部实现。
- 如果 map 缺入口，它能否指出缺口和合适的回写位置。

到这里，AIPD 已经产生第一层价值：Agent 不再完全依赖你每次手动喂文件。

## 不要把 docs 当成项目认知

你正在读的 `docs/` 是 AIPD 的用户学习文档。目标项目里的 `_adoc/` 是给 Agent 执行任务时读取的事实源。两者读者和生命周期不同，不能互相替代。

更细的分层和检索规则，可按需查看 [ADOC 分层](../modules/adoc-layers.md)与 [Map 与检索](../modules/maps-and-retrieval.md)。
