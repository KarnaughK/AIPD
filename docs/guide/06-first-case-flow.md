# 06 第一次 case 执行闭环

AIPD2 不是单纯知识库。

知识库解决“项目长期记忆”，case / step 解决“这一次事项怎么组织、恢复、派发和验收”。

## 为什么不能只靠聊天

一次真实任务经常包含需求、猜测、否定、临时方案、后续想法和验收口径。

如果这些都只存在聊天里，Agent 开工时很容易混淆：

- 哪些是已经确认要做的。
- 哪些只是讨论过。
- 哪些已经被推翻。
- 哪些是不能做的边界。
- 应该读哪些上下文。

Case 的作用就是把这一次事项固定下来。

## Case 是什么

Case 是一次具体事项的文件化容器。

它通常记录：

- 目标。
- 场景和边界。
- 上下文索引。
- 要做和不做。
- step 列表。
- 验收标准。
- Weave 候选和归档状态。

Case 不是普通 todo。它的价值在于让长期任务可以恢复。

当上下文压缩或中断后，恢复链路应该是：

```text
AGENTS.md -> _adoc/index.md -> _adoc/map.md -> _adoc/case/index.md -> 当前 case.md -> 当前 step
```

## Step 是什么

Step 是 case 内可派发、可恢复、可验收的执行单元。

一个 step 应该写清：

- 本 step 的目标。
- 必读上下文。
- 任务清单。
- 验收标准。
- 不做范围。
- 执行记录。

执行 Agent 不应该靠调用方在 prompt 里展开完整任务正文，而应该读取 step 文件、case 文件和 step 列出的上下文。

## 分身 Agent 在哪里出现

case-run 中，每个 step 通常可以派发给 worker、explorer 或特定角色 Agent。

分身 Agent 不是低上下文工人，而是从主 Agent 当前认知 fork 出来的克隆体。它进入局部探索或执行分支，消化扫描、验证、修改、日志和 diff 等过程成本。

完成后，它只回流：

- 结论
- 依据
- 风险
- 建议
- 改动文件
- 验证结果

主 Agent 保留用户沟通、目标边界、最终判断和状态写回。

## 最小闭环

一次 AIPD2 开发任务的最小闭环可以这样理解：

```text
读取 map
-> 创建 case
-> 拆 step
-> 执行 step
-> 验收结果
-> 记录执行状态
-> 产出 weave 候选
-> 归档 case
```

到这里，你已经理解了 AIPD2 的基本结构：项目记忆负责让 Agent 找对上下文，case / step 负责让一次任务可执行、可恢复、可验收。
