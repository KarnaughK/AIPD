# 01 为什么 Agent Coding 需要项目记忆

本章只回答一个问题：Coding Agent 已经能自己搜索、读代码和执行命令，为什么项目还需要 AIPD？

## 从人喂上下文到 Agent 自己找

Vibe Coding 的早期形态更像局部协作：人知道要改哪里，手动 `@` 文件、贴背景、解释边界，AI 在一个小范围里生成或修改代码。

这种方式的问题是人的搬运成本高，但上下文控制权还在人手里。人挑错了文件，AI 才会读错；人挑对了，任务通常也不会跑太远。

Agent Coding 改变了这件事。Codex、Claude Code、Cursor Agent 等工具可以自己搜索仓库、读取文件、运行测试、修改多个模块，甚至调度其他 Agent。它们不再等待人把每个文件送到面前。

能力变强后，一个新的问题浮了出来：**Agent 能找到相关上下文，不代表它能找到正确上下文。**

## 一个常见失败

你让 Agent “给导出功能增加会员权限”。它很快找到：

- `export` 相关页面和接口。
- 一个公共 `hasPermission()` 方法。
- 会员套餐字段和权限枚举。

从代码形状看，把这些东西接起来很合理。但项目可能还有代码没有表达的事实：

- 导出权限属于组织能力，不跟个人套餐直接绑定。
- 公共方法只覆盖菜单显隐，不保证后端授权。
- 某个旧入口故意保留，是为了兼容历史数据。
- 上一次团队已经否决过“统一权限 helper”，因为它把两套语义混在了一起。

Agent 的搜索没有失败。失败的是项目没有把“为什么”和“边界”放在一条稳定可读的路径上。

## 代码、聊天和搜索各自缺什么

### 代码擅长表达现在怎么做

代码是最终实现事实，但它不一定记录产品取舍、历史原因、用户语言和跨模块约定。把这些都塞进注释，也会让实现和解释一起膨胀。

### 聊天擅长高带宽协作

聊天适合讨论、试探、反驳和快速补充背景。但它会压缩、会中断，也混有未采纳想法。整段聊天不能直接成为项目长期事实。

### 搜索擅长发现线索

全文搜索、语义搜索和 Agent 探索能发现候选文件，却很难单独判断哪个入口是权威的、哪些历史 Case 不该读、哪些规则只适用于某个模块。

所以项目需要的不是“再多写一些文档”，而是一套记忆机制：

1. 长期事实有明确归属。
2. Agent 能从自然语言快速命中正确入口。
3. 当前任务的目标和状态能从聊天转成可恢复文件。
4. 已验证的新经验能在任务结束后写回。

## AIPD 的切入点

AIPD 不让模型本身变聪明。它改变模型进入项目时能读到什么、任务变长时状态留在哪里、做完后什么会被下一次继承。

```text
项目长期认知：_adoc / map / 局部 README
当前目标状态：Case / phase / Work Package
最终真实实现：L6 代码
```

这三类事实分开维护，又在每次任务中被串起来。结果不是“Agent 永不犯错”，而是它少猜、错了更容易定位、长任务更容易恢复，项目也不会每轮从零开始。

## 你不需要先接受整套框架

一个只有几个文件的一次性原型，可能根本不需要 AIPD。一个长期项目也不必第一天填完 L1-L5、建立 OKR、创建多个 Case。

最小起点只有两件事：

- 给 Agent 一个明确的项目认知入口。
- 给高频问题一张能命中正确上下文的 map。
