# 01 从 Vibe Coding 到 Agent Coding

AIPD 不是从“要写一套文档规范”开始的。

它来自 AI Coding 使用方式变化后的一个现实问题：Agent 越会自己干活，项目上下文越不能只靠人临时喂。

## Vibe Coding 阶段

早期 AI Coding 更像局部协作。

人负责判断要改哪里，手动 `@` 文件、贴需求、解释背景。AI 主要在给定上下文里改一段函数、补一个组件、写一段脚本。

这种模式下，上下文控制权还在人手里。AI 不一定知道整个项目，但它也不需要知道太多。人把相关文件和边界挑出来，AI 在一个小范围里执行。

问题是：人的上下文搬运成本很高。每次任务都要重新解释背景，重新找文件，重新告诉 AI 哪些历史取舍不能动。

## Agent Coding 阶段

Claude Code、Codex、Cursor Agent 这类工具出现后，AI 不再只等人喂文件。

Agent 可以自己搜索、读文件、跑命令、定位入口、修改多个文件，甚至把一次任务拆成多个执行步骤。

这让 AI Coding 的效率上了一个台阶，但也带来新的不稳定性：Agent 能自己找上下文，不代表它总能找到正确上下文。

它面对的是一个真实项目：

- 代码只表达当前实现，不一定表达历史原因。
- README 通常只写入口和安装。
- Issue、PR、聊天记录太碎。
- 目录名和文件名不一定能暴露业务边界。
- 上一次 Agent 踩过的坑，下一次 Agent 不一定知道。

于是 Agent 会开始猜。

它可能读到局部代码后做出“看起来合理”的重构，却破坏了产品边界；也可能只看到当前实现，不知道这个实现是历史讨论后的取舍。

## AIPD 的切入点

AIPD 解决的不是“让 Agent 更会写代码”。

它解决的是：当 Agent 已经能自己执行任务时，项目如何让它稳定继承长期上下文。

这需要三件事：

- 项目有一套长期知识库，记录方向、场景、核心模型、产品边界和工程规则。
- Agent 能从用户一句自然语言稳定命中该读的上下文，而不是在项目里乱翻。
- 每次执行产生的新判断和经验能回写，下一次 Agent 可以继承。

这就是后面几章要讲的主线：

```text
项目知识库 -> L1-L5 存储结构 -> map 读取结构 -> weave 回写机制 -> case / step 执行闭环
```
