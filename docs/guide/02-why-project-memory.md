# 02 为什么需要项目知识库

Agent 进入项目后，代码能告诉它“现在怎么做”，但很难告诉它“为什么这么做”。

真实项目里，很多关键判断并不在代码里：

- 为什么这个功能先做 B 端，不做 C 端。
- 为什么这个字段展示成这个名字。
- 为什么这个接口不能复用另一个接口。
- 为什么这个页面没有抽组件。
- 为什么某个抽象以前试过，后来被放弃。

这些判断如果只留在聊天、会议、脑子、Issue 或 PR 里，下一次 Agent 进来就会重新猜。

## README 不够

README 适合做项目首页、安装入口和快速说明，但它不适合承载完整项目记忆。

一个持续迭代的项目会有方向、场景、核心模型、产品边界、工程约定、历史取舍和执行经验。把这些都塞进 README，会让首页膨胀成手册；不写进去，Agent 又读不到。

AIPD2 因此把这些内容放进 `_adoc/`。

`_adoc/` 不是普通说明文档，而是面向 Agent 的项目认知事实源。它的目标是让 Agent 在改代码前知道：

- 项目为什么存在。
- 当前要服务哪些场景。
- 哪些模型和术语是稳定的。
- 哪些功能边界不能随便改。
- 哪些工程规则需要跨模块遵守。
- 新经验应该回写到哪里。

## AIPD2 和其他工具的关系

AIPD2 不替代 Coding Agent。Codex、Claude Code、Cursor Agent 等工具负责推理、执行、工具调用和平台能力。

AIPD2 也不替代规格工具或开发纪律工具。它关注的是项目长期上下文、历史原因和经验沉淀。

可以粗略理解为：

| 工具 / 层次 | 主要解决的问题 |
|---|---|
| Coding Agent | 具体执行、工具调用、代码修改 |
| Superpowers 等流程工具 | AI 做事流程不稳定，需要开发纪律 |
| OpenSpec 等规格工具 | 单次变更需要可审查的规格和任务边界 |
| AIPD2 | 项目长期上下文、历史原因和经验沉淀 |

这些工具可以一起使用。AIPD2 提供项目记忆，Agent 和其他流程工具在这个记忆上执行任务。

## AI 让真正的问题变了

人类写代码时，代码量、重复和长文件很快会变成大问题，因为人写得慢、读得慢、批量修改也慢。

Agent 写代码以后，问题的重心开始变化。多生成一些代码、多改几处重复逻辑，通常不是最难的；真正难的是改一个地方牵出一片。一个 helper、service 或共享方法如果被很多业务场景依赖，Agent 做局部修改时就必须重新判断所有调用方的隐含语义。

所以 AIPD2 不只是在项目里多放一些文档。它更底层的目标是帮助项目降低上下文牵连：让任务尽量成为输入输出明确、上下文自足、可以局部验收的黑箱。更完整的解释见 [上下文解耦](../modules/context-decoupling.md)。

## 两套源码的心智模型

理解 `_adoc/` 后，可以用“两套源码”来概括 AIPD2 的心智模型。

一套是传统代码源码：`src/`、业务代码、脚本、配置。它告诉系统怎么运行。

另一套是 AI 认知源码：`_adoc/`、map、case、step、局部 README。它告诉 Agent 为什么这样设计、应该读哪里、任务边界是什么、经验如何继承。

`docs/` 的作用是帮助人理解这套体系；`_adoc/` 的作用是让 Agent 执行任务时直接使用这套体系。
