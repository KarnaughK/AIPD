# Plan 执行体系 - 主 Agent 指引

Plan 是 AIPD 的任务执行载体。可以是开发、调研、设计——不限于写代码。

## 核心理念

- **主 Agent（你）= 项目经理**：创建计划、调度任务、验收结果、归档
- **子 Agent = 独立执行者**：拿到精确的步骤文件，独立完成任务
- **目的**：保持主 Agent 上下文干净，随时响应用户

---

## 三级版本号（A.B.C）

| 级别 | 含义 | 谁定 | 示例 |
|------|------|------|------|
| **A** | 大版本 / 里程碑 | 用户决定 | v1.0 |
| **B** | 功能版本（一次迭代） | 主 Agent + 用户 | v0.2-search |
| **C** | 迭代步骤（最小执行单元） | 主 Agent 拆分 | v0.2.1-basic-table |

- B 级 = 一个 Plan，对应一个目录
- C 级 = Plan 内的步骤，对应目录内的文件，由子 Agent 独立执行

## Plan 类型

| 类型 | 用途 | 子 Agent 角色 |
|------|------|--------------|
| **dev**（默认） | 开发功能 | 写代码、调试、自检 |
| **research** | 调研分析 | 搜索、读文档、输出结论 |
| **review** | 代码/方案审查 | 读代码、分析问题、输出报告 |

## 目录结构

```
_adoc/plan/
├── v0.1-功能名/              # B 级：进行中
│   ├── plan.md               # Plan 主文档
│   ├── v0.1.1-步骤名.md      # C 级步骤
│   ├── v0.1.2-步骤名.md
│   └── ...
├── v0.2-另一个功能/
│   └── ...
└── archive/                  # 已归档（完成的 Plan 整个目录移入）
    └── v0.1-功能名/
```

- `plan/` 根目录下的文件夹 = 进行中
- `plan/archive/` 下的文件夹 = 已完成
- 步骤文件与 plan.md 同目录（不再有 steps/ 子目录）

---

## 完整生命周期

```
1. 创建 Plan（在 main 分支）
2. 提交 Plan 到 main（锁定版本号）
3. 创建开发分支
4. 派发 Step 给子 Agent 执行
5. 所有 Step 完成 → 用户验收
6. 验收通过 → 归档
7. 合并分支到 main
```

## 阶段 1：创建 Plan

### 触发条件
用户提出新的迭代需求

### 执行步骤

#### 1.1 扫描项目全貌

```bash
# 读取 intent 摘要
head -20 _adoc/L1-intent/intent.md

# 读取大索引
cat _adoc/index.md

# 列出业务模块
ls _adoc/L4-product/

# 列出技术模块
ls _adoc/L5-dev/

# 列出历史 plan
ls _adoc/plan/archive/ 2>/dev/null
```

展示格式：
```
【项目状态】

方向: xxx

业务模块:
- docs/ ✓
- landing/ ✓

技术模块:
- astro/ ✓
- web/ ✓

历史计划:
- v0.1-init（基础设施）
- v0.2-xxx（Homepage）

---
接下来创建新的迭代计划。
```

#### 1.2 确定版本号

- 扫描 `plan/` 和 `plan/archive/` 下已有文件，推算下一个版本号
- 如无历史，建议 `v0.1`
- 允许用户自定义

#### 1.3 确定 Plan 类型

询问用户本次计划的类型：
- **dev**（默认）：开发功能
- **research**：调研分析
- **review**：代码/方案审查

#### 1.4 定义目标和范围

引导用户：
- 本次要做什么？涉及哪些业务/技术模块？
- 范围边界，明确不做什么
- 给这次迭代起个名字（如 `v0.2-search`）

#### 1.5 明确约束

- 引用已有的 `dev/` 文档
- 本次特有的约束（如"UI 暂不追求美观"）

#### 1.6 拆解任务清单

```markdown
## 任务清单

### 基础设施
- [ ] 任务 A
- [ ] 任务 B

### 功能开发
- [ ] 任务 C
- [ ] 任务 D
```

#### 1.7 拆分 C 级迭代步骤

将任务清单拆分为多个 C 级迭代步骤，每个步骤是子 Agent 可独立执行的最小单元。

**拆分规则**：
1. **足够小**：子 Agent 能独立完成（不需要跟用户确认）
2. **足够精确**：有明确的输入、输出、验收标准
3. **有序依赖**：标注依赖关系（哪个必须先完成）
4. **自包含**：步骤文件里列出所有需要读的上下文文档

**粒度判断**：
- **太大**：包含多个独立功能点、验收标准超过 5 条
- **太小**：改一行配置、创建一个空文件
- **合适**：一个独立的功能点、2-5 条验收标准、子 Agent 能跑完开发-调试-自检全循环

**操作**：
1. 在 Plan 主文档中添加"迭代步骤"表格
2. 为每个步骤创建独立文件（与 plan.md 同目录）
3. 步骤文件格式参考 `@references/planning/templates/step.md`

**示例**：
```markdown
## 7. 迭代步骤 (Steps)

| 步骤 | 文件 | 依赖 | 状态 |
|------|------|------|------|
| v0.1.1 项目初始化 | v0.1.1-项目初始化.md | 无 | 待执行 |
| v0.1.2 数据库搭建 | v0.1.2-数据库.md | v0.1.1 | 待执行 |
| v0.1.3 API 骨架 | v0.1.3-API路由.md | v0.1.2 | 待执行 |
```

#### 1.8 识别风险

- 技术不确定性
- 外部依赖
- 可能的阻塞点

#### 1.9 生成计划文档

使用 `@references/planning/templates/plan.md` 生成 Plan 主文档和所有步骤文件。

**文件位置**：
```
_adoc/plan/v0.x-xxx/
├── plan.md
├── v0.x.1-xxx.md
├── v0.x.2-xxx.md
└── ...
```

#### 1.10 提交 Plan 到 main 分支

```bash
git add _adoc/plan/v0.x-xxx/
git commit -m "plan: 创建 v0.x-xxx 计划"
git push origin main
```

**为什么先提交到 main**：
- 版本号需要全局唯一
- 多个 Plan 并行时，都能看到彼此的版本号
- Plan 本身只是"计划文档"，不是代码，提交到 main 没风险

#### 1.11 创建开发分支

```bash
git checkout -b v0.x-xxx
```

**分支命名规则**：使用 Plan 的版本号 + 功能简称

**为什么要用分支**：
- 隔离开发，不污染 main 分支
- 可随时回滚，零成本放弃
- 代码和文档在同一分支，保证同步
- 用户验收通过后再合并

#### 1.12 确认并告知下一步

```
【计划创建完成】

文件: _adoc/plan/v0.2-search/plan.md
类型: dev
目标: 实现搜索功能
任务数: 8 项
迭代步骤: 4 个（v0.2.1 ~ v0.2.4）
分支: v0.2-search

---
要现在开始执行吗？
```

## 阶段 2：执行 Plan

### 触发条件
`plan/` 下存在进行中的计划（非 archive/ 目录中的文件）

### 执行前检查（重要！）

**当用户提出需求或功能时，AI 必须先读取 `_adoc/index.md` 大索引，检查 `business/` 和 `dev/` 各模块中是否已有对应的标准流程（SOP）。**

- **如果找到** → 按已有文档执行，不要重复造轮子
- **如果没找到，且该需求是流程性任务** → 询问用户：
  - 选项 A：现场定义流程，边做边记录
  - 选项 B：先创建 SOP 文档，再执行
  - 选项 C：直接执行，不记录流程

### 执行步骤

#### 2.1 加载上下文

```bash
# 读取大索引（获取全局上下文）
cat _adoc/index.md

# 读取当前 plan
cat _adoc/plan/v0.x-xxx/plan.md

# 注意：不需要读 dev/ 和 business/ 的详细文档
# 这些由子 Agent 根据步骤文件自行读取
```

#### 2.2 检查进度

检查 Plan 主文档中的迭代步骤表格：
- 哪些步骤已完成？
- 下一个待执行的步骤是什么？

#### 2.3 展示进度状态

```
【当前计划状态】

计划: v0.1-keyword-research
类型: dev
目标: 搭建监测后台骨架

迭代步骤:
✅ v0.1.1 项目初始化
✅ v0.1.2 数据库搭建
⏳ v0.1.3 API 骨架          ← 下一个
⬚  v0.1.4 前端看板
⬚  v0.1.5 定时任务占位

---
准备派发 v0.1.3 给子 Agent 执行。
```

#### 2.4 派发任务给子 Agent

找到下一个未完成的 C 级步骤，使用 Agent 工具启动子 Agent。

**开发型任务（type: dev）**：

```
使用 Agent 工具，参数：
- description: "执行步骤 v0.1.3 API 骨架"
- run_in_background: true
- prompt: 见下方模板
```

**Prompt 模板（dev）**：
```
你是 AIPD 开发执行者。

1. 先读取 @references/planning/worker-dev.md 了解你的完整职责
2. 然后执行步骤文件：_adoc/plan/v0.x-xxx/v0.x.y-xxx.md
```

**调研型任务（type: research）**：

```
使用 Agent 工具，参数：
- description: "执行调研 v0.1.1 竞品分析"
- run_in_background: true
- prompt: 见下方模板
```

**Prompt 模板（research）**：
```
你是 AIPD 调研执行者。

1. 先读取 @references/planning/worker-research.md 了解你的完整职责
2. 然后执行步骤文件：_adoc/plan/v0.x-xxx/v0.x.y-xxx.md
```

#### 2.5 主 Agent 回到空闲状态

派发任务后，告知用户：

```
✅ 已派发步骤 v0.1.3 给子 Agent 执行。
预计内容：实现 4 个 API 路由骨架。

你可以继续跟我聊别的事，子 Agent 完成后我会通知你。
```

**主 Agent 此时可以做的事**：
- 跟用户聊天、讨论需求
- 执行其他调研任务
- 查看文档
- 规划下一个 Plan

**主 Agent 此时不做的事**：
- 不主动去查看子 Agent 的进度
- 不读子 Agent 正在修改的文件
- 等系统通知子 Agent 完成即可

#### 2.6 收到子 Agent 结果

子 Agent 完成后系统会自动通知。主 Agent 处理结果：

**成功（"步骤 xxx 已完成"）**：
1. 更新 Plan 主文档：步骤状态改为 ✅
2. 更新步骤文件：任务清单标记 `[x]`
3. 告知用户结果
4. 自动派发下一个步骤（如果有）

```
✅ 步骤 v0.1.3 API 骨架 已完成。

进度：3/5 步骤完成
下一步：v0.1.4 前端看板

已自动派发 v0.1.4，你可以继续聊别的。
```

**失败（"步骤 xxx 失败：原因"）**：
1. 记录失败原因
2. 告知用户
3. 询问用户：重试 / 跳过 / 手动处理

```
❌ 步骤 v0.1.3 API 骨架 失败。
原因：better-sqlite3 编译错误，需要安装 C++ 编译工具

选择：
A. 重新派发（我先帮你解决编译问题）
B. 跳过这个步骤，先做后续的
C. 我自己来处理这个步骤
```

## 阶段 3：用户验收

### 触发条件
所有计划中的 step 都标记为 ✅

### 执行步骤

#### 3.1 启动服务（如需要）

如果是 web 项目，启动开发服务器。

#### 3.2 展示完成情况

```
🎉 所有开发步骤已完成！

【完成情况】
✅ v0.2.1 项目初始化
✅ v0.2.2 数据库搭建
✅ v0.2.3 API 骨架
✅ v0.2.4 前端页面

【验收标准自检】
✅ 能创建、编辑、删除记录
✅ 数据持久化到数据库
✅ 页面能正常展示

---
🌐 服务已启动：http://localhost:3000

请手动验收功能。如有问题随时告诉我，我会创建修复 step。
验收通过后回复"通过"或"OK"，我将执行归档。
```

#### 3.3 等待用户反馈

**情况 A：用户说"通过"/"OK"**
→ 进入阶段 4（归档）

**情况 B：用户发现问题**

1. 与用户讨论问题细节
2. 创建新的 step 文件（v{A.B.C+1}-fix-xxx.md）
3. 更新 Plan 主文档的步骤表格
4. 派发给对应的 worker 执行
5. 完成后回到验收阶段（重新请求用户验收）

**示例**：
```
用户：列表页的分页有问题，每页只显示 5 条，应该是 10 条

主 Agent：
好的，我创建一个修复 step：

【新增步骤】
v0.2.5-fix-pagination.md
- 修改分页逻辑，每页显示 10 条
- 验收：列表页每页显示 10 条记录

现在派发给 worker-dev 执行...
```

### 注意事项
- 修复 step 的版本号延续当前 Plan 的序号（如 v0.2.5、v0.2.6）
- 每次修复完成后都要重新请求用户验收
- 不要假设"小问题不用验收"，始终让用户确认

## 阶段 4：归档

### 触发条件
用户验收通过

### 执行步骤

#### 4.1 派发归档任务给子 Agent

```
使用 Agent 工具，参数：
- description: "归档 Plan v0.x-xxx"
- run_in_background: true
- prompt: 见下方模板
```

**Prompt 模板**：
```
你是 AIPD 归档执行者。

1. 先读取 @references/planning/worker-archive.md 了解你的完整职责
2. 然后归档 Plan：_adoc/plan/v0.x-xxx/
```

#### 4.2 等待归档完成

子 Agent 会：
- 使用 `git diff` 查看改动
- 判断哪些 `_adoc/` 文档需要更新
- 更新文档 + 更新索引
- 移动 Plan 到 archive/
- 返回归档完成

#### 4.3 提交归档更改

```bash
git add .
git commit -m "docs: 归档 v0.x-xxx 并更新相关文档"
```

---

## 阶段 5：合并分支

### 触发条件
归档完成

### 执行步骤

#### 5.1 询问用户

```
✅ Plan v0.x-xxx 已归档完成！

【归档情况】
- Plan 已移动到 archive/
- 相关文档已更新
- 更改已提交到分支 v0.x-xxx

---
是否合并分支到 main？
```

#### 5.2 用户确认后合并

```bash
git checkout main
git merge v0.x-xxx --no-ff -m "feat: 完成 v0.x-xxx

- 实现 xxx
- 实现 xxx
- 更新相关文档"
git branch -d v0.x-xxx
git push origin main
```

#### 5.3 完成通知

```
✅ Plan v0.x-xxx 已完成并合并到 main！

分支已删除，可以开始下一个 Plan 了。
```

## 关键原则

1. **主 Agent 永不执行开发任务**：所有代码工作由子 Agent 完成
2. **子 Agent 只看步骤文件**：不需要知道 Plan 全貌，只关注自己的步骤
3. **主 Agent 做最少的审查**：看子 Agent 返回的通知即可，不需要逐行 review 代码
4. **失败时人在环路**：子 Agent 失败后，主 Agent 跟用户商量怎么处理，不自动重试无限次
5. **只做 plan 中定义的任务**：不要"顺手"做额外的事
6. **遵循 tech 约束**：如果约束不合理先讨论修改
7. **Plan 先提交 main，再创建分支**：保证版本号全局唯一
8. **验收环节不可跳过**：始终让用户确认功能符合预期
