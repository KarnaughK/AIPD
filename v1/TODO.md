## 沉淀 SOP

**状态**: 待设计

**背景**: AIPD 有"读"的机制（核心行为：读 index → 读模块 → 读 SOP）和"做"的机制（Plan 执行），但缺少"写回"的机制——执行完之后，认知怎么回流到五层知识结构里。

**已达成的共识**:

- 沉淀不是独立动作，而是每件事的"收尾环节"
- 两种路径，同一个收尾：
  - 对话路径：聊完了 → 沉淀
  - Plan 路径：Plan 设计 → 执行 → 归档 → 沉淀
- 应该是一条统一的框架级 SOP，放在 `references/` 下
- skill.md 留钩子指向它，Plan 归档指引也加钩子（归档完提示"是否需要沉淀"）

**沉淀 SOP 的核心流程**:

1. 回顾这次做了什么（素材来自对话 or Plan 归档记录）
2. 判断产出归属（_adoc/ 的哪一层？还是 AIPD 框架自身的 reference？）
3. 判断新建模块还是追加到已有模块
4. 写入 + 更新索引

**覆盖的场景**:

- 对话沉淀：聊了方向/设计/决策后，结构化写入 _adoc/
- Plan 完成后沉淀：执行过程中产生的经验/规范/SOP 回流到 _adoc/
- 问题解决后沉淀：踩坑方案供后续复用
- 框架自身沉淀：reference/ 里的 SOP 本身也是"做完事后的总结"
