# AIPD Case Create

## 职责边界

**只做**：与用户讨论事项 → 按需扫描上下文 → 将上下文索引、边界和必要步骤写入 case.md → 按用户意图创建最少 steps

**不做**：不执行任何 step（执行用 `/aipd-case-run`），不写业务代码

### Case Create 生命周期

用户通过 `aipd-case-create` 启动任务后，当前对话默认处在 case 创建、方案讨论、上下文整理、文档补充和 step 拆解阶段。只要用户没有明确结束这个阶段，就不要进入执行。

结束信号只认明确表达：

- 用户说“设计结束”“case create 到这里”“可以开始干活”“开始执行”等。
- 用户显式调用 `/aipd-case-run` 或 `aipd-case-run`。

如果用户说“开始干活”但没有调用 `aipd-case-run`，提醒用户切换到 `aipd-case-run` 执行，不要在 `case-create` 里直接执行 step 或写业务代码。

这个边界不依赖任务复杂度判断。不要先判断“前端是否复杂”再决定能不能执行；只要仍在 `case-create` 生命周期内，就只做设计、建模、上下文沉淀和 step 准备。

### Case Create 自身的渐进式披露

`aipd-case-create` 这个 skill 自身也要渐进式工作。不要一上来把所有 case 模板、所有执行规则、所有技术栈 SOP 都塞给用户；先把当前对话处在 AIPD 上下文空间里的什么位置说清楚，再按场景加载对应流程。

进入 case-create 后，先轻量说明三件事：

1. **当前阶段**：这里是 case 创建 / 方案讨论 / 上下文整理 / step 设计阶段，不是执行阶段。
2. **Case 概念**：case 是一次具体事项的上下文容器，负责沉淀目标、场景分流、上下文索引、边界、候选事项、必要 steps 和观察锚点。
3. **下一步判断**：先判断项目类型和 case 类型，再决定加载哪条 SOP；不同技术栈和不同认知层级的 case，不共享同一套执行细节。

推荐开场不是长篇解释，而是短句确认当前状态，例如：

```md
我先按 `case-create` 处理：这里不执行代码，只确认 case 目标、场景分流、上下文索引和必要 step。先判断这是前端实现、后端实现、综合模块，还是 L2/L3/L4 目标型研究。
```

渐进式披露规则：

- 用户只是在描述目标时，只解释 case-create 当前要判断什么，不提前展开完整模板。
- 用户开始讨论上下文时，再披露上下文检索包：层级判断、必读文档、代码入口、兜底搜索、风险边界。
- 用户开始设计 step 时，再披露 step 拆分草案规则：先聊“第几步干什么”，确认后再写详细 step。
- 用户进入 Vue 前端实现型 case 时，再按需加载 `@references/L5-dev/vue-case-create-guide.md`；涉及 Mermaid / `.mmd` 架构图的创建、修改、评审或预览时，交给 `aipd-mermaid` 处理领域规范和预览逻辑。
- 用户进入 React、后端、综合项目或 L2/L3/L4 目标型 case 时，只参考通用 case-create 方法，不照搬 Vue 前端实现 SOP。

逐层确认规则：

- “一层一层推进”不是把方案按层级写得很完整，也不是一次给出从第一层到第四层的完整推演。
- 它的意思是：一次只推进当前这一层，先让用户确认当前层的判断是否成立；如果第一层或第二层方向错了，后续层级不应被提前展开成细节方案。
- 可以提示下一层要讨论什么，但不要替用户把下一层、下下层的目录、step、实现细节一并确认。
- 用户确认当前层后，再进入下一层；用户纠偏时，先收束当前层，不要继续推进后续层。

原则：case-create 提供的是分层判断和推荐流程，不是固定状态机。Agent 应相信自己的项目识别能力，但必须把判断依据写进 case 的“场景分流”和“上下文索引”。

## 执行流程

### 第一步：判断项目类型和 case 类型

触发后先判断“这个项目是干什么的”和“这个 case 到底要干什么”。不要默认把最近一次 Vue 前端项目里的经验套到所有 case。

可以先用项目基础信息自动识别：

- 目录下只有前端代码，存在 `package.json`、`src/views`、`src/components`、Vue / React / Vite / Nuxt / Next 等特征：大概率是前端实现型 case。
- 目录下同时有前端、后端、数据库、服务编排或多语言模块：属于综合项目，必须先问清楚本次 case 的目标范围，再加载对应流程。
- 目录下主要是服务端框架、接口、数据库、队列、定时任务、脚本或基础设施：大概率是后端 / 平台实现型 case。
- 用户目标是研究 L4 / L3 / L2、梳理产品边界、核心对象、用户场景、OKR 或方法论：属于目标型 / 认知型 case，不按前端代码实现流程拆 step。

如果项目类型或 case 类型不清楚，先问一句最小确认：

```md
这个 case 是偏前端实现、后端实现、综合项目里的某个模块，还是 L2/L3/L4 这类目标型研究？
```

分流原则：

- Vue 前端实现型 case：先读取 `@references/L5-dev/vue-case-create-guide.md`；需要创建、修改、评审或预览 Mermaid / `.mmd` 架构图时，使用 `aipd-mermaid`，由它按领域加载 Vue 架构图和 provider 规范。
- React 前端实现型 case：可以参考前端流程，但不要照搬 Vue 组件、provider、SFC、`aipd_vue_architect` 等细节；按 React 项目结构重新判断组件、hooks、路由、状态和测试入口。
- 后端实现型 case：重点通常变成接口契约、领域模型、数据表、事务、权限、队列、幂等、日志、测试和部署；不要套用前端组件图和目录落位规则。
- 目标型 / 认知型 case：重点是问题定义、资料来源、判断框架、观察锚点和 Weave 候选位置；step 未必对应代码修改，也未必需要架构图落到代码目录。

相信大模型的判断能力，但必须把判断写进上下文索引：项目类型、case 类型、为什么选择这条流程、哪些近期经验只作为参考不能直接套用。

### 场景 SOP 按需加载

`case-create` 主入口只保留场景分流，不展开具体技术栈 SOP。命中具体场景后再读取对应 guide：

- Vue 前端实现型 case：读取 `@references/L5-dev/vue-case-create-guide.md`。
- Mermaid / `.mmd` 架构图：创建、修改、评审、按需预览都路由到 `aipd-mermaid`；case-create 只负责判断图是否属于本 case 设计输入，以及是否需要落位到代码就近目录。
- Vue `useXxx.ts/js`、provide / inject、页面级 API 数据源：普通 case 方案可读取 `@references/L5-dev/vue-provider-guide.md`；如果是在图中表达这些边界，交给 `aipd-mermaid` 按领域加载。
- React / Next / Nuxt / Express 等后续技术栈：新增对应 guide，不把细节继续塞进本文件。

### 第二步：问用户要做什么

触发后不要先大范围扫描。如果用户目标不清楚，先直接问用户要完成什么；如果用户已经说明目标，就只做必要的轻量识别和上下文检索。

先判断用户这次是要：

- **完整 case 拆解**：用户明确要制定执行计划、拆步骤、准备连续开发。
- **框架 / 上下文搭建**：用户说“先看一眼”“先把上下文给我”“外围框架处理好”“先建 case”“不要多做”“后面逐个调整”等。

如果是框架 / 上下文搭建，默认只创建 case 框架和一个上下文整理 step。不要预先创建未来业务调整 step，除非用户明确要求拆出后续步骤。

底层逻辑：逐层确认，不要企图一步到位。用户说清一个点，就只推进这个点；当前层确认后再进入下一层。不要一次性把后续三四层的目录、步骤、实现细节都展开，因为用户可能在第一层或第二层就改变方向。有需要执行、恢复或验收时再固化成 step；没需要时继续聊，把可能有价值的后续事项放入候选区。

如果用户在 `case-create` 中继续补充业务流程、接口规则、组件结构或异常分支，这些都仍属于当前 case 的设计输入。把它们写入 case doc、上下文索引、候选区或已确认 step，不要把讨论本身包装成执行 step。

如果用户明确认可了某段方案预览、叙事路径、目录草案、接口设计、组件关系、执行策略或关键取舍，并且后续 step 会依赖它，不能只写一句缩略关键词。必须把它以“已确认设计输入 / 执行依据”的形式写入 case.md、case doc 或对应 step。压缩只能压掉口语和重复，不能压掉推理顺序、关键取舍和示例结构。

### 第三步：上下文检索

创建 case 前，先把用户描述转成上下文检索包。这个动作不是执行 step，而是为了避免 case 建在错误上下文上。

根据用户描述，精准读取相关文件：
1. 主 Agent 读 `_adoc/index.md` 了解项目模块分布（一次，轻量）。
2. 读取 `_adoc/map.md`，判断用户意图对应的 L3 / L4 / L5 / 局部 README / L6 代码入口；当前流程是 case 创建，因此只读取当前要创建 / 关联的 case 状态，不把历史 case 当作普通上下文来源。
3. 优先寻找本次事项相关的核心概念索引、产品功能文档、工程实现规则、页面 README、设计/原型资料和代码入口。
4. 需要探索、验证、批量读取或整理 case 草案时，按 Agent Entry 中的分身 Agent 逻辑 fork 主 Agent 克隆体，主 Agent 只审阅结果回流。

上下文检索包至少包含：

- 层级判断：本次涉及 L3 / L4 / L5 / 局部 README / L6 中哪些层；case 只记录本次事项状态和上下文索引。
- 必读文档：执行前必须读的 `_adoc` 文档、页面 README、设计/原型资料。
- 代码入口：页面、弹窗、组件、接口、权限工具、路由或配置文件等入口。
- 兜底搜索：缺少地图或命中不清楚时可用的 `rg` 关键词。
- 风险边界：概念歧义、产品边界、跨端约定、权限码/接口字段一致性等。
- 场景分流：项目类型、case 类型、适用流程，以及哪些 Vue / 前端经验只能参考不能照搬。

**原则**：主 Agent 负责和用户讨论目标、边界、取舍和验收口径，不直接消费高噪声上下文。分身 Agent 是主 Agent 的克隆体，不是低上下文工人；它带着同样的上下文进入局部探索分支，用于计划、探索、压缩信息和整理草案，然后只回流结论、依据、风险、建议和 case 草案摘要。

`case-create` 阶段的分身不直接执行业务 step。代码探索、接口核对、组件关系梳理和批量读取可以交给分身，但分身回流的结果仍由主 Agent 与用户确认后写入 case / doc / step。

如果分身用于上下文检索，派发目标必须明确为“只返回上下文包”。不要让分身直接创建 step、写代码或扩展成完整执行计划。

### 第四步：与用户确认方案

把分身 Agent 返回的关键认知和 case 草案汇总给用户，讨论执行方案，确认上下文索引、边界和步骤拆分。

如果用户已经明确说“不要多做”或只要求搭框架，不要把可能的后续工作自动拆成 step 文件；最多在 `case.md` 的“后续候选事项 / 待拆分”中用文字列出，等待用户确认后再创建 step。

### 第五步：生成 case 文件

```bash
mkdir -p _adoc/case/c{X.Y}-{名称}/steps
mkdir -p _adoc/case/c{X.Y}-{名称}/doc
```

**case.md 必须包含**：
- 目标：这个 case 要完成什么
- 场景分流：项目类型、case 类型、本 case 适用的流程，以及不适用的近期经验
- 上下文索引：执行时优先读哪些 `_adoc` 文档、页面 README、代码入口、设计/原型/调研资料；必须包含层级判断、必读文档、代码入口、兜底搜索和风险边界
- 本次边界：明确做什么和不做什么
- steps 列表：只列已确认要执行的 step；框架搭建型 case 可以只有一个上下文整理 step，甚至暂时没有可执行 step
- 后续候选事项：未确认的调整点只记录为候选，不创建 step 文件
- 验收标准和 Weave 反向编织候选位置
- 自迭代观察锚点：后续审计 transcript / session 时，检查 Agent 是否按本 case 的上下文索引和 AIPD 检索流程工作
- Weave 候选位置：只记录可能需要回写到 L3 / L4 / L5 / 局部 README / map 的稳定知识方向；一次性过程、验收记录和临时决策留在 case / step

已确认的步骤文件写入 `steps/`，格式参考 `@references/case/templates/step.md`。

### Mermaid / `.mmd` 路由规则

case-create 不直接维护 Mermaid 写法、领域图规范或预览脚本。涉及 Mermaid / `.mmd` 架构图时，统一路由到 `aipd-mermaid`：

- 创建 / 修改图：让 `aipd-mermaid` 读取当前项目 map、目标图和对应领域规范。
- 评审图：让 `aipd-mermaid` 检查图的边界、节点职责、线条语义和是否过度塞细节。
- 预览图：只有用户明确说“预览、渲染、看图、生成 PNG”时，才让 `aipd-mermaid` 执行预览。

这个动作属于 case-create 的设计讨论，不算执行业务 step。用户没有要求看图时，不要主动渲染。

### 架构图落位规则

如果 case-create 产出了会指导后续代码实现的 `.mmd` 架构图 / 组件图，不要让它长期只停留在 `_adoc/case/{case}/doc/`。创建 step 拆分方案时，必须判断这些图是否需要落到目标代码目录就近位置。

默认规则：

- 如果目标代码目录还不存在，或本次实现会新建弹框 / 页面 / 模块目录，拆分草案的第一个实现 step 应先创建目标目录，并把实现需要的架构图落到该目录。
- 如果目标代码目录已经存在，第一步不强行建目录，但要把架构图落到现有就近 README / `_map.md` / `*-map.mmd` 位置，或在 step 中明确为什么不需要落位。
- 长期指导实现和维护的图，优先**移动**到代码就近目录，case.md 只保留链接；避免 case 和代码目录维护两份会漂移的图。
- 只作为设计过程记录、方案备选或已废弃判断的图，可以留在 case `doc/`，但必须在 case 里标明它不是实现侧事实源。
- 不要为了落图创建空泛文档层。图应放到最接近后续修改入口的位置，例如目标页面目录、弹框目录或模块根目录。

Step 拆分草案中应体现这个落位动作，例如：

```md
- cX.Y.1：创建目标弹框目录，并将 case 中的组件图移动到弹框目录就近位置，case.md 改为链接该图。
- cX.Y.2：基于已落位的组件图实现弹框骨架和数据入口。
```

### 自迭代观察锚点

case-create 生成或更新 case 时，应在 case 中留下 3-6 条可审计的观察锚点。它们用于后续 `aipd-learn` 判断 AIPD 机制有没有按设计生效，而不是只等用户多次纠正后才发现问题。

观察锚点优先围绕：

- Agent 是否读取 `_adoc/map.md` 或说明缺失，并用 `rg` / README 兜底。
- Agent 是否读取本 case 上下文索引里的 L3 / L4 / L5 / 局部 README。
- 涉及核心概念时，Agent 是否先对齐 L3 术语和关系，而不是直接猜业务含义。
- 涉及跨模块工程规则时，Agent 是否读取 L5 实现规则，而不是只看目标页面代码。
- 涉及页面、弹窗、组件内部细节时，Agent 是否读取就近 README。
- 如果执行偏离 SOP，偏离点是提示词没看清、地图没命中、文档缺失，还是 AIPD 流程设计不够硬。

### Step 创建边界

创建 step 前先做意图校准。用户说“开始设计 step / 开始拆 step / 写一下 step”时，表示进入**编写 step 这个大模块的阶段**，不表示可以立刻落详细 step 文件。

编写 step 的第一步是先输出轻量拆分方案，和用户确认“第几步要干什么”：

```md
【Step 拆分草案】
- cX.Y.1：{一句话说明这一步要干什么}
- cX.Y.2：{一句话说明这一步要干什么}
- cX.Y.3：{一句话说明这一步要干什么}

待确认：
- 这个拆分粒度是否合适？
- 有没有哪一步太粗、太细，或顺序不对？
```

只有用户确认拆分方案基本没问题后，才写详细 step 文件，包括上下文文档、任务清单、验收标准和不做边界。用户纠正“太少 / 太粗 / 顺序不对 / 好好查一下”时，先重新读取必要上下文并重出拆分草案，不要直接补写文件。

意图校准参考：

| 用户表达 | 默认处理 |
|---|---|
| “先把上下文 / 框架处理好” | 创建 case 框架，最多创建 `cX.Y.1-context-*` |
| “千万不要多做 / 先别改业务” | 不创建未来业务调整 step |
| “后面逐个调整 / 后续再说” | 在“后续候选事项”记录，不创建 step |
| “开始设计 step / 开始拆 step / 写一下 step” | 进入 step 编写阶段，先展示轻量拆分草案，确认后再写详细 step |
| “帮我拆一下步骤 / 制定执行计划” | 可以拆多个 step，但先展示轻量拆分草案让用户确认 |
| 用户明确列出多个独立任务并要求建 case | 按任务拆 step |

不要因为模板里有多个 step 示例，就机械创建 `cX.Y.2`、`cX.Y.3`。每个 step 都会增加用户审核成本，未确认的 step 是噪声。

不要为了显得完整而一次性规划所有步骤。case-create 的交付重点是“当前最小可用上下文 + 已确认的下一步”，不是完整路线图。可以说明后续可能会继续讨论哪些层，但不要把未确认层级写成已经成立的细节方案。

step 只能表示“已确认、可派发执行”的工作。正在讨论的问题、设计阶段缺口、需要继续确认的接口或组件关系，写入 case doc 或候选区，不创建“设计确认 step”“当前讨论 step”这类把 `case-create` 自身工作包装成 step 的文件。

### Step 拆分上下文隔离规则

设计 step 前先读取 `@references/case/step-context-isolation.md`，用它检查拆分边界。

step 的核心不是按组件、文件或人力分工机械切块，而是按**上下文是否能独立执行**切块。一个 step 应该能只凭自己的 step 文件、case.md 和列出的上下文文档完成。如果两个 step 之间必须共享上一轮未沉淀的临时判断、半成品方案或隐形状态，优先合并成一个 step，或者把共享判断先沉淀到 case 文档 / 局部 README / 架构图里，再拆分。

聊天里已经被用户确认、且本 step 会依赖的方案，不属于“可省略背景”。创建详细 step 时，必须写入 `## 已确认设计输入` 或等价区块，让执行 Agent 不需要回忆聊天就能继承这段判断。

大型任务和超长任务优先写成 case / step，不直接把长任务正文交给子 Agent。子 Agent 的派发 prompt 应只包含领域指引链接、step 文件链接、case 文件链接和返回格式；任务清单、上下文文档和验收标准都写入 step 文件。

### Step 推荐 Agent / 指引规则

创建 step 时尽量声明 `推荐 Agent`：

- 调研、只读定位、资料整理：`explorer`
- 普通开发、修复、脚本和文档修改：`worker`
- Vue 页面、Vue 组件、HTML/CSS、组件通信、前端状态组织：`aipd_vue_architect`
- Vue `useXxx.ts/js`、provide / inject、页面级 API 数据源、详情数据下发、provider/controller 边界：`aipd_vue_provider`
- 不确定时留空，并在 step 目标和上下文文档里写清楚判断依据

推荐 Agent 是执行角色建议，也是一份领域执行指引。平台支持 custom agent 时，`case-run` 优先派发对应 Agent；平台不支持或注册失败时，`case-run` 必须降级为 `worker/explorer + @references/agent-guides/{推荐 Agent}.md`，让执行 Agent 直接读取这份指引。

因此，Vue 相关 step 的上下文文档不需要把整份指引复制进去；只要推荐 Agent 写对，`case-run` 会负责把对应指引纳入执行 prompt。推荐 Agent 不替代 step 的任务边界和上下文文档。只有当 step 强依赖 Main Agent 当前尚未沉淀的聊天判断时，才需要 fork 上下文分身。

### 第六步：告知下一步

如果已经有明确可执行 step，告知用户执行 `/aipd-case-run` 开始执行。

如果只是搭好 case 框架和上下文索引，告知用户：当前 case 已可作为后续讨论入口，等用户确认具体调整点后再补 step。
