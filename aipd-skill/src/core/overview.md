# AIPD 框架总览

AIPD 把软件项目从构想到交付过程中产生的经验和决策，按领域规则结构化沉淀为项目认知，让 AI 能基于这些认知持续、准确地参与项目。

## 核心原则

- **经验总结型框架**：边实践边迭代，框架随项目实践持续演进
- **_adoc/ 是服务于 AI 的项目知识库**：只存整理过的文档，不存原始数据和一次性脚本
- **文件优先的上下文承接**：聊天只是运行缓存，文件才是长期事实源；会影响后续恢复路径的小步确认、状态变化、设计边界和下一步游标，都应及时写回 case、phase、work package、README 或 map
- **任务上下文解耦**：在人与 AI 编写任务、设计 case、对齐目标时，优先找到复杂度爆点并做最小必要解耦，让执行可以横向铺模块

## 文件优先的上下文承接

AIPD 不把长期任务连续性建立在聊天上下文上。聊天负责推进讨论和执行，文件负责承接状态、边界和恢复入口。

当一次确认、调研准备、设计决策、phase 跳转、work package 派发或执行结果会影响后续判断时，Agent 应先把它写成可恢复的文件 checkpoint，再进入下一步。这样上下文压缩、中断、换 Agent 或长任务续跑时，都能沿 `AGENTS.md -> _adoc/index.md -> _adoc/map.md -> case.md -> 当前 phase -> 当前游标文件 / work package` 恢复，而不是依赖聊天记忆。

这不是要求每句话都落文件。只有具备恢复价值的信息才需要固化：当前问题、确认结论、open / assumed / deferred、调研边界、停止条件、返回位置、下一步游标、执行记录和受影响下游。临时闲聊、未采纳想法和可由现有文档自然推导的重复信息，不应膨胀文档。

## 上下文解耦

很多记忆方案关注上下文关系串联：现在说过的一句话，未来某个阶段可以被重新提取，并和新上下文建立关联。AIPD 仍然需要项目知识库来增强 AI 获取上下文的能力，但在任务执行侧，它更重视把任务上下文拆成可独立理解的小黑箱。

这种解耦不是否认系统内存在关联，而是把关联上移到规则层。AIPD 通过 L1-L5、map、局部 README 以及 case / work package 的流程上下文索引，把“必须共同遵守的东西”沉淀为全局约定；L6 是项目里的真实代码实现，不单独进入 `_adoc/`。人与 AI 在编写任务前，再把输入输出、边界、必读上下文和验收口径写清楚。

这样做的收益是：每个小任务范围足够小，AI 可以在黑箱内部自由实现，只要符合输入输出和验收逻辑即可。代码细节未必总是最优，但在局部模块足够小、边界足够清楚时，它通常可用；有需要再调，没需要不调。

更完整的核心认知见 `_adoc/L3-core/index.md`。

## L1-L6 分层

从抽象到具象组织项目知识，AI 按需加载。L1-L5 是 `_adoc/` 中的项目认知，L6 是项目代码本身：

| 层级 | 路径 | 解决的问题 |
|------|------|-----------|
| L1 | `_adoc/L1-intent/` | 项目方向、目标和长期取舍，让 AI 不偏离目标 |
| L2 | `_adoc/L2-research/` | 方向所处的外部世界，包括用户、场景、需求、痛点、竞品、行业/玩法范式和调研结论 |
| L3 | `_adoc/L3-core/` | 项目内部靠哪些核心模型成立，包括核心对象、领域语言、核心流程、数据模型、增长模型或商业模型 |
| L4 | `_adoc/L4-product/` | 把 L3 核心模型落成产品功能、功能边界、业务规则、用户可见行为和相关实现入口 |
| L5 | `_adoc/L5-dev/` | 产品功能落到代码时，跨模块、跨平台、跨页面都要遵守的业务实现逻辑和工程规则 |
| L6 | 具体代码实现，不在 `_adoc/` 内 | 最终产出的真实实现；前端、后端、爬虫、脚本等项目类型各自拥有不同代码入口 |

L1-L5 的详细指引见各层 `guide.md`；L6 直接对应具体项目代码。

## 多 Agent 架构

- **主 Agent = 项目负责人**：与用户对话、规划任务、调度分身 Agent、审查结果
- **分身 Agent = 主 Agent 克隆体**：从主 Agent 当前认知 fork 出来，进入局部探索分支完成开发/调研/归档

**核心设计哲学：主 Agent 主线是稀缺资源，分身 Agent 的分支过程成本是可消耗资源。**

AIPD 不把分身 Agent 当成低上下文工人，也不靠大量 if-else 规则规定什么时候派工。更准确的模型是：主 Agent 留在当前主线维护用户沟通、目标边界和最终判断；分身 Agent 带着同一份认知去消化探索过程，完成后只把结论、依据、风险、建议、改动文件和验证结果回流给主线。

同样一件事，主 Agent 直接做可能消耗 100K 主线 token；fork 分身 Agent 做，分身 Agent 可以多消耗，但主 Agent 只接收压缩结果。AIPD 不优先优化总 token 消耗，而是优化主 Agent 的单轮上下文消耗和任务连续性。

详细调用方式见 `agent-guide.md`。不同平台的具体分身 Agent 调度方式由构建时的平台覆盖文件提供。

## 执行体系

- **Case**：把短周期目标写入 `case.md` 的 Case Contract，再按 Think / Design / Execute / Verify / Close 推进，由主 Agent 维护目标边界和状态
- **Work Package**：Case 内可派发、可恢复、可验收的目标包，不是微步骤
- **OKR**：追踪飞书阶段目标与关键结果，对齐 AI 的执行方向
- **Weave**：把已完成、已实现、已验收的稳定信息编织回 L3 / L4 / L5 / 局部 README / map；未完成 case 中的候选先留在 Close 归档候选
