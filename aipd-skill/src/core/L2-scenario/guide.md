# 第二层：Research — 外部世界调研

> 状态：未经实践验证的初始文档。方向性参考，非 SOP。

## 这一层在做什么

从项目方向（Intent）出发，发散探索这个方向所处的外部世界。

L2 不只是痛点层。痛点是 L2 的一种产物，但 L2 的范围更宽：用户、场景、需求、竞品、行业范式、玩法范式、市场观察、流量来源、变现方式和替代方案，都可以先放在这里。

重点是"发散"而非"收敛"。这一层的价值在于尽可能多地理解外部世界，而不是急于定义解决方案或内部模型。

## 可能涉及的活动

- 用户访谈、竞品调研、行业分析
- 从自身经验中提取痛点（"我在做 X 时，最烦的是 Y"）
- 场景故事：描述一个具体的人在具体的情境下遇到了什么问题
- 需求分类：哪些是核心痛点，哪些是锦上添花
- 反向思考：如果不做这个项目，用户会怎么凑合？凑合的代价是什么？
- 同类产品、同类网站、同类玩法、内容结构、增长方式和商业模式观察
- 搜索需求、传播路径、用户回访方式、广告/订阅/内购/联盟等变现形态记录

## 灵感与思路

**先记录外部事实，不急着内化成模型**

L2 里的资料可以很杂。竞品网站、用户反馈、市场观察、玩法截图、SEO 入口、广告形态、用户现有替代方案，都可以先作为外部事实记录下来。

只有当这些外部事实被抽象成“我们自己的长期成立方式”时，才进入 L3。例如：

- "某猜词网站有每日挑战" 是 L2 竞品观察。
- "每日挑战制造回访和分享，是本站留存模型的一部分" 才是 L3 成立模型。

**从"人"出发，不从"功能"出发**

一个常见的错误是直接列功能清单（"要有搜索、要有导出、要有权限管理"）。这一层应该先回答：谁在什么场景下会遇到什么问题？

比如：
- "一个运营同学每天要从三个后台导出数据，手动合并成一张表，花 40 分钟" — 这是场景
- "需要数据导出功能" — 这是功能，不是场景

**痛点的层次**

不是所有痛点都值得解决。可以用这个视角分层：
- 频率：这个问题多久出现一次？
- 严重度：出现时影响多大？
- 现有方案：用户现在怎么绕过的？绕过的成本高吗？

**从边缘场景找核心**

有时候最有价值的洞察藏在边缘场景里。主流场景大家都能想到，但边缘场景往往暴露系统的本质约束。

比如做提示词管理工具时：
- 主流场景：管理和编辑提示词
- 边缘场景：一个提示词改了一个字，线上效果崩了，怎么回滚？
- 洞察：版本管理和回滚才是核心需求，不是编辑器

**竞品和商业资料也属于 L2**

如果项目是一个游戏网站、内容网站、工具站或 SEO 型产品，外部世界不只有用户痛点，还包括流量从哪里来、用户为什么回来、别人怎么变现、页面如何承接搜索需求。

这些资料先放 L2。后续如果其中某些判断变成项目自己的增长模型、内容模型、流量模型、留存模型或商业模型，再提炼到 L3。

**不要过早过滤**

这一层允许"不靠谱"的想法存在。过早过滤会丢掉有价值的信号。先记录，后面在 L3 Core 再做提炼。

## 产出形式

没有固定格式。可以是：
- 一份场景清单（每个场景一段话）
- 用户旅程地图
- 痛点矩阵（频率 × 严重度）
- 竞品和同类玩法清单
- 市场、流量、内容和变现观察笔记
- 甚至只是一堆零散的笔记

关键是：记录下来，不要只存在脑子里。后续的 L3 Core 需要从这些外部素材中提炼项目自己的核心成立模型。
