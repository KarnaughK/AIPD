# AIPD2 - AI 原生软件开发框架

AIPD2 不是 Vue、Next 或 Spring 这样的代码框架。它面向的是 AI Agent 深度参与的软件开发过程：把项目知识、上下文检索、任务执行、Agent 协作和经验回写组织成一套可持续运转的开发框架。

---

AI Agent 已经能写很多代码。

但它让软件开发暴露出一个更底层的问题：过去很多开发规范，原本是围绕人的认知限制长出来的。

人不能一次稳定阅读太多上下文。人类写代码时，需要把复杂系统切成短函数、小文件、清晰分层和稳定接口。一个方法十来行，有输入和输出，人读完实现以后，就可以在脑子里把细节压缩成一个名字，然后继续理解下一个方法。

这套规范非常合理。它是在替人类节省工作记忆。

但 AI Agent 的工作方式不一样。

当前的 Agent 更擅长直接接收一大段上下文，在上下文窗口里把逻辑现读现盘清楚，然后快速生成和修改代码。它并不会像一个熟练工程师那样，稳定地把很多小块实现总结成长期工作记忆，再带着这些压缩块继续往后推理。

于是，过去服务于人的抽象方式，在 AI 面前经常反过来变成负担：

- 对人来说，短函数和命名是认知压缩。
- 对 Agent 来说，如果实现不在附近，函数名可能只是一个需要跳转的黑箱。
- 对人来说，横向分层让职责清楚。
- 对 Agent 来说，横向分层可能把一次任务的输入、处理、业务判断和输出拆散到很多文件里。
- 对人来说，复用减少手写成本。
- 对 Agent 来说，共享 helper 和 service 可能让一个局部修改牵出一片隐性依赖。

AIPD2 从这个判断出发：AI 时代的软件开发，不只是要让 Agent 会写代码，而是要重新组织项目上下文，让 Agent 能稳定理解、执行、恢复和回写。

一句话：

**传统代码组织是在替人类减少上下文；AIPD2 试图替 Agent 保留正确的局部上下文。**

## AIPD2 解决什么问题

AIPD2 解决的不是“Agent 不会写代码”，而是：

- Agent 进入项目后，不知道这个项目为什么长成现在这样。
- 代码能告诉 Agent 现在怎么做，但很难告诉它历史原因和取舍。
- 聊天记录会断，Issue 和 PR 太碎，README 通常只写安装和入口。
- 一次任务聊久了以后，需求、猜测、否定、临时方案和稳定判断混在一起。
- Agent 改一个共享方法时，很容易不知道外部有多少地方依赖了它的隐含语义。
- 做完一次任务后，如果经验不回写，下一次 Agent 还是会重踩坑。

AIPD2 的核心目标，是让项目不再只拥有代码源码，还拥有一套 AI 可用的认知源码。

## 底层判断：从横向分层到纵向黑箱

传统开发常常优先横向分层：

```text
controller -> service -> repository -> model -> util
```

或在前端里：

```text
page -> component -> store -> api -> helper -> constants
```

这种结构对人很友好。它能控制文件长度、降低重复、帮助人把复杂实现压缩成一个个可记忆的抽象块。

但 Agent 做局部任务时，更需要的是纵向黑箱：

```text
一个页面 / 一个接口 / 一个模块 / 一个 step
从输入到输出尽量自足
```

一个接口可以是一个文件，也可以是一个文件夹里的三五个文件。重点不是物理文件数量，而是这个黑箱内部的上下文是否完整，外部是否会随意引用它的内部实现。

AI 友好的代码组织，不是追求“没有依赖”，而是追求“理解无跳转”。

当 Agent 修改一个黑箱时，它应该尽量一次读到：

- 源头输入。
- 参数和状态。
- 业务判断。
- 中间处理。
- 用户可见输出。
- 验收口径。

文件变长、局部重复、目录内文件变多，不再自动等于坏味道。真正危险的是：一个小改动必须跨很多横向依赖理解，而且不知道会不会影响外部调用方。

## 新的优先级：先解耦，后抽取

人类写代码时，代码量膨胀和重复很快会成为大问题，因为人写得慢、读得慢、批量改也慢。

Agent 写代码以后，成本结构变了。多生成一点代码、多消耗一点 token，通常是可以后处理的成本。真正昂贵的是牵连不确定性。

所以 AI 时代的开发优先级应该更接近：

1. 先降低牵连面。
2. 再控制代码量。
3. 最后再抽取复用。

也就是：

**Decouple first, DRY later.**

前期允许页面、接口、组件或 step 在自己的黑箱内各写各的。等重复逻辑经过实践证明稳定，再判断是否抽成共享能力、基础设施或二次打包产物。

重复三次不自动证明应该抽象。调用次数多也不自动证明 helper 合理。抽象的判断标准不是减少字符，而是减少理解成本和外部牵连。

## AIPD2 的做法

AIPD2 用两套结构支撑这个目标。

第一套是项目认知结构，回答“长期知识放哪里”：

```text
L1 Intent   -> 项目方向、目标和长期取舍
L2 Research -> 外部世界、用户、场景、竞品和调研
L3 Core     -> 项目成立依赖的核心模型和底层认知
L4 Product  -> 产品功能、功能边界和用户可见行为
L5 Dev      -> 工程规则、实现约定和跨模块协作
L6 Code     -> 真实代码，不放进 _adoc/
```

第二套是执行能力结构，回答“Agent 做事时怎么用这些知识”：

```text
map 检索 -> case 创建 -> step 执行 -> 分身回流 -> weave 回写
```

两套结构合在一起，AIPD2 才成立：

- 没有 L1-L5，Agent 只能读代码，读不到原因。
- 没有 map，Agent 知道有文档，也不一定找得到入口。
- 没有 case，长任务会被聊天上下文拖散。
- 没有 step，任务难以拆成可恢复、可验收的黑箱。
- 没有分身 Agent，主线会被搜索、验证、日志和 diff 吞掉。
- 没有 weave，经验不会变成下一次可继承的项目记忆。

## 核心能力

### `_adoc/`：项目认知事实源

`_adoc/` 不是普通说明文档。它是给 Agent 读取的项目认知层。

它记录代码里不容易表达的内容：

- 为什么做这个方向。
- 哪些场景和用户已经被讨论过。
- 哪些术语、模型和边界是稳定的。
- 哪些功能不能随便改。
- 哪些工程规则需要跨模块遵守。
- 哪些经验应该被下一次 Agent 继承。

### `map`：第一跳检索

Agent 不能只靠目录树猜上下文。

`_adoc/map.md` 是项目记忆地图。用户说一句自然语言，Agent 先通过 map 把任务路由到相关的 L3 / L4 / L5 / 局部 README / 代码入口。

目标不是把所有内容塞进一个大提示词，而是让第一跳稳定：

```text
用户意图 -> map 命中入口 -> 读取必要认知 -> 进入代码
```

### `Case / Step`：把任务变成可恢复黑箱

AI 协作里，很多任务不是难在执行，而是难在边界。

Case 记录一次事项的目标、上下文索引、边界、steps 和验收口径。Step 是 Case 内可派发、可恢复、可验收的执行单元。

当聊天上下文被压缩或中断时，恢复不靠猜：

```text
AGENTS.md -> _adoc/index.md -> _adoc/map.md -> case/index.md -> 当前 case -> 当前 step
```

### 分身 Agent：消化过程成本

主 Agent 不应该吞下所有过程成本。

读长日志、扫跨文件 diff、查接口、跑测试、验证浏览器表现，这些工作很重要，但会污染主线判断。

AIPD2 把分身 Agent 理解成从主 Agent 当前认知 fork 出来的克隆体。它进入局部探索分支，完成后只回流压缩结果：

- 结论。
- 依据。
- 风险。
- 建议。
- 改动文件。
- 验证结果。

### Weave：把稳定经验写回项目

如果一个经验只存在于当前聊天里，它就没有真正成为项目能力。

Weave 负责把讨论、step 结果、case 归档、代码 diff、错误日志或外部资料中的稳定信息，回写到合适的位置：`_adoc/`、局部 README、map 或 case 记录。

它不存档所有聊天，只沉淀未来任务可能复用的判断。

## 快速开始

AIPD2 这个仓库本身不是一个“打开即用”的应用。

它是 **AIPD2 Skill 的打包源码项目**：源码在本仓库里，最终使用时需要由本地 Agent 把它打包成 Skill，并放到本机对应 Agent 的 Skill 目录下。

这件事不应该再要求用户手工记命令。用户只需要用 Codex、Claude Code 或其他支持 Skill 的 Agent 打开本项目，然后把目标说清楚：

```text
请阅读本仓库的项目说明和 AIPD 认知，把 AIPD2 打包并安装到我本机可用的 Skill 目录。
```

具体怎么 build、怎么 install、放到哪个目录，由 Agent 根据仓库里的执行入口自己判断和执行。当前相关入口包括：

- `AGENTS.md`：Agent 进入项目后的默认协作规则。
- `_adoc/index.md`：AIPD2 仓库自身的项目认知入口。
- `_adoc/map.md`：把“构建 / 安装 / Codex 适配 / Skill”这类任务路由到具体上下文。
- `scripts/`：实际构建、开发安装、用户级安装和项目级安装脚本。

安装完成后，在你的目标项目里让 Agent 使用：

```text
/aipd2
```

如果目标项目还没有 AIPD 结构，`/aipd2` 会引导初始化 `_adoc/`、`AGENTS.md` 和基础索引。

如果目标项目已经初始化，`/aipd2` 会读取项目认知入口，判断当前状态，并把你导向合适的下一步：创建 case、执行 case、更新 AIPD 架构、编织经验，或继续普通开发任务。

## 包含的 Skill

| skill | 命令 | 作用 |
|---|---|---|
| `aipd2` | `/aipd2` | 总入口：识别项目状态，加载轻量认知，引导下一步 |
| `aipd2-case-create` | `/aipd2-case-create` | 创建 case，整理目标、边界、上下文索引和 steps |
| `aipd2-case-run` | `/aipd2-case-run` | 执行 case，读取上下文并派发分身 Agent |
| `aipd2-weave` | `/aipd2-weave` | 把稳定经验回写到 ADOC、局部 README、map 或 case |
| `aipd2-learn` | `/aipd2-learn` | 采集会话定位信息，辅助 AIPD2 框架自迭代 |
| `aipd2-update` | `/aipd2-update` | 更新已初始化项目中的 AIPD 架构、AGENTS 和 map |
| `aipd2-case-archive` | `/aipd2-case-archive` | 归档 case，整理 Weave Candidate，合并分支 |
| `aipd2-git-push` | `/aipd2-git-push` | 检查当前分支和提交状态，并推送远端 |
| `aipd-mermaid` | `/aipd-mermaid` | 创建、修改、评审或按需渲染 Mermaid 架构图 |

更完整的 skill 职责说明见 [docs/modules/skills-overview.md](docs/modules/skills-overview.md)。

## 给 Agent 的执行入口

本仓库保留了构建和安装脚本，但 README 不把它们当成给用户背诵的操作手册。

如果你要安装、调试或更新 AIPD2，建议直接让 Agent 打开本项目，并说明目标：

```text
请把这个 AIPD2 Skill 源码项目构建并安装到本机 Codex。
```

或：

```text
请修改 AIPD2 源码后，按本项目规则重新构建，并告诉我是否需要重新安装。
```

Agent 应先读取 `AGENTS.md`、`_adoc/index.md` 和 `_adoc/map.md`，再根据任务进入 `scripts/`、`src/skills/`、`src/platforms/` 等代码入口。人只需要关注目标、验收结果和风险，不需要记住具体打包命令。

开发者需要查看脚本和安装细节时，读 [docs/modules/build-and-install.md](docs/modules/build-and-install.md)。

## 项目结构

```text
AIPD-2/
├── docs/             # 面向人的学习文档
├── src/
│   ├── core/          # AIPD 核心认知、模板和通用规则
│   ├── platforms/     # Codex / Claude Code 等平台适配
│   └── skills/        # 各个 skill 的源码
├── scripts/           # 构建、开发安装、用户安装和项目安装脚本
├── dist/              # 构建产物
├── _adoc/             # AIPD2 仓库自身的项目认知
└── v1/                # v1 历史归档
```

## 适合什么时候用

AIPD2 适合那些已经开始让 AI 深度参与开发，但发现“上下文管理”比“生成代码”更难的项目。

如果你的项目只有几个文件，或者只是一次性原型，AIPD2 可能太重。

如果你的项目会持续迭代，有业务语言、历史取舍、跨模块规则、多 Agent 执行、长期任务恢复和经验沉淀需求，AIPD2 会更有价值。

一句话总结：

**AI Agent 负责执行，AIPD2 负责让项目记住自己。**
