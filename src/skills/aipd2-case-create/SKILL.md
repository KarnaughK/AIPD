# AIPD2 Case Create

## 职责边界

**只做**：与用户讨论事项 → 按需扫描上下文 → 将上下文索引、边界和必要步骤写入 case.md → 按用户意图创建最少 steps

**不做**：不执行任何 step（执行用 `/aipd2-case-run`），不写业务代码

### Case Create 生命周期

用户通过 `aipd2-case-create` 启动任务后，当前对话默认处在 case 创建、方案讨论、上下文整理、文档补充和 step 拆解阶段。只要用户没有明确结束这个阶段，就不要进入执行。

结束信号只认明确表达：

- 用户说“设计结束”“case create 到这里”“可以开始干活”“开始执行”等。
- 用户显式调用 `/aipd2-case-run` 或 `aipd2-case-run`。

如果用户说“开始干活”但没有调用 `aipd2-case-run`，提醒用户切换到 `aipd2-case-run` 执行，不要在 `case-create` 里直接执行 step 或写业务代码。

这个边界不依赖任务复杂度判断。不要先判断“前端是否复杂”再决定能不能执行；只要仍在 `case-create` 生命周期内，就只做设计、建模、上下文沉淀和 step 准备。

## 执行流程

### 第一步：问用户要做什么

触发后不要先扫描，直接问用户要完成什么。

先判断用户这次是要：

- **完整 case 拆解**：用户明确要制定执行计划、拆步骤、准备连续开发。
- **框架 / 上下文搭建**：用户说“先看一眼”“先把上下文给我”“外围框架处理好”“先建 case”“不要多做”“后面逐个调整”等。

如果是框架 / 上下文搭建，默认只创建 case 框架和一个上下文整理 step。不要预先创建未来业务调整 step，除非用户明确要求拆出后续步骤。

底层逻辑：渐进式推进，不要企图一步到位。用户说清一个点，就推进一个点；有需要执行、恢复或验收时再固化成 step；没需要时继续聊，把可能有价值的后续事项放入候选区。

如果用户在 `case-create` 中继续补充业务流程、接口规则、组件结构或异常分支，这些都仍属于当前 case 的设计输入。把它们写入 case doc、上下文索引、候选区或已确认 step，不要把讨论本身包装成执行 step。

### 第二步：上下文检索

创建 case 前，先把用户描述转成上下文检索包。这个动作不是执行 step，而是为了避免 case 建在错误上下文上。

根据用户描述，精准读取相关文件：
1. 主 Agent 读 `_adoc/index.md` 了解项目模块分布（一次，轻量）。
2. 如果存在 `_adoc/context-map.md`，读取它，判断用户意图对应的 L3 / L4 / L5 / 局部 README / case。
3. 优先寻找本次事项相关的核心概念索引、产品功能文档、工程实现规则、页面 README、设计/原型资料和代码入口。
4. 需要探索、验证、批量读取或整理 case 草案时，按 Agent Entry 中的分身 Agent 逻辑 fork 主 Agent 克隆体，主 Agent 只审阅结果回流。

上下文检索包至少包含：

- 层级判断：本次涉及 L3 / L4 / L5 / 局部 README / case 中哪些层。
- 必读文档：执行前必须读的 `_adoc` 文档、页面 README、设计/原型资料。
- 代码入口：页面、弹窗、组件、接口、权限工具、路由或配置文件等入口。
- 兜底搜索：缺少地图或命中不清楚时可用的 `rg` 关键词。
- 风险边界：概念歧义、产品边界、跨端约定、权限码/接口字段一致性等。

**原则**：主 Agent 负责和用户讨论目标、边界、取舍和验收口径，不直接消费高噪声上下文。分身 Agent 是主 Agent 的克隆体，不是低上下文工人；它带着同样的上下文进入局部探索分支，用于计划、探索、压缩信息和整理草案，然后只回流结论、依据、风险、建议和 case 草案摘要。

`case-create` 阶段的分身不直接执行业务 step。代码探索、接口核对、组件关系梳理和批量读取可以交给分身，但分身回流的结果仍由主 Agent 与用户确认后写入 case / doc / step。

如果分身用于上下文检索，派发目标必须明确为“只返回上下文包”。不要让分身直接创建 step、写代码或扩展成完整执行计划。

### 第三步：与用户确认方案

把分身 Agent 返回的关键认知和 case 草案汇总给用户，讨论执行方案，确认上下文索引、边界和步骤拆分。

如果用户已经明确说“不要多做”或只要求搭框架，不要把可能的后续工作自动拆成 step 文件；最多在 `case.md` 的“后续候选事项 / 待拆分”中用文字列出，等待用户确认后再创建 step。

### 第四步：生成 case 文件

```bash
mkdir -p _adoc/case/c{X.Y}-{名称}/steps
mkdir -p _adoc/case/c{X.Y}-{名称}/doc
```

**case.md 必须包含**：
- 目标：这个 case 要完成什么
- 上下文索引：执行时优先读哪些 `_adoc` 文档、页面 README、代码入口、设计/原型/调研资料；必须包含层级判断、必读文档、代码入口、兜底搜索和风险边界
- 本次边界：明确做什么和不做什么
- steps 列表：只列已确认要执行的 step；框架搭建型 case 可以只有一个上下文整理 step，甚至暂时没有可执行 step
- 后续候选事项：未确认的调整点只记录为候选，不创建 step 文件
- 验收标准和经验沉淀位置
- 自迭代观察锚点：后续审计 transcript / session 时，检查 Agent 是否按本 case 的上下文索引和 AIPD 检索流程工作

已确认的步骤文件写入 `steps/`，格式参考 `@references/case/templates/step.md`。

### Mermaid 按需预览规则

默认不要在创建或修改 `.mmd` 文件后自动预览。Mermaid 渲染会消耗额外 token 和工具调用成本，除非用户明确要看图，否则只保留源文件路径即可。

当用户说“我看一下那个 mmd 文件”“预览一下流程图”“把架构图渲染出来”“看看这张图对不对”等明确查看意图时，才执行预览：

1. 使用 `mermaid-preview` skill 渲染被创建/修改的 Mermaid 文件。
2. 如果当前上下文没有加载 `mermaid-preview`，优先调用已安装脚本：

```bash
python3 ~/.codex/skills/mermaid-preview/scripts/render_mermaid.py path/to/file.mmd
```

3. 如果提示缺少 `mmdc`，用 `--allow-npx` 重跑。
4. 读取脚本输出的 `output:` 路径，并用本地图片查看工具把 PNG 加入聊天上下文。
5. 最终回复必须包含源 `.mmd` 路径、生成 PNG 路径和渲染结果。

这个动作属于 case-create 的设计查看，不算执行业务 step。用户没有要求看图时，不要主动渲染。

### 架构图落位规则

如果 case-create 产出了会指导后续代码实现的 `.mmd` 架构图 / 组件图，不要让它长期只停留在 `_adoc/case/{case}/doc/`。创建 step 拆分方案时，必须判断这些图是否需要落到目标代码目录就近位置。

默认规则：

- 如果目标代码目录还不存在，或本次实现会新建弹框 / 页面 / 模块目录，拆分草案的第一个实现 step 应先创建目标目录，并把实现需要的架构图落到该目录。
- 如果目标代码目录已经存在，第一步不强行建目录，但要把架构图落到现有就近 README / `_map.md` / `*-map.mmd` 位置，或在 step 中明确为什么不需要落位。
- 长期指导实现和维护的图，优先**移动**到代码就近目录，case.md 只保留链接；避免 case 和代码目录维护两份会漂移的图。
- 只作为设计过程记录、方案备选或已废弃判断的图，可以留在 case `doc/`，但必须在 case 里标明它不是实现侧事实源。
- 不要为了落图创建空泛文档层。图应放到最接近后续修改入口的位置，例如目标页面目录、弹框目录或模块根目录。

Step 拆分草案中应体现这个落位动作，例如：

```md
- cX.Y.1：创建目标弹框目录，并将 case 中的组件图移动到弹框目录就近位置，case.md 改为链接该图。
- cX.Y.2：基于已落位的组件图实现弹框骨架和数据入口。
```

### 自迭代观察锚点

case-create 生成或更新 case 时，应在 case 中留下 3-6 条可审计的观察锚点。它们用于后续 `aipd2-learn` 判断 AIPD 机制有没有按设计生效，而不是只等用户多次纠正后才发现问题。

观察锚点优先围绕：

- Agent 是否读取 `_adoc/context-map.md` 或说明缺失，并用 `rg` / README 兜底。
- Agent 是否读取本 case 上下文索引里的 L3 / L4 / L5 / 局部 README。
- 涉及核心概念时，Agent 是否先对齐 L3 术语和关系，而不是直接猜业务含义。
- 涉及跨模块工程规则时，Agent 是否读取 L5 实现规则，而不是只看目标页面代码。
- 涉及页面、弹窗、组件内部细节时，Agent 是否读取就近 README。
- 如果执行偏离 SOP，偏离点是提示词没看清、地图没命中、文档缺失，还是 AIPD 流程设计不够硬。

### Step 创建边界

创建 step 前先做意图校准。用户说“开始设计 step / 开始拆 step / 写一下 step”时，表示进入**编写 step 这个大模块的阶段**，不表示可以立刻落详细 step 文件。

编写 step 的第一步是先输出轻量拆分方案，和用户确认“第几步要干什么”：

```md
【Step 拆分草案】
- cX.Y.1：{一句话说明这一步要干什么}
- cX.Y.2：{一句话说明这一步要干什么}
- cX.Y.3：{一句话说明这一步要干什么}

待确认：
- 这个拆分粒度是否合适？
- 有没有哪一步太粗、太细，或顺序不对？
```

只有用户确认拆分方案基本没问题后，才写详细 step 文件，包括上下文文档、任务清单、验收标准和不做边界。用户纠正“太少 / 太粗 / 顺序不对 / 好好查一下”时，先重新读取必要上下文并重出拆分草案，不要直接补写文件。

意图校准参考：

| 用户表达 | 默认处理 |
|---|---|
| “先把上下文 / 框架处理好” | 创建 case 框架，最多创建 `cX.Y.1-context-*` |
| “千万不要多做 / 先别改业务” | 不创建未来业务调整 step |
| “后面逐个调整 / 后续再说” | 在“后续候选事项”记录，不创建 step |
| “开始设计 step / 开始拆 step / 写一下 step” | 进入 step 编写阶段，先展示轻量拆分草案，确认后再写详细 step |
| “帮我拆一下步骤 / 制定执行计划” | 可以拆多个 step，但先展示轻量拆分草案让用户确认 |
| 用户明确列出多个独立任务并要求建 case | 按任务拆 step |

不要因为模板里有多个 step 示例，就机械创建 `cX.Y.2`、`cX.Y.3`。每个 step 都会增加用户审核成本，未确认的 step 是噪声。

不要为了显得完整而一次性规划所有步骤。case-create 的交付重点是“当前最小可用上下文 + 已确认的下一步”，不是完整路线图。

step 只能表示“已确认、可派发执行”的工作。正在讨论的问题、设计阶段缺口、需要继续确认的接口或组件关系，写入 case doc 或候选区，不创建“设计确认 step”“当前讨论 step”这类把 `case-create` 自身工作包装成 step 的文件。

### Step 推荐 Agent 规则

创建 step 时尽量声明 `推荐 Agent`：

- 调研、只读定位、资料整理：`explorer`
- 普通开发、修复、脚本和文档修改：`worker`
- Vue 页面、Vue 组件、HTML/CSS、组件通信、前端状态组织：`aipd_vue_architect`
- 不确定时留空，并在 step 目标和上下文文档里写清楚判断依据

推荐 Agent 是执行角色建议，不替代 step 的上下文文档。`case-run` 阶段会优先让带角色 Agent 基于 step / case / 上下文文档执行；只有当 step 强依赖主 Agent 当前尚未沉淀的聊天判断时，才需要 fork 上下文分身。

### 第五步：告知下一步

如果已经有明确可执行 step，告知用户执行 `/aipd2-case-run` 开始执行。

如果只是搭好 case 框架和上下文索引，告知用户：当前 case 已可作为后续讨论入口，等用户确认具体调整点后再补 step。
