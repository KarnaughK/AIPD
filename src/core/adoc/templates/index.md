## 上下文检索

- 项目级记忆地图：`_adoc/map.md`
- 旧项目兼容入口：`_adoc/context-map.md`
- 进入具体任务前，先判断任务涉及 L3 核心概念、L4 产品功能、L5 工程实现规则、局部 README、L6 代码入口、case 中的哪些上下文。
- 如果 `map.md` 缺失或未覆盖本次任务，按相关核心词、功能线名、页面名、接口名、权限码、组件名使用 `rg` 兜底搜索，并把新发现的稳定入口回写到合适的 map。
- 如果讨论、step 结果、case 归档、代码 diff、错误日志或外部资料产生了可复用知识，用 `aipd2-weave` 判断回写到 L3 / L4 / L5 / 局部 README / map / case。

## 读取原则

- 任务处在哪个层级，就读取对应层级及必要上下游认知。
- 代码模块、页面、组件的局部认知，优先读取就近 `README.md`。
- `_adoc/L3-core/` 应维护核心概念索引，列出标准名、别名/黑话、关系、常见误解和细节文档入口。
- `_adoc/L5-dev/` 更适合存产品功能到代码实现之间的业务实现逻辑、跨模块工程规则、索引和通用 SOP。
- L6 是具体代码实现，不在 `_adoc/` 内单独建层。
- 页面、弹窗、组件内部的最后一层实现细节，放代码目录就近 `README.md`，不要塞进 L5。
- 如果用户当前指令与 AIPD 认知冲突，先指出冲突和风险，再继续。
