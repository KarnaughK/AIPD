# Case 机制总览

Case 是 AIPD 的事项执行载体：一次需要 AI 参与处理、带上下文索引、可拆 step、可沉淀经验并归档的具体事项。

Case 不只是任务清单。它的首要价值是把“本次事项最强相关的上下文”显式告诉 AI，包括项目认知文档、页面 README、代码入口、设计/原型、调研资料和历史决策。后续执行时，主 Agent 先读取 case 的上下文索引，再判断是否需要派发 step。

## 三级版本号（A.B.C）

| 级别 | 含义 | 谁定 | 示例 |
|------|------|------|------|
| **A** | 大版本 / 里程碑 | 用户决定 | c1.0 |
| **B** | 功能版本（一次迭代） | 主 Agent + 用户 | c0.2-search |
| **C** | 迭代步骤（最小执行单元） | 主 Agent 拆分 | c0.2.1-basic-table |

- B 级 = 一个 Case，对应一个目录
- C 级 = Case 内的步骤，由子 Agent 独立执行

## Case 类型

| 类型 | 用途 | 子 Agent 角色 |
|------|------|--------------|
| **dev**（默认） | 开发功能 | 写代码、调试、自检 |
| **research** | 调研分析 | 搜索、读文档、输出结论 |
| **review** | 代码/方案审查 | 读代码、分析问题、输出报告 |

## 目录结构

```
_adoc/case/
├── c0.1-功能名/
│   ├── case.md
│   ├── steps/
│   │   ├── c0.1.1-步骤名.md
│   │   └── c0.1.2-步骤名.md
│   └── doc/
└── archive/
    └── c0.1-功能名/
```

## 完整生命周期

1. 创建 Case（在 main 分支）
2. 提交 Case 到 main（锁定版本号）
3. 创建开发分支
4. 读取 case 上下文索引，加载最相关上下文
5. 有未完成 Step → 派发给子 Agent 执行
6. 没有 Step → 基于 case 上下文继续讨论或补充 Step
7. 所有 Step 完成 → 用户验收
8. 验收通过 → 归档
9. 合并分支到 main

## 关键原则

1. **主 Agent 永不执行开发任务**：所有代码工作由子 Agent 完成
2. **Case 先提供上下文索引**：主 Agent 按 case.md 读取最强相关上下文，不全量扫描项目
3. **子 Agent 只看步骤文件和指定上下文**：不需要知道 Case 全貌
4. **主 Agent 做最少的审查**：看子 Agent 返回的通知即可
5. **失败时人在环路**：子 Agent 失败后与用户商量，不自动无限重试
6. **只做 case 中定义的任务**：不要"顺手"做额外的事
7. **遵循 dev 约束**：约束不合理先讨论修改
8. **Case 先提交 main，再创建分支**：保证编号全局唯一
9. **验收环节不可跳过**：始终让用户确认结果符合预期
