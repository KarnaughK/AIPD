# 第四层：Product — 设计产品功能模块

**目标**: 基于 L1/L2/L3，设计具体的产品功能模块，产出 `_adoc/L4-product/` 文档

**触发条件**: `intent.md` 完整，但 `_adoc/L4-product/` 为空或文件很少

**核心理念**: L4-product 描述产品功能模块（做什么），不是页面、接口、代码目录或技术项目。一个产品功能模块可以被多个研发项目实现，一个研发项目也可能承载多个产品功能模块。

## AIPD2 与项目 _adoc 的关系

AIPD2 仓库里的 L4 文档是方法论，用来指导 AI 在具体项目里创建和维护 `_adoc/L4-product/`。

具体项目里的 `_adoc/L4-product/` 才是实例文档，记录这个项目真实存在的产品功能模块、功能边界、用户旅程、验收标准和业务 SOP。

因此，AIPD2 的 L4 不应该沉淀某个业务项目的具体模块内容，而应该沉淀：

- 如何从 L1 方向、L2 场景、L3 核心模型推导产品模块
- 如何判断一个功能是否值得成为独立模块
- 如何写清模块边界、用户旅程、验收标准和 SOP
- 如何避免把技术实现误写进产品层

## L4 负责什么

L4 回答的是“用户可以完成哪些产品能力，以及这些能力的边界在哪里”。

它应该包含：

- 功能模块的核心价值
- 用户能完成的关键旅程
- 模块内的业务规则和状态流转
- 模块和模块之间的依赖关系
- 可验收的功能标准
- 必要的业务 SOP

它不负责：

- 技术选型、目录结构、组件拆分、接口路径
- 某个页面的 provider/controller/data flow
- 数据库表结构和部署方式
- 当前迭代的任务拆分

这些内容分别放到 `_adoc/L5-dev/`、页面 README、代码或 `_adoc/case/`。

## 与 L1/L2/L3 的关系

创建 L4 时，不是从页面或接口倒推模块，而是从前三层往下推：

- **L1 Intent**：决定哪些功能值得做，哪些功能即使能做也不该做
- **L2 Scenario**：决定功能要服务哪些真实用户场景，避免自嗨式功能堆叠
- **L3 Core**：决定核心实体、领域边界和状态流转，避免产品模块破坏底层模型

一个合格的 L4 模块，应该能说清楚它继承了哪个方向、解决了哪个场景、围绕哪个核心模型工作。

## 命名规范

用功能描述命名，不用产品项目名：

- ✅ `keyword-discovery`、`keyword-validation`、`rule-check`
- ❌ `keyword-research-backend`、`seo-dashboard`（技术项目名，放 L5-dev/）

也不要直接用页面名或接口名命名：

- ✅ `contract-editing`、`contract-review`、`account-settings`
- ❌ `constructEdit-page`、`POST-save-contract`、`src/views/my/constructEdit`

页面是实现入口，接口是实现手段，L4 模块是产品能力。

## 执行步骤

### 步骤 1: 复述项目方向

```bash
cat _adoc/L1-intent/intent.md
```

```
【当前项目方向】

项目: xxx
愿景: xxx
核心目标: xxx

---
基于这个方向，我们来设计需要哪些产品功能模块和研发实现模块。
```

### 步骤 2: 引导模块拆分

- "要实现这个愿景，用户需要完成哪些核心任务？"（→ L4 产品功能模块）
- "研发上需要哪些项目/服务来支撑？"（→ L5 研发实现模块）
- "有没有优先级？哪些是 MVP 必须的？"

### 步骤 3: 主动提供建议

基于 intent 分析：
- 推测可能需要的产品功能模块和研发实现模块
- 画出模块之间的依赖关系
- 区分"核心功能"和"增强功能"

### 步骤 4: 收集模块信息

对每个确定的模块，收集：
- 核心能力（一句话描述价值）
- 用户旅程（关键交互流程）
- 验收标准（怎么算做完了）
- 依赖关系（依赖/被依赖）
- 不做什么（明确边界，防止后续 Agent 越界实现）
- 关联的 L3 核心实体或状态流转

### 步骤 5: 判断是否需要 SOP

不是每个模块都要 SOP。只有当模块存在稳定、重复、跨角色或容易误解的业务流程时，才单独写 `sop-xxx.md`。

适合写 SOP：

- 流程有多个角色参与
- 流程有明确先后顺序和状态变化
- 流程经常被 case 修改或引用
- 流程做错会造成业务风险

不适合写 SOP：

- 只是一个简单按钮或表单
- 规则还不稳定，仍处于探索期
- 只是技术操作步骤

### 步骤 6: 创建模块文档

```
_adoc/L4-product/
├── 模块A/
│   ├── index.md           # 模块说明 + Spec
│   └── sop-xxx.md         # 标准流程（按需）
└── 模块B/
    └── index.md

_adoc/L5-dev/
├── index.md               # 研发模块索引 + 环境说明
├── 项目A/
│   └── README.md          # 研发约束 + 技术栈
└── 项目B/
    └── README.md
```

## 关键原则

- 每个模块应该是可独立交付的产品能力
- 模块之间尽量低耦合
- 先设计核心功能，增强功能可以后续补充
- L4 只描述产品能力，不写代码组织和接口细节
- 模块边界不清时，优先回到 L2 场景和 L3 核心模型，而不是从现有页面结构硬拆

### DDD 领域驱动设计指引

在产品模块设计阶段，可以借用 DDD 思路划分边界：

**核心思路**：
- **先识别领域，再定技术**：讨论业务模块时，引导用户思考"这里有几个不同的领域？各自的核心实体是什么？"
- **实体独立性**：核心实体不应从属于某个来源或流程，而是独立存在。例如"关键词"是独立实体，不是"游戏"的附属品
- **领域边界 = 模块边界**：每个领域内聚自己的数据和逻辑，领域之间通过明确的接口（事件/API）通信

**设计时的引导问题**：
1. "这个系统里有几个不同的领域？它们各自关心什么？"
2. "核心实体是什么？它属于哪个领域？"
3. "领域之间怎么通信？A 领域产出的东西怎么传给 B 领域？"
4. "数据库表结构是否反映了领域划分？有没有一张表混了两个领域的职责？"

**常见反模式**：
- ❌ 把核心实体绑死在某个来源上（如 keyword 绑 game_id）
- ❌ 一张表既存原始数据又存分析结果
- ❌ 领域之间直接共享内部数据结构
- ✅ 各领域有自己的表，通过核心实体 ID 关联
- ✅ 一个后端项目可以包含多个领域模块，共享基础设施（数据库、定时任务）

## 模块设计方法论

### 模块化判断标准

- 这个能力可以单独交付吗？
- 这个能力有独立的用户价值吗？

如果两个都是"否"，考虑合并到其他模块。

### 粒度判断

**太大的信号**：
- 模块描述超过 3 页
- 包含多个独立的用户旅程
- "核心能力"需要用"和"连接多个能力
→ 拆分成多个模块

**太小的信号**：
- 不能独立交付
- 必须和另一个模块一起才有意义
- 验收标准只有 1-2 条
→ 合并到相关模块

**合适的粒度**：
- 一个明确的核心能力
- 2-5 个相关的用户旅程
- 4-8 条验收标准
- 1-2 页内容

### 用户旅程写法

描述"用户能做什么"，不是"系统怎么实现"。

**好的写法**：用户可以创建 Prompt 并保存为新版本。

**不好的写法**：调用 POST /api/prompt 接口，数据存入 SQLite。

示例：
```markdown
### Journey: 创建新提示词
1. 用户点击"新建"，输入名称并选择分组
2. 在编辑器中定义 Input Variables
3. 编写 Prompt Body，引用变量
4. 保存，系统自动生成 v1 版本
```

### 验收标准写法

粒度要适中：
- **太粗**：用户能管理 Prompt（无法验证）
- **太细**：点击按钮后 200ms 内响应（过于技术化）
- **合适**：编辑器支持变量高亮与自动补全（可验证、面向用户）

### 依赖关系处理

- 依赖过多 → 考虑合并成一个模块，或重新划分边界
- 跨模块功能 → 确定主要归属，在相关模块的依赖关系中标注

## 常见误区

### 把页面当模块

页面通常是多个产品能力的入口，不一定等于一个 L4 模块。比如“合同编辑页”可能同时承载基础信息、金额、交付任务、附件等能力；如果这些能力有独立业务边界，L4 应按能力拆，而不是按页面拆。

页面级结构、数据流、事件流和修改入口，应该放在页面目录 README 或 `_adoc/L5-dev/` 的研发指引中。

### 把接口当模块

接口是实现手段，不是产品能力。`保存合同` 可以是一个用户旅程中的动作，但 `POST /contract/save` 不应该成为 L4 模块。

### 把技术项目当模块

`vue-admin`、`express-api`、`spring-service` 这类是 L5-dev 的研发实现模块。只有当它们对应稳定的产品能力时，才在 L4 里出现能力描述。

### 只写功能清单，不写边界

功能清单只能告诉 AI “有什么”，不能告诉 AI “不要做什么”。L4 必须写清楚不做范围，否则后续 Agent 很容易把增强需求、技术重构或临时想法误当成当前模块职责。
