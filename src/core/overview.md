# AIPD 框架总览

AIPD 把软件项目从构想到交付过程中产生的经验和决策，按领域规则结构化沉淀为项目认知，让 AI 能基于这些认知持续、准确地参与项目。

## 核心原则

- **经验总结型框架**：边实践边迭代，框架随项目实践持续演进
- **_adoc/ 是服务于 AI 的项目知识库**：只存整理过的文档，不存原始数据和一次性脚本

## 五层认知结构

从抽象到具象组织项目知识，AI 按需加载：

| 层级 | 名称 | 解决的问题 |
|------|------|-----------|
| L1 | Intent（方向） | AI 理解项目目标，不做偏离方向的事 |
| L2 | Scenario（场景） | AI 知道用户痛点，设计有的放矢 |
| L3 | Engine（引擎） | AI 理解核心数据模型，不破坏架构 |
| L4 | Product（产品） | AI 知道功能边界，不越界实现 |
| L5 | Dev（研发） | AI 遵守研发规范，代码风格一致 |

详细指引见各层 `guide.md`。

## 多 Agent 架构

- **主 Agent = 项目负责人**：与用户对话、规划任务、调度子 Agent、审查结果
- **子 Agent = 同源分身**：从主 Agent 当前认知 fork 出来，进入局部未来分支完成开发/调研/归档

**核心设计哲学：主 Agent 上下文是稀缺资源，子 Agent 上下文是可消耗资源。**

AIPD 不把子 Agent 当成低上下文工人，也不靠大量 if-else 规则规定什么时候派工。更准确的模型是：主 Agent 留在当前时间线维护用户沟通、目标边界和最终判断；子 Agent 带着同一份认知去消化探索过程，完成后只把结论、依据、风险、建议、改动文件和验证结果回流给主线。

同样一件事，主 Agent 直接做可能消耗 100K 主线 token；fork 子 Agent 做，子 Agent 可以多消耗，但主 Agent 只接收压缩结果。AIPD 不优先优化总 token 消耗，而是优化主 Agent 的单轮上下文消耗和任务连续性。

详细调用方式见 `agent-guide.md`。不同平台的具体子 Agent 调度方式由构建时的平台覆盖文件提供。

## 执行体系

- **Case**：将功能拆分为可执行步骤，由子 Agent 逐步完成，主 Agent 调度审查
- **OKR**：追踪项目阶段目标与关键结果，对齐 AI 的执行方向
