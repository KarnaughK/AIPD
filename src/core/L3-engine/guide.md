# 第三层：Core — 核心成立模型

> 状态：未经实践验证的初始文档。方向性参考，非 SOP。

## 这一层在做什么

从 L2 的外部世界资料中，提炼项目内部最稳定、最不容易变、最能解释项目如何成立的核心模型。

核心对象、数据模型和核心交互关系是 L3 的常见形态，但不是全部。L3 的本质不是“有哪些表”或“有哪些对象”，而是“这个项目靠哪些模型成立”。

## 什么是核心成立模型

不是所有抽象都能进入 L3。核心成立模型的特征：

- 它是系统中最不容易变的部分
- 去掉它，系统就不成立
- 它会长期约束多个 L4 功能，而不是只服务一个功能点
- 它已经从外部观察变成了本项目自己的内部模型

比如：
- 电商系统的引擎：商品 + 订单 + 支付（不是推荐算法、不是营销活动）
- 提示词管理工具的引擎：提示词 + 测试用例 + 版本（不是编辑器 UI、不是团队协作）
- 内容平台的引擎：内容 + 创作者 + 分发（不是评论系统、不是广告）
- 猜词游戏网站的产品模型：谜底词 + 猜测 + 反馈规则 + 玩法模式
- 猜词游戏网站的商业模型：SEO 流量 + 回访机制 + 广告/订阅变现
- AIPD2 的框架模型：上下文解耦 + map 检索 + Case/Step + Weave 回写

## 灵感与思路

**从外部观察里提炼内部模型**

L2 可以记录大量外部事实：竞品、用户、痛点、玩法、流量、变现、替代方案。L3 要做的是从这些材料中提炼“我们自己的成立方式”。

例如：

- "竞品 A 有每日挑战" 是 L2。
- "每日挑战用于制造回访、分享和可传播战绩" 可以成为 L3 的留存/增长模型。
- "实现一个每日挑战页面" 是 L4。

**从场景需求里删出模型**

不是往上加东西，而是往下删。把场景需求里的所有元素列出来，然后问：
- 如果去掉这个，系统还能运转吗？
- 这个是核心成立模型，还是某个功能点的附属？
- 这个模型会约束多个功能，还是只影响一个页面？

删到不能再删，剩下的就是 L3。

**数据模型是常见骨架，不是唯一骨架**

很多项目的核心模型需要落到具体数据关系上。设计 L3 时可以带着技术思维：
- 核心对象有哪些？
- 对象之间是什么关系？（一对多、多对多、包含、引用）
- 哪些字段是本质的，哪些是派生的？

比如"提示词管理"的引擎：
- 一个提示词对应多个测试用例
- 测试用例可以动态积累、手动添加、从数据中提取
- 提示词有版本，版本之间可以 diff

这几句话就定义了引擎的骨架。后面的产品功能模块和研发实现约束都从这里长出来。

但对游戏、内容、SEO、社区或商业网站，L3 也可能包含不完全等同于数据对象的模型：

- 流量如何进入。
- 用户为什么回来。
- 内容如何扩展。
- 价值如何变现。
- 多语言、多地区、多渠道如何成为长期结构。

**引擎设计的常见陷阱**

- 把 UI 交互当成引擎（"拖拽排序"不是引擎，"有序列表"才是）
- 把技术实现当成引擎（"用 Redis 缓存"不是引擎，"需要快速读取的热数据"才是）
- 引擎太大（如果引擎包含 10 个以上核心对象，可能还没提炼到位）
- 引擎太小（如果引擎只有 1 个对象，可能遗漏了关键关系）
- 把竞品事实直接当成核心模型（"竞品有排行榜"不是 L3，"排行榜承担竞争留存"才可能是 L3）
- 把单个功能当成核心模型（"做订阅页"不是 L3，"订阅/广告如何构成变现模型"才可能是 L3）

**从已有系统反推引擎**

如果是改造已有项目，可以反向操作：
- 看数据库里哪些表的数据量最大、关联最多
- 看代码里哪些模块被最多其他模块依赖
- 看用户操作日志里哪些对象被操作最频繁

这些往往就是引擎的线索。

## 产出形式

- 核心对象清单（3-7 个，附一句话说明）
- 对象关系图（可以是文字描述，不一定要画图）
- 关键约束（"一个 X 必须属于一个 Y"、"Z 创建后不可删除"）
- 核心成立模型说明（产品模型、增长模型、内容模型、流量模型、留存模型、商业模型等，按项目需要取舍）
- L2 外部观察如何被提炼成 L3 内部模型的判断依据

L3 文档应该短而精。如果越写越像竞品资料库，说明内容还停留在 L2；如果越写越像功能清单，说明内容已经滑到 L4。
