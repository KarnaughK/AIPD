# 第三层：核心引擎 — 基础指引

> 状态：未经实践验证的初始文档。方向性参考，非 SOP。

## 这一层在做什么

从场景需求中提炼出最不变的抽象——核心对象、数据模型、核心交互关系。

这是项目的"发动机"。方向可以微调，功能可以增减，但引擎一旦确定，后续一切围绕它生长。

## 什么是"核心引擎"

不是所有抽象都是引擎。引擎的特征：

- 它是系统中最不容易变的部分
- 去掉它，系统就不成立
- 其他功能都是围绕它展开的

比如：
- 电商系统的引擎：商品 + 订单 + 支付（不是推荐算法、不是营销活动）
- 提示词管理工具的引擎：提示词 + 测试用例 + 版本（不是编辑器 UI、不是团队协作）
- 内容平台的引擎：内容 + 创作者 + 分发（不是评论系统、不是广告）

## 灵感与思路

**从场景需求里"删"出引擎**

不是往上加东西，而是往下删。把场景需求里的所有元素列出来，然后问：
- 如果去掉这个，系统还能运转吗？
- 这个是核心对象，还是核心对象的附属？

删到不能再删，剩下的就是引擎。

**数据模型是引擎的骨架**

引擎不是一个概念，它需要落到具体的数据模型上。设计引擎时就要带着技术思维：
- 核心对象有哪些？
- 对象之间是什么关系？（一对多、多对多、包含、引用）
- 哪些字段是本质的，哪些是派生的？

比如"提示词管理"的引擎：
- 一个提示词对应多个测试用例
- 测试用例可以动态积累、手动添加、从数据中提取
- 提示词有版本，版本之间可以 diff

这几句话就定义了引擎的骨架。后面的产品架构和技术架构都从这里长出来。

**引擎设计的常见陷阱**

- 把 UI 交互当成引擎（"拖拽排序"不是引擎，"有序列表"才是）
- 把技术实现当成引擎（"用 Redis 缓存"不是引擎，"需要快速读取的热数据"才是）
- 引擎太大（如果引擎包含 10 个以上核心对象，可能还没提炼到位）
- 引擎太小（如果引擎只有 1 个对象，可能遗漏了关键关系）

**从已有系统反推引擎**

如果是改造已有项目，可以反向操作：
- 看数据库里哪些表的数据量最大、关联最多
- 看代码里哪些模块被最多其他模块依赖
- 看用户操作日志里哪些对象被操作最频繁

这些往往就是引擎的线索。

## 产出形式

- 核心对象清单（3-7 个，附一句话说明）
- 对象关系图（可以是文字描述，不一定要画图）
- 关键约束（"一个 X 必须属于一个 Y"、"Z 创建后不可删除"）

引擎文档应该短而精。如果超过一页，可能还没提炼到位。
