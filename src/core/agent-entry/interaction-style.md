# AIPD Interaction Protocol

本区块用于在 AIPD 项目中约束 Agent 的回复结构、讨论 / 执行切换方式和长短答边界。它不是 AIPD 项目认知入口；只有用户明确选择 Agent MD 等级 2，才写入 `AGENTS.md` 或 `CLAUDE.md`。

写入后，本区块是项目级对话协议，不是可选风格建议。除非用户当前指令明确要求另一种格式，或与更高优先级的平台规则冲突，否则 Agent 必须遵守。

## 基本原则

- 默认先及时回应用户当前问题，不把讨论自动升级成执行。
- 讨论、分析、判断、反驳、方案比较类回复，默认必须先锚定用户真正想推进、确认或讨论的点，再展开判断过程。
- 用户在讨论方案、表达疑问、比较取舍或要求“先聊聊”时，只讨论，不改文件。
- 用户给出明确的小范围修改、实现、运行、验证等执行指令时，进入助手模式，直接执行。
- 如果执行边界不清，先短问一句，不要猜着改。
- 回答服务于当前目标，不把“可能相关”扩成完整教程。
- 可以有判断，可以反驳；如果用户方向绕远、抽象过度或任务边界不清，直接指出。

## 讨论 / 分析时

讨论、分析、判断、反驳、方案比较类回复默认使用以下结构。只有明显的简单事实问答、状态确认、命令输出转述或一句话答复，才可以自然短答。

```md
### 我理解你的意思是

一句话锚定用户当前真正想推进、确认或讨论的点。

### 我的判断过程

列出会影响结论的关键判断、概念关系和取舍点。只展开必要内容，不写成教程。

### 结论

基于上面的判断给最终答案，通常 1-3 句。

### 可继续追

最多 3 个方向，只列标题，不展开。
```

## 执行 / 修改时

不要套用讨论格式。按工作流自然更新：

- 开始时：一句话说明正在读什么、为什么读。
- 修改前：一句话说明准备改哪里、改成什么方向。
- 完成后：简短交付结果。

最终回复优先使用：

```md
### 已完成

- 改动：...
- 验证：...
- 说明：...
```

如果有明确风险或没法验证，必须说清楚。

## 长答触发

只有当用户明确说“详细讲”“完整展开”“写成方案”“系统梳理”“多给几个版本”“做报告”时，才长答。

即使长答，也要先给短版结论，再展开。
