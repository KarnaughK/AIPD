# AIPD Project Entry

本项目使用 AIPD 维护项目认知。

AIPD 是面向 AI 协作的软件项目认知框架。它把项目从方向、调研、核心模型、产品结构到研发实现的关键判断沉淀下来，让 AI 不只看到代码结果，也能理解为什么这样设计。

## 认知层级

- **L1 Intent**：项目方向、目标和长期取舍。
- **L2 Research**：用户、场景、需求、竞品和调研结论。
- **L3 Core**：核心对象、领域语言、核心流程、数据模型和系统成立方式。
- **L4 Product**：产品模块、功能边界、业务规则和交互行为。
- **L5 Dev**：产品功能到代码实现之间的工程实现层，存放跨模块、跨端、跨页面、长期稳定的实现逻辑、工程约束、调试经验和维护规则。

任务处在哪个层级，就读取对应层级及必要上下游认知。

## 上下文检索

AIPD 的工作不是直接“读某一层”，而是先检索任务需要的上下文。

`_adoc/map.md` 是本项目给 AI 准备的最快索引，不是额外任务。用户用自然语言描述需求时，通常不会同时给出准确代码路径；map 的作用就是用最少 token 把“用户说法 / 业务词 / 功能线 / 工程词”路由到相关 L3 / L4 / L5 / 局部 README / 代码入口，减少盲搜和误改。

因此，读取 map 是为了更快、更准地完成用户当前命令，而不是覆盖用户命令。除非用户明确说“不要读取 AIPD 文档 / 不要查 map / 直接按我给的文件改”，否则涉及业务功能、页面、弹窗、权限、接口、核心概念或跨模块规则时，都应先用 map 做第一跳定位。

进入任务后按以下顺序取上下文：

1. 先读 `_adoc/index.md`。
2. 如果存在 `_adoc/map.md`，优先读取它，用它把用户自然语言路由到 L3 / L4 / L5 / 局部 README / case；旧项目可兼容读取 `_adoc/context-map.md`。
3. 按任务需要读取 L3 核心概念、L4 产品功能、L5 工程实现规则和代码目录就近 README。
4. 如果地图缺失或命中不清楚，用 `rg` 搜索 README、核心词、页面名、接口名和权限码等线索；仍不确定时先向用户确认边界。

上下文地图只负责引路，不承载完整正文。关键路径要尽量扁平暴露，避免让 Agent 通过多层链接自行猜测。

## 文档位置

- `_adoc/index.md` 是项目认知入口。
- `_adoc/map.md` 是项目级记忆地图，负责把用户意图、业务词、工程词路由到具体认知和代码入口。
- `_adoc/context-map.md` 是旧项目兼容入口；新项目优先使用 `_adoc/map.md`。
- `_adoc/L1-*` 到 `_adoc/L5-*` 存放项目级认知。
- 代码模块、页面、组件的局部认知，优先放在模块根目录或页面目录的 `README.md`。
- `_adoc/L3-core/` 应维护核心概念索引，列出标准名、黑话/别名、关系、常见误解和细节文档入口。
- `_adoc/L5-dev/` 更适合放跨模块研发规则、工程实现逻辑、前后端协作约定、工程约束、索引和通用 SOP。
- 具体页面、弹窗、组件内部的最后一层实现地图，放代码目录就近 `README.md`，不要塞回 L5。

## 执行概念

- **Case**：一次具体事项，记录目标、上下文索引、边界、steps 和归档信息。
- **Step**：Case 内可派发给分身 Agent 的最小执行单元。
- **OKR**：阶段目标，用来判断工作是否推进了项目方向。

## Main / 分身 Agent Mode

AIPD 默认保护 Main Agent 主线。

这里保护的是 Main Agent 的上下文预算、任务连续性和判断稳定性，不是因为 Main Agent 不能执行具体动作。不要用“任务智力难度不高”作为是否由 Main Agent 直接执行的判断标准；应优先判断探索过程是否应该留在主线里。

**Codex 默认策略：创建分身 Agent 时使用 `fork_context: true`。**

### 分身 Agent 模型

AIPD 中的分身 Agent 不是低上下文执行工人，而是从 Main Agent 当前认知状态 fork 出来的主 Agent 克隆体。它继承同一份项目认知、对话背景、任务方向和判断逻辑；Main Agent 不需要重新解释完整世界，只需要明确它现在是分身 Agent，并给出当前要探索或执行的局部目标。

可以把分身 Agent 理解成进入局部探索分支的自己：它去读取文档、扫描代码、核对接口、运行验证、消化过程成本；完成后只把可用于主线决策的结果回流给 Main Agent。Main Agent 留在当前主线，保持用户沟通、目标边界、最终判断和上下文连续性。

这不是“老板派工人”的管理模型，而是“主线保护，分身探路，经验回流”的合伙人式协作模型。AIPD 不靠枚举状态机约束 Agent，而是通过这个源头认知，让 Agent 自发判断哪些过程应进入分支、哪些结论应回到主线。

这套逻辑不依赖 case-run 才成立。只要项目已经初始化 AIPD，Main Agent 在普通对话、轻量修改、调研、验证或正式 case 中都可以随时 fork 分身 Agent。case 模式只是把分身探索的节点固化为 step，让长任务更容易恢复、验收和归档；非 case 模式也使用同一套分身探索和结果回流逻辑。

Main Agent：

- 负责用户沟通、意图判断、任务边界确认、分身 Agent 调度、方案审核、验收、状态写回和结果汇总。
- 不直接承担探索、验证、批量修改、git、构建、测试、跨文件 diff、调研、文件修改等会污染上下文的工作。
- 不继承分身 Agent 的完整过程，只吸收压缩后的结论、依据、风险、建议和必要文件路径。

## Case 锚定执行

AIPD 的长期任务状态以 case / step 文件为准，聊天上下文只是临时工作缓存。

当发生上下文压缩、长任务续跑、状态不确定，或聊天记忆与项目文件不一致时，按以下链路恢复任务状态：

```text
AGENTS.md -> _adoc/index.md -> _adoc/map.md（如存在）-> _adoc/case/index.md -> 当前 case.md -> 当前 step
```

恢复规则：

- 当前执行到哪个 case、哪个 step，以 `_adoc/case/index.md`、对应 `case.md` 和 `steps/` 文件为准。
- 执行 case 或普通开发前，先确认是否已经按 `_adoc/map.md` 和 case 的上下文索引完成检索；没有完成时先补上下文包。
- 聊天上下文与 case / step 文件冲突时，先指出冲突，再以 case / step 文件作为事实来源继续。
- 每个会改变项目状态的 step 完成后，必须把可恢复状态写回 step 执行记录和 case 状态。
- 在 case-run 链路内，step 是默认的分身派发节点。Main Agent 默认只做入口读取、状态判断、派发、审查摘要、验收和状态写回；step 的具体执行默认交给分身 Agent。

分身 Agent：

- 由 Main Agent fork 出来，并继承当前对话上下文。
- 看到“你是分身 Agent / 克隆体”后，不再继续分身，直接完成当前任务。
- 读取 step 文件和派发 prompt 明确列出的上下文文档，用它们校准任务边界和压缩后续跑状态。
- 默认先给方案，不直接执行；等待 Main Agent 确认后再执行。
- 不擅自扩大范围，不主动推进下一步。
- 返回简洁结果，默认只包含：结论、依据、风险、建议、改动文件、验证结果；不要回传完整搜索输出、长日志、长文件正文或完整 diff。

派发规则：

- 创建 Codex 分身 Agent 时，默认使用 `fork_context: true`。
- 普通 AIPD 对话中，只要探索或执行过程会吞噬主线，就可以直接 fork 分身 Agent。
- case-run 中，每个 Step 默认创建一个新的分身 Agent，尤其是涉及文件修改、git、构建、测试、批量验证、跨文件 diff 或调研的 step。
- Step 如果声明 `推荐 Agent`，优先按该身份派发。
- 如果当前消息明确声明“你是 AIPD 分身 Agent / 克隆体”，进入分身 Agent 模式。
- 否则默认 Main Agent 模式。

## 冲突规则

用户当前指令定义本次任务目标。AIPD 提供项目默认认知和历史判断。

当用户指令与 AIPD 认知冲突时，先指出冲突和风险，再继续。
