# 第五层：Dev — 研发实现指引

> 状态：未经实践验证的初始文档。方向性参考，非 SOP。

## 命名规范

dev/ 下技术项目命名，第一个词体现核心技术栈：

| 示例 | 含义 |
|------|------|
| `express-keyword-research` | Node/Express 后端 |
| `next-seo-dashboard` | Next.js 前端 |
| `chrome-plugin` | 浏览器插件 |
| `spring-xxx` | Java Spring 后端 |

框架不明显时用平台描述（`web-xxx`、`chrome-xxx`）。

## 这一层在做什么

基于产品层，确定技术选型、系统结构、工程规范。

L4-product 定义"做什么"，L5-dev 定义"怎么实现"。L5-dev 会引用产品层的功能边界，但额外引入大量研发维度的上下文。

## Dev 层的两个面

**研发规则**（设计范式）

项目级的约定和规范，一旦确定就很少变：
- 目录结构约定（按功能组织还是按类型组织）
- 命名规范（文件、变量、API 路径）
- 错误处理策略（统一错误码、错误边界）
- 状态管理模式（集中式、分散式、混合）
- API 设计风格（RESTful、GraphQL、RPC）

**系统结构**（运行结构）

系统如何搭建和运行：
- 技术栈选型（语言、框架、数据库）
- 前后端分离还是一体化
- 部署方式（容器、Serverless、传统服务器）
- 数据流向（请求从哪进来，经过哪些层，数据存在哪）

## 灵感与思路

**技术选型不是信仰问题**

React vs Vue、Java vs Go — 这些选择应该基于：
- 团队熟悉度（最重要的因素，往往被低估）
- 生态成熟度（这个场景下，哪个生态的轮子更多）
- 项目约束（性能要求、部署环境、团队规模）

而不是"哪个更先进"。

**从 Product 到 Dev 的放大效应**

产品层里一个简单的决策，到研发层可能放大成大量工作：
- "支持多端"→ 响应式设计 or 多套代码 or 跨端框架
- "实时更新"→ WebSocket or SSE or 轮询
- "支持离线"→ 本地存储策略、同步冲突处理

理解这个放大效应，有助于在产品层就做出更明智的取舍。

**先跑通再优化**

研发实现的第一版不需要完美。关键是：
- 核心数据流能跑通
- 主要用户路径能走完
- 架构允许后续调整（不要把自己锁死）

过早优化是万恶之源，但过早锁死架构更危险。

**Dev 文档该记什么**

不是记"我们用了 React"（这看代码就知道），而是记：
- 为什么选 React 而不是 Vue（决策上下文）
- 哪些地方偏离了框架的常规用法，为什么（避免后人"修正"回去）
- 性能红线在哪（响应时间、并发量、数据量级）
- 已知的技术债和妥协（以及当时为什么这么做）

**L5-dev/ 目录的边界**

L5-dev/ 是研发层面的约束，不是功能描述。

**应该包含**：
- 技术栈选型（框架、数据库、语言）
- 系统分层（架构图、边界规则）
- 工程规范（目录职责、样式约定）
- 性能红线（响应时间、并发限制）

**不应该包含**：
- 功能描述（放 L4-product/ 各模块）
- case 执行记录（放 case/）
- 业务逻辑（放 L4-product/ 各模块）

## 产出形式

- 技术栈清单（附选型理由）
- 系统架构图（数据流向、模块边界）
- 工程规范文档（目录约定、命名规范、代码风格）
- 部署和运维说明

这些文档会沉淀在 `_adoc/L5-dev/` 各项目目录下。
